## 바이브 코딩이 좋아서가 아니라, 무서워서 정리했다

AI에게 "이 기능 만들어줘"라고 던지면 200줄짜리 PR이 1분 만에 돌아온다. 처음에는 신기했고, 두 번째는 편했고, 세 번째부터는 무서웠다. 작동은 하는데, 왜 작동하는지 모르겠고, 어디서 무엇이 수정됐는지 추적이 안 된다.

바이브 코딩(Vibe Coding) — 자연어로 의도만 던지고 AI가 코드를 생성하는 방식 — 의 진짜 문제는 코드가 많이 나오는 게 아니다. **코드 검토와 설계라는 개발자의 본업이 사라진다**는 점이다.

## 현업에서 모두가 겪고 있는 4가지 문제

Reddit, 오픈소스 커뮤니티, Claude Code·Cursor 사용자 게시판을 보면 같은 불만이 반복된다. 내가 겪은 게 나만의 문제가 아니었다. 핵심은 네 가지로 수렴된다.

1. **코드 생성 속도가 인간의 이해 속도를 초과한다.** 30초에 200줄이 떨어지는데 사람이 한 줄씩 읽어서 이해하는 데는 30분이 걸린다. 시간이 누적될수록 격차는 벌어진다.
2. **AI가 파일을 계속 추가·수정하면서 구조 일관성이 무너진다.** "new file here, duplicate function there" — AI는 기존 구조를 존중하지 않고 자신의 해결책에만 집중한다.
3. **"왜 이렇게 구현됐는지" 맥락이 사라진다.** 한 달 뒤 같은 코드를 보면 누가 짰는지 모르는 외부인의 코드처럼 느껴진다.
4. **QA·리뷰가 생성 속도를 따라가지 못한다.** PR이 쌓이고, 리뷰는 형식만 남는다.

그래서 현업의 흐름은 이미 한 번 꺾였다. **"AI가 생성한 코드를 인간이 모두 읽는다"는 접근을 포기**하고 있다. 대신 이렇게 바뀌었다.

- 코드 *자체*를 검증하지 않는다
- **구조·계약·변경 범위·테스트**를 검증한다
- AI를 "개발자"가 아니라 "작업 유닛"처럼 관리한다

즉, 인간은 이제 "타이핑"보다 **"감독" 역할**로 이동하는 중이다. 아래 6단계는 그 감독 역할을 구체화한 가이드레일이다. 단계별로 "개발자가 무엇을 하고, AI가 무엇을 하는지"를 분명히 갈라놓는 것이 핵심이다.

## Phase 1. 가이드레일 구축 — AI에게 페르소나를 입힌다

코드를 한 줄도 짜기 전에 먼저 하는 일이다.

`.cursorrules`, `CLAUDE.md`, `AGENTS.md` 같은 프로젝트 루트 지침 파일에 AI가 준수해야 할 규칙을 명문화한다. 프레임워크 코딩 스타일, 폴더 구조, 상태 관리 방식, 네이밍 컨벤션, 금지된 패턴까지.

```markdown
# 예시: .cursorrules

- Next.js 16 App Router를 사용한다
- 상태 관리는 Zustand만 사용한다 (Redux 금지)
- 컴포넌트는 src/components/{도메인}/ 하위에 배치한다
- 모든 API 호출은 src/lib/api/ 의 클라이언트를 거친다
- 한국어 주석 금지, 영어로 작성한다
```

이걸 하지 않으면 AI는 매번 다른 스타일로 코드를 생성한다. 어제는 axios로 호출했다가 오늘은 fetch로 호출하고, 한 번은 styled-components를 쓰고 다음 번엔 Tailwind를 쓴다. 일관성 없는 코드베이스는 결국 사람이 다 정리해야 한다.

### Scope Constraint — "무엇을 만들까"보다 "어디까지 건드릴 수 있는가"

가이드레일 중에서도 **가장 효과가 큰 한 가지**가 있다. 작업 범위 제한이다.

AI는 기본적으로 "문제 해결"만 목표로 한다. 그래서 기존 구조를 존중하지 않고 새 파일·새 훅·새 패턴을 계속 만든다. 이걸 막으려면 *작업 범위* 자체를 명시적으로 잠가야 한다.

```text
You may only edit:
- src/features/auth/**
- src/shared/api/auth.ts

Do not:
- create new folders
- duplicate utilities
- change design system
- introduce new state libraries
```

작업 시작 전에 이걸 프롬프트로 박아두면, AI가 엉뚱한 디렉토리까지 손대는 것을 막을 수 있다. 디렉토리·파일 단위뿐 아니라 함수 단위, diff size 제한까지 거는 팀도 있다. 실제로 현업 사용자들이 가장 효과 크다고 입을 모으는 기법이다.

**핵심:** AI가 일관성 없는 코드를 생성하거나 엉뚱한 위치에 파일을 만드는 것을 사전 차단한다. "무엇을 만들까?"보다 **"어디까지 건드릴 수 있는가?"**를 강하게 제한한다.

## Phase 2. 요구사항 정의 및 명세 — 구현 전에 설계도부터

"로그인 기능 만들어줘"라고 던지지 않는다. 그러면 AI는 자기가 본 적 있는 가장 일반적인 로그인을 만들어준다. 우리 서비스에 맞는 로그인이 아니다.

대신 이렇게 묻는다.

```
"OAuth 구글 로그인 + 이메일 매직 링크 두 방식을 지원하는
로그인 기능을 만들려고 합니다.
구현 계획을 먼저 서술해줘. 코드는 아직 짜지 마세요.

- 사용할 라이브러리
- 데이터 모델
- 화면 흐름
- 에러 케이스
- 보안 고려사항"
```

이 단계에서 PRD나 데이터 모델을 AI와 대화하며 확정한다. 논리적 결함이 있으면 여기서 잡힌다. AI가 "Supabase Auth를 쓰겠다"고 했는데 우리는 Auth.js를 쓰고 있다면, 이 단계에서 바로잡아야 한다. 코드를 다 짜놓고 나서 발견하면 100줄을 갈아엎어야 한다.

**핵심 규칙:** "만들어줘" 대신 "계획을 먼저 서술해줘"라고 말한다.

## Phase 3. 테스트 코드 우선 생성 — 무엇이 성공인지 정의한다

바이브 코딩의 진짜 핵심이다. 코드를 짜기 전에 **무엇이 성공인지** 정의한다.

확정된 설계도를 바탕으로 AI에게 테스트 코드부터 작성하게 한다.

```
"앞서 합의한 설계도를 바탕으로,
이 로그인 기능을 검증할 테스트 케이스를 먼저 작성해줘.
Vitest를 사용하고, 다음 시나리오를 다뤄야 합니다.

1. 정상 OAuth 로그인 성공
2. 이메일 매직 링크 발송 성공
3. 매직 링크 만료 시 에러
4. CSRF 토큰 누락 시 거부
5. 동일 이메일 중복 가입 차단"
```

유저플로우는 이렇다.

1. 개발자가 기능을 설명한다.
2. AI가 검증할 테스트 케이스를 생성한다.
3. 개발자가 테스트 코드를 리뷰하며 요구사항이 맞는지 확인한다.

테스트를 사람이 직접 검토하는 게 중요하다. 테스트가 잘못되어 있으면 AI가 잘못된 코드를 통과시키고도 "성공"이라고 보고한다. **테스트는 사양이다.** 사양을 사람이 승인하지 않으면 안 된다.

## Phase 4. 원자적 코드 생성 — 한 번에 한 가지만

200줄짜리 PR이 한 번에 떨어지면 검토가 안 된다. 그래서 단계를 쪼갠다.

AI에게 "테스트를 하나씩 통과할 수 있는 최소 단위의 코드"를 요구한다.

- ❌ "로그인 기능 전체를 한 번에 구현해줘"
- ✅ "먼저 OAuth 콜백 라우트만 만들어. 테스트 1번을 통과시키는 게 목표야"

여러 파일을 동시에 수정해야 할 때는 반드시 **어떤 파일들이 왜 수정되는지 요약을 먼저 받는다**.

```
"수정 시작 전에 어떤 파일들을 어떤 이유로 수정할지 목록을 줘.
내가 OK 하면 진행해."
```

이렇게 하면 AI가 엉뚱한 파일까지 건드리는 것을 막을 수 있다. "이 파일은 왜 수정해?"라는 질문 한 번이 나중에 한 시간짜리 디버깅을 막아준다.

## Phase 5. 차이점 검토 및 검증 — Diff와 교차 검증

AI가 작성한 코드를 무비판적으로 수용하지 않는다.

### Diff 활용

에디터의 Diff 뷰로 추가된 라인과 삭제된 라인을 스캔한다. 변경 사항이 30줄 미만이면 한 줄씩 읽고, 그 이상이면 다음 질문을 한다.

- 이 함수가 호출되는 곳은 어디인가?
- 삭제된 코드는 정말로 안 쓰이는가?
- 추가된 의존성은 정말 필요한가?
- 환경 변수나 시크릿이 하드코딩되지 않았는가?

### 교차 검증

가장 효과적인 트릭이다. **현재 코드를 작성한 모델과 다른 AI에게 리뷰를 시킨다**.

```
"Claude가 작성한 이 변경 사항을 검토해줘.
보안 취약점이나 성능 저하 요소가 있는지,
숨겨진 부작용이 있는지 점검해줘."
→ GPT-5나 Gemini에게 요청
```

같은 모델에게 자기가 짠 코드를 리뷰하라고 하면 자기 합리화를 한다. 다른 모델은 다른 학습 데이터와 다른 패턴 인식을 갖고 있어서, 첫 모델이 놓친 것을 잡아내는 경우가 의외로 많다. 특히 보안 관련 이슈에서 효과가 크다.

### Builder Agent vs Reviewer Agent — 역할 분리

한 발 더 나아가면, 생성 AI와 리뷰 AI를 *역할 자체로* 분리하는 패턴이 있다.

```text
Builder Agent
→ 코드 생성에 집중

Reviewer Agent
→ 아키텍처 위반 탐지
→ 중복 로직 탐지
→ 타입 누락 탐지
→ 불필요 abstraction 탐지
→ 디자인 시스템 토큰 미사용 탐지
```

같은 모델이라도 시스템 프롬프트를 다르게 주면 행동이 완전히 달라진다. Builder에게는 "구현 우선", Reviewer에게는 "의심 우선"을 주입한다. Claude Code의 서브에이전트, Cursor의 멀티 에이전트 모드, OMC 같은 오케스트레이터가 이 패턴을 표준화하고 있다. 사람이 매번 "이건 보안 봐줘, 이건 성능 봐줘"라고 지시하는 것보다 훨씬 일관성 있는 리뷰가 나온다.

## Phase 6. 기록 및 동기화 — 작은 단위로 커밋

수정 사항을 추적 가능하게 기록하는 마지막 단계다.

작업이 완료되면 AI에게 현재 변경 사항을 요약한 커밋 메시지를 작성하게 한다.

```
"방금 수정한 변경 사항을 Conventional Commits 형식의
커밋 메시지로 정리해줘. 본문에는 왜 이 변경이 필요했는지,
어떤 테스트가 통과하는지 적어줘."
```

추적성을 확보하는 핵심은 **아주 작은 기능 단위로 커밋을 남기는 것**이다. 한 PR에 50개 파일이 묶이면 롤백이 불가능하다. "OAuth 콜백 라우트 추가" → "매직 링크 발송 로직 추가" → "CSRF 보호 추가"처럼 한 번에 한 가지만 커밋한다.

나중에 "이 시점에서 버그가 들어왔다"는 게 명확해야, 그 커밋만 롤백할 수 있다.

### AI Work Log — 작업 의도까지 함께 남긴다

커밋 메시지만으로는 부족하다. "AI에게 어떤 의도로 무엇을 시켰는지"가 사라지면 한 달 뒤 그 코드를 누구도 해석하지 못한다. 그래서 별도의 작업 로그를 둔다.

```markdown
# 2026-05-13
## Auth Refactor
Changed:
- authStore.ts
- useAuth.ts

Reason:
- duplicated login logic between web/mobile

Risk:
- token refresh flow not yet covered by tests

Builder: Claude
Reviewer: GPT-5 (passed)
```

`docs/ai-worklog/` 같은 디렉토리에 날짜별로 누적하면 된다. 별 거 아닌 것 같지만, **추후 디버깅과 신규 합류자 온보딩에서 진짜 빛난다**. "이 코드 왜 이래?"라는 질문 한 번에 일주일이 날아가는 걸 막아준다.

## 6단계 요약표

| 단계 | 행동 | 개발자의 역할 | AI의 역할 |
| --- | --- | --- | --- |
| **Phase 1** | 규칙 설정 | 프로젝트 제약·범위 명시 | 규칙 숙지·페르소나 설정 |
| **Phase 2** | 설계 확인 | 구현 계획 승인 | 로직·아키텍처 제안 |
| **Phase 3** | 테스트 생성 | 테스트 케이스 검증 | 검증용 자동화 코드 작성 |
| **Phase 4** | 코드 구현 | 원자적 단위 실행 명령 | 기능 구현 코드 생성 |
| **Phase 5** | 코드 리뷰 | Diff 최종 수락 | 피드백 반영·교차 검증 |
| **Phase 6** | 커밋·기록 | 버전 관리 확정 | 변경 요약·작업 로그 작성 |

## 생성기보다 분석기로 — AI 활용의 숙련 곡선

초보와 숙련자가 AI를 쓰는 방식은 정확히 반대다.

- **초보**: "전체 기능 만들어줘"
- **숙련자**: "중복 제거해", "구조 분석해", "dependency graph 만들어", "dead code 찾아", "레이어 위반 탐지해"

즉, **AI를 생성기보다 분석기로 쓴다**. 이게 유지보수성이 압도적으로 높다. RPCS3 같은 오픈소스 프로젝트도 "AI를 refactoring에는 쓸 수 있지만 무분별한 생성은 문제"라는 입장을 공개적으로 밝힌 적이 있다. 같은 AI라도 분석 작업에 쓰면 위험이 줄고, 생성에 쓰면 위험이 늘어난다.

분석용 프롬프트 몇 가지를 상비해두면 좋다.

- "이 모듈의 import graph를 추출해줘"
- "src/features/ 안에서 중복된 유틸리티 함수를 찾아줘"
- "shared layer가 feature layer를 역참조하는 곳이 있는지 찾아줘"
- "최근 30일 커밋에서 dead code가 된 것 같은 export를 찾아줘"

## Repository Map — AI가 어디를 건드렸는지 시각화

"어디에 어떻게 작성됐는지 추적이 어렵다"는 문제를 해결하기 위해, 최근에는 코드베이스 시각화 도구들이 빠르게 늘고 있다.

- import graph
- dependency graph
- call graph
- feature ownership map

```text
Auth Feature
 ├── api/
 ├── hooks/
 ├── store/
 ├── ui/
```

이런 트리·그래프를 항상 최신 상태로 유지하고, AI 작업 후 변경 영역을 색칠해서 보면 "AI가 무슨 짓을 했는지" 한눈에 들어온다. Claude Code의 codebase summary, Cursor의 codebase indexing, 별도 도구로는 dependency-cruiser, madge 같은 게 많이 쓰인다. 솔로 빌더라면 `madge --image` 한 줄로 PNG로 뽑아 README에 박아두는 것만으로도 충분하다.

## Human Approval Point — 자율을 양보하지 말아야 할 4 영역

AI 자율도가 올라가는 만큼, **자율을 절대 주면 안 되는 영역**도 명확해진다.

| 영역 | 이유 |
| --- | --- |
| **Auth** | 토큰 저장/검증, CSRF, 세션 관리는 실수 한 번에 계정 전체가 털린다 |
| **Payment** | 결제 금액·환불 로직은 금전 손실로 직결 |
| **Infra** | DB 마이그레이션, 환경 변수, 배포 스크립트는 롤백 비용이 비대칭적으로 크다 |
| **Schema migration** | 한 번 잘못 마이그레이션하면 운영 데이터가 망가진다 |

이 4영역은 **인간 승인 게이트**를 반드시 둔다. AI가 만든 PR을 자동 머지하지 않는다. 최근에 보고되는 vibe coding 보안 사고도 대부분 이 영역에서 발생한다. 자동화 욕심으로 이 게이트를 풀면 안 된다.

## AI 운영 5 Layer 구조 — 생성 시스템과 운영 시스템은 다르다

규모가 커지면 "프롬프트 잘 쓰기"의 한계가 온다. 이때 필요한 건 더 좋은 모델이 아니라 **운영 레이어**다.

### Layer 1 — Repo Rule System

각 프로젝트 루트에 다음 파일을 둔다.

```text
/AGENTS.md          # 에이전트 페르소나·역할 분담
/ARCHITECTURE.md    # 폴더 책임·레이어 규칙
/CODING_RULES.md    # naming, forbidden pattern, design system
```

프로젝트 자체가 AI onboarding 문서를 들고 있는 구조다. 매번 같은 설명을 프롬프트에 붙이지 않아도 된다.

### Layer 2 — AI Work Log

앞서 본 `docs/ai-worklog/` 디렉토리. 모든 AI 작업의 의도·변경 파일·리스크·리뷰어를 기록한다.

### Layer 3 — Diff-first Workflow

`git diff` 와 `git add -p` 가 기본 단위가 된다. 전체 파일 읽지 않고 **추가된 dependency / 새 util / duplicated logic / state mutation** 만 보면 90%는 잡힌다.

### Layer 4 — AI Reviewer Agent

Builder 가 코드를 만들면 Reviewer 가 즉시 아키텍처 위반·중복·타입 누락·불필요 추상화를 점검한다. PR 자동 코멘트로 붙여두면 사람의 리뷰 부담이 크게 줄어든다.

### Layer 5 — Human Approval Point

auth / payment / infra / schema 변경에는 사람 게이트. 자동화하지 않는다.

이 5개 레이어가 갖춰지면, AI가 만들어내는 코드의 양이 늘어나도 운영이 무너지지 않는다. 솔로 빌더라면 Layer 1·3·5 만으로도 시작할 수 있고, 팀이 커지면 Layer 2·4 를 추가하면 된다.

## 가장 적용하기 까다로운 단계는 어디인가

여섯 단계를 다 지키면 좋지만, 현실은 그렇지 않다. 내 경험으로는 Phase 3(테스트 우선)가 가장 어렵다.

이유는 단순하다. **눈에 보이는 진척감이 없기 때문이다**. 테스트 코드 30분 짜는 동안 화면에는 아무것도 안 뜬다. 그동안 옆 사람은 AI에게 "만들어줘"라고 던지고 30분 만에 동작하는 화면을 띄우고 있다. 비교가 안 된다.

그런데 일주일 뒤가 다르다. 테스트 없이 AI에게 짜라고 한 코드는 작은 변경 한 번에 흔들린다. 어디가 망가졌는지 모르니까 매번 손으로 클릭해서 확인한다. 그게 결국 더 느리다.

Phase 3을 건너뛰지 않으려면 트릭이 하나 있다. **테스트 작성도 AI에게 시킨다**. 사람이 "어떤 시나리오를 검증할지"만 정의하고, 코드는 AI가 짠다. 그러면 30분이 5분으로 줄어든다. 진척감 문제도 같이 해결된다.

## 2025 → 2026, 패러다임은 이미 한 번 이동했다

올해와 작년의 결정적 차이는 모델 성능이 아니라 *질문의 방향*이다.

| 2025 | 2026 |
| --- | --- |
| **AI가 코드를 생성**한다 — 어떻게 더 잘 만들 것인가? | **AI 생성물을 어떻게 통제할 것인가**? |
| 프롬프트 엔지니어링이 중심 | Context Engineering·운영 레이어가 중심 |
| 개인 생산성 도구 | 팀·프로젝트 단위 운영 체계 |

상위권 AI 활용 개발자들은 이제 "AI가 코드를 잘 짜게 하는 것"보다 다음에 더 많은 시간을 쓴다.

- AI가 구조를 망치지 않도록 막기
- 변경 범위를 추적 가능하게 유지하기
- 리뷰 비용을 줄이는 자동화 만들기
- 컨텍스트를 잃지 않게 유지하기

멀티 프로젝트·에이전트 오케스트레이션·장기 유지보수까지 들어간 팀이라면 "프롬프트 잘 쓰기"보다 **AI 운영체계 설계**가 다음 레벨업의 핵심이다.

## 마무리 — AI는 도구다, 운전자가 아니다

이 플로우의 본질은 단순하다. **설계와 검토는 사람이, 구현은 AI가**.

AI에게 운전대를 통째로 넘기면 어디로 가는지 모르고, AI를 안 쓰면 너무 느리다. 그 사이에서 균형을 잡는 방법이 가이드레일이다. 6단계 모두를 매번 지키지 못해도, 적어도 Phase 1(규칙·범위 제한)과 Phase 5(Diff 검토)는 양보하지 말기 바란다. 이 둘만 지켜도 코드베이스의 일관성과 추적성이 무너지지 않는다.

지금 진행 중인 프로젝트에서 이 흐름 중 가장 적용하기 까다로운 단계는 어디인가? 댓글로 공유해주시면 더 구체적인 사례로 풀어보겠다.
