## 1. 스키마와 트랜잭션

### 01      스키마

> 데이터베이스의 구조와 제약 조건에 관한 전반적인 명세를 기술한 메타데이터

#### 스키마가 정의하는 것

- 데이터 개체

- 속성

- 관계

- 데이터 조작시 데이터 값들이 갖는 제약 조건

### 02      트랜잭션

> 데이터베이스의 상태를 변화시키는 기능을 수행하는 작업 단위

#### 특징

- 병행제어, 회복 작업 시 처리되는 작업의 단위

- - 병행: 여러개의 트랜잭션들이 동시에 실행

- 사용자가 시스템에 서비스 요구 시 시스템이 응대  하는 작업 단위

- 하나의 트랜잭션은 commit 되거나 rollback 된다

#### 상태

- **활동(Active)** : 트랜잭션이 실행중인 상태

- **실패(Failed)** : 트랜잭션 실행에 오류가 발생하여 중단된 상태

- **철회(Aborted)** :트랜잭션이 비정상적으로 종료되어 rollback 연산을 수행한 상태

- **부분 완료(Partially Committed)** : 트랜잭션의 마지막 연산까지 실행했지만, Commit 연산이 실행되기 직전의 상태

- **완료(Committed)** : 트랜잭션이 성공적으로 종료되어 Commit 연산을 실행한 후의 상태

활동 > 부분완료 > 완료

활동 > 실패 > 철회

## 2. DBMS 구성

![image](/images/migrated-tistory/deiteobeiseu-seukima/img.jpg)
DBMS의 구성

### 01      데이터사전

- 시스템 카탈로그

- db에 저장되는 데이터에 관한 정보 메타데이터를 저장한다

### 02      DDL 컴파일러

- 데이터 정의어로 작성된 스키마를 해석한다

- 저장 데이터 관리자의 도움을 받아 새로운 DB 구축 한다

**데이터 사전에 저장되는것**

- 스키마의 정의

- DDL로 작성된 기존 스키마의 삭제나 수정요청

### 03      DML 프리 컴파일러

- 응용 프로그램에 삽입된 데이터 조작어를 추출 => DML 컴파일러에게 전달

- 데이터 조작어와 관련없는 코드는 해당언어의 컴파일러에 저장

### 04      DML 컴파일러

- 데이터 조작어로 작성된 데이터의 처리(삽입, 수정, 삭제, 검색) 요구를 분석

- 런타임 데이터베이스 처리기가 이해할 수 있도록 해석

### 05      런타임 데이터베이스 처리기

- 저장 데이터 관리자를 통해 DB 접근

- DML 컴파일러로 부터 전달받은 데이터 처리요구를 DB에서 실제로 실행

### 06      트랜잭션 관리자

- 사용자가 DB 접근을 할 수 있는지 접근 권한을 검사

- DB 무결성을 유지하기위한 제약조건위반 여부를 확인

- 회복, 병행 수행

### 07      저장 데이터 관리자

- 사용자 DB와 데이터 사전을 관리

- 운영체제의 도움을 받아 DB에 접근

## 3. 데이터베이스 스키마

데이터베이스에서

- 자료의 구조

- 자료의 표현 방법,

- 자료 간의 관계

를 형식 언어로 정의한 구조

- **외부단계(사용자가 보는 데이터)** : 외부 스키마

분리를 해놓음으로써 데이터의 독립성을 유지할 수 있다**(외부/개념 매핑)**

- **개념단계(전체데이터)** : 개념 스키마

물리적인 데이터의 독립성을 유지할 수 있다**(개념/내부 매핑)**

- **내부단계(DBMS가 보는 데이터)** : 내부 스키마

**=> 데이터베이스(물리적인 저장된 데이터)**

### 01      외부 스키마

- **일반 사용자, 응용 프로그래머가 접근하는 계층**

- 사용자에게 필요한 데이터베이스를 정의한것

- 하나의 DB에 여러개의 외부 스키마가 존재할수있음

- 하나의 외부 스키마를 여러개의 응용프로그램이나 사용자가 공유할 수 있음

### 02      개념 스키마

- DB의 전체적인 논리적 구조

- **하나의 DB에서 하나의 개념 스키마만 존재**

- 데이터베이스 관리자(DBA)가 직접 만듦

- 일반적으로 언급하는 스키마

- 개념스키마가 가지고 있는 내용

- 데이터의 **관계**

- **제약조건**

- **무결성**에 대한 내용

- 데이터에 대한 **보안 정책**

- **접근 권한**

### 03      내부 스키마

- **DBMS가 보는 데이터**

- 물리적 저장장치에 DB가 실제로 저장되는 방법

- 사용자는 접근 권한이 없고 시스템 프로그래머나 DB 시스템 설계자가 보는 관점의 스키마

- 정의하는것

- 레코드 **구조**

- 필드 **크기**

- 레코드 **접근 경로**

- **데이터 압축**

## 4. 매핑(사상)

### 01      매핑(Mapping)/사상

> 연관 관계가 있는 다른 값으로 연결(대응)시키는 것을 매핑이라 한다

### 02      외부/개념 매핑

- **외부 스키마**의 데이터와 관계있는 **개념 스키마**의 값으로 대응한다

- **응용 인터페이스** 라고도 한다

### 03      개념/내부 매핑

- **개념 스키마**의 데이터와 연관되어 어떤 **내부 스키마**의 물리적 장치에 저장되는 대응

- **저장 인터페이스** 라고도 한다

## 5. 데이터 독립성

### 01      데이터 독립성

> 특정 스키마를 변경해도 상위 수준의 스키마 정의에 영향을 주지 않는 성질

### 02      논리적 데이터 독립성

- **외부 단계와 개념 단계** 사이의 **독립성**

- 개념 스키마가 변경되도 외부 스키마에는 영향이 없다

### 03      물리적 데이터 독립성

- **개념 단계와 내부 단계** 사이의 **독립성**

- 저장 장치 구조 변경과 같이 내부 스키마가 변경되어도 개념스키마에 영향이 없도록 해야 한다

- CHAR(20) -> CHAR(30) 은 가능하나 반대로는 변경이 안되게끔 해야한다
