> 💡 [FCM(Firebase Cloud Messaging)](https://firebase.google.com/docs/cloud-messaging?hl=ko&_gl=1*1oacq3*_up*MQ..*_ga*ODIzMzc4NzMyLjE3NTAxNDU0NDQ.*_ga_CW55HF8NVT*czE3NTAxNDU0NDMkbzEkZzAkdDE3NTAxNDU1NjMkajYwJGwwJGgw) 이란?  
> 메세지를 안정적으로 전송할 수 있는 크로스 플랫폼 메시징 솔루션


## 주요 기능


### 알림 메시지 또는 데이터 메시지 전송


사용자에게 표시되는 알림 메시지를 전송합니다. 또는 데이터 메시지를 전송하고 애플리케이션 코드에서 임의로 처리합니다.


### 다양한 메시지 타겟팅


**단일 기기**, **기기 그룹**, **주제**를 구독한 기기 등 3가지 방식으로 클라이언트 앱에 메시지를 배포할 수 있습니다.


## 기본 원리


송수신을 위한 두 가지 주요 구성요소

- Cloud Functions for Firebase 또는 앱 서버와 같이 메시지를 작성, 타겟팅, 전송할 수 있는 신뢰할 수 있는 환경
- 해당 플랫폼별 전송 서비스를 통해 메시지를 수신하는 Apple, Android 또는 웹(자바스크립트) 클라이언트 앱

    ![image.png](/images/migrated-notion/fcm-firebase-cloud-messaging-e-daehan-gibon-gaenyeom/image.png)


## **FCM 메시지 종류**


| **메시지 유형**  | **주요 특징**                                                 | **사용 시기**          |
| ----------- | --------------------------------------------------------- | ------------------ |
| **알림 메시지**  | 사용자에게 표시됨 (title/body 포함)백그라운드 시 OS가 표시함포그라운드 시 앱에서 수신 가능 | 마케팅, 리마인더, UI 알림 등 |
| **데이터 메시지** | 커스텀 key-value만 포함, 앱 코드가 처리함포그라운드/백그라운드에서 직접 처리 가능        | 동기화, 앱 내부 로직 트리거   |


➡️ **혼합형** 메시지도 가능 (알림 + 데이터 함께 포함)

- 마케팅/리마인더만 사용 예정이기에 다뤄야 할 부분은 **알림 메시지**

## **전송 방법**


| **전송 방식**           | **사용 시**                   |
| ------------------- | -------------------------- |
| Firebase 콘솔         | 테스트용, 마케팅용, A/B 테스트        |
| Admin SDK / HTTP v1 | 실서비스, 자동화, 예약 푸시           |
| Topics / Group      | 팬아웃(push fan-out), 대규모 전송용 |

## **주요 주의사항 및 제한**


| **항목**           | **설명**                                                |
| ---------------- | ----------------------------------------------------- |
| **Token 만료**     | 앱 삭제 시 등록 토큰이 무효화됨 (이후 메시지 등록 시`NotRegistered` 오류 발생) |
| **팬아웃 제한**       | 동시 팬아웃 1,000개, 속도는 약 10,000 QPS (보장 아님)               |
| **Collapse Key** | 동일 키 가진 메시지는 최근 메시지 1개만 유지됨 (축소형)                     |
| **Payload 제한**   | 4,096바이트 (콘솔 전송은 1,000자 제한 있음)                        |
| **VPN, 방화벽**     | 포트 5228~5230 및 443 열려 있어야 함 (mtalk 도메인)               |
| **TTL(수명)**      | 기본: 4주 (28일), 직접 설정 가능 (ttl 필드)                       |
| **우선순위**         | high: 즉시 전송 (백그라운드 포함)normal: 유휴 시 전송 지연될 수 있음        |
| **Apple**        | APNs에 따라 우선순위 5 (보통)만 허용됨 – high는 오류 발생               |


# 프론트 개발 시 주의할점


## 주제(Topic) 메시지 한도


기기가 특정 주제(Topic)에 구독하거나 구독을 해제하는 속도를 의미함


예제) 마케팅 알람 ON/OFF ⇒ FCM Topic 구독 여부 토글하는것과 같은 개념으로 생각하면 됨

- 구독/해제는 너무 잦으면 QPS 제한에 걸릴 수 있음(프로젝트 당 3,000QPS로 제한)
- 서버쪽에서 실제로 구독.해제가 성공했는지 상태 추적은 안됨

### <u>우려되는 문제사항</u>

- 사용자 A가 초안에 마케팅 알림 토글을 연속해서 5번 누름 > 클라이언트는 서버로 5번 요청 보냄 > 서버는 5번 구독/해제 반복 > 여러 유저가 동시에 하면 제한에 걸릴 수 있음(예: 429 오류)

### <u>클라이언트에서 예방할 수 있는 방안</u>

- 디바운싱 : 토글 후 일정시간 동안 추가 요청 막기
- 최종 상태만 서버로 전송(isPending 상태로 컨트롤 가능)

### <u>백엔드에서 예방할 수 있는 방안</u>

- 중복 요청 무시: 동일한 토큰에 대해 직전 상태와 같다면 처리 스킵

## **전송 메시지 기본 구조**


### **🔸 알림 메시지**


```json
{
  "message": {
    "token": "device_token_here",
    "notification": {
      "title": "Title",
      "body": "Message Body"
    }
  }
}
```


### **🔸 데이터 메시지**


```json
{
  "message": {
    "token": "device_token_here",
    "data": {
      "key1": "value1",
      "key2": "value2"
    }
  }
}
```


### **🔸 알림 + 데이터**


```json
{
  "message": {
    "token": "device_token_here",
    "notification": {
      "title": "Title",
      "body": "Body"
    },
    "data": {
      "customKey": "customValue"
    }
  }
}
```

## **어떤 메시지를 언제 써야 하나요?**


| **상황**                 | **추천 메시지 유형**                    |
| ---------------------- | -------------------------------- |
| 푸시 클릭 시 앱을 열고 특정 화면 이동 | **데이터 메시지** (클릭 이벤트 직접 처리)       |
| 단순 마케팅 알림              | **알림 메시지**                       |
| 알림 + 데이터 동시 전달         | **혼합형 메시지**                      |
| 앱 실행 여부와 관계없이 이벤트 처리   | **데이터 메시지** 사용하여 백그라운드도 수신 가능하도록 |

## 축소형 vs 비축소형 메시지


| 종류   | 사용 시나리오                                                                                                                         | 전송 방법                                                                                                                                        |
| ---- | ------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| 비축소형 | 모든 메시지가 클라이언트 앱에서 중요하므로 모두 전송되어야 합니다.                                                                                           | 알림 메시지를 제외한 모든 메시지가 기본적으로 비축소형입니다.
                                                                                                          |
| 축소형  | 클라이언트 앱과 관련이 없는 이전 관련 메시지를 렌더링하는 새로운 메시지가 있으면 FCM이 이전 메시지를 대체합니다. 서버에서 데이터 동기화를 시작하는 데 사용되는 메시지 또는 기한이 지난 알림 메시지를 예시로 들 수 있습니다. | 메시지 요청에 적절한 파라미터를 설정합니다.
• Android: **`collapseKey`**
• Apple: **`apns-collapse-id`**
• 웹: **`Topic`**
• 기존 프로토콜(모든 플랫폼): **`collapse_key`** |

## 우선순위 설정

- **보통 우선순위****:** 보통 우선순위 메시지는 앱이 포그라운드 상태이면 즉시 전송됩니다. 백그라운드 앱의 경우 전송이 지연될 수 있습니다. 새로운 이메일 알림, UI 동기화 유지, 백그라운드 앱 데이터 동기화와 같이 시간이 크게 중요하지 않은 메시지의 경우 보통 전송 우선순위를 선택하세요.
- **높은 우선순위****:** FCM이 기기가 잠자기 모드인 경우에도 높은 우선순위 메시지를 즉시 전송하려고 시도합니다. 높은 우선순위 메시지는 사용자에게 표시되는 시간에 민감한 콘텐츠를 위한 것입니다.

    > ⚠️ Apple 기기로 데이터 메시지를 전송할 때 우선순위를 5 또는 보통 우선순위로 **설정해야** 합니다. 높은 우선순위로 전송된 메시지는 FCM백엔드에서 **`INVALID_ARGUMENT`** 오류와 함께 거부됩니다.

## **설정 기본값 요약**


| **설정 항목**      | **기본값**                                                                                                                                           |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| ttl (메시지 유효기간) | 0~28일 지정가능 (기본값: 4주)

-TTL이 0초인 메세지에 축소형 메시지 제한을 적용하지 않음
- `time_to_live`값이 `0`이면 즉시 전송할수없는 메시지는 삭제
  - 이러한 메시지는 저장되지않음으로 알림 메시지를 보낼 때 지연시간이 최소화됨 |
| Collapse Key   | 앱 패키지 이름                                                                                                                                          |
| 메시지 우선순위       | Android: normal
APNs: 5 (보통) ⇒ 안그럼 에러남                                                                                                            |
| Payload 크기     | 4KB (콘솔은 1,000자)                                                                                                                                  |
| 축소형 메시지 저장 한도  | Android 100개, 기기당 축소 키 4개까지                                                                                                                       |

## **기술 제안서 작성 시 고려 포인트**

- **메시지 종류에 따라 분기 처리** 설계 필요 (Notification vs Data)
- **기기 상태 (백/포그라운드)에 따라 처리 방식 달라짐**
- **FCM 토큰 갱신 처리 (onTokenRefresh) 및 주기적 업로드 로직 구현 필수**
- **축소형 메시지 남용 시 배터리 영향 고려하여 비축소형 전환 고려**
- **대규모 전송은 팬아웃 제한과 QPS 고려하여 타이밍 설계**
