# repository와 commit

1. `레포지토리(repository)` : 저장소
    - 프로젝트의 변경사항들이 저장되는 곳
2. `커밋(commit)` : 하나의 버전으로 남기는 동작 & 결과물
    - `git commit -m [내용]` : 현재 staging area에 있는 것을 commit으로 남기는 명령어

# git 세팅 명령어

- **`git init`** : 현재 디렉토리를 Git이 관리하는 프로젝트 디렉토리(=working directory)로 설정하고 그 안에 레포지토리(.git 디렉토리) 생성
- **`git config`** **`user.name 'codeit'`** : 현재 사용자의 아이디를 'codeit'으로 설정(커밋할 때 필요한 정보)
- **`git config user.email 'teacher@codeit.kr'`** : 현재 사용자의 이메일 주소를 'teacher@codeit.kr'로 설정(커밋할 때 필요한 정보)

# git의 3가지 작업 영역

1. `working directory` : 작업을 하는 프로젝트 디렉토리
2. `staging area` : git add를 한 파일이 존재하는 영역 (커밋을 하면 staging area에 머물게 된다)
3. `repository` : working directory의 변경 이력들이 저장되는 영역

# git이 보는 파일의 4가지 상태

- `Untracked` : 변동사항이 추척되고 있지 않는 상태
- `Tracked` : 추척 되고 있는 상태
    - `Staged` : 내용이 수정되고나서 staging area에 올라와있는 상태 (스테이징 된 상태)
    - `Unmodified` : 변동 사항이 최신 커밋의 모습과 비교했을 때 전혀 바뀐게 없는 상태
    - `Modified` : 조금이라도 바뀐 내용이 있는 상태이지만 스테이징이 되어있지 않는 상태

### git의 상태에 대해 확인 할 수 있는 명령어

- **`git status`** : Git이 현재 인식하고 있는 프로젝트 관련 상태들을 출력

# git add


> `git add`  
> : **staging area**에 파일을 추가하는 명령어


### git add 사용법

- **`git add [파일 이름]`** : 수정사항이 있는 특정 파일을 staging area에 올리기 (세부 경로 작성)
- **`git add [디렉토리명]`** : 해당 디렉토리 내에서 수정사항이 있는 모든 파일들을 staging area에 올리기
- **`git add .`** : working directory 내의 수정사항이 있는 모든 파일들을 staging area에 올리기
- **`git add -A`** : 전체 프로젝트의 모든 변경 사항 추가
- **`git add -p`** : 변경된 파일의 특정 부분만 선택적으로 스테이징 가능. 각 변경 사항을 하나씩 검토하여 스테이징 할 지 여부를 선택 할 수 있다.

### git add로 커밋에 포함될 수 있는 상태

- **untracked** : 아직 git에서 추적하지 않던 새로운 파일을 스테이징 할 수 있다
- **modified** : 기존에 git에서 관리 중인 파일에 수정사항이 생기면, 그 수정사항을 스테이징 할 수 있음
- **Deleted** : 파일이 삭제되었을 때 그 변경 사항도 스테이징 할 수 있음.

### git add와 working directory의 상태 변화

- **Untracked** : git이 추적하지 않은 파일
- **modified** : 파일이 수정되었으나 스테이징 되지 않은 상태
- **staged** : 파일이 스테이징되었고 커밋 할 준비가 된 상태

**즉, git add 는 modified 상태에 있는 파일을 staged 상태로 변경하는 역할을 한다.**


# git reset


> `git reset [commit ID]`  
>   
> commit, staging area, working directory의 상태를 조정하는 명령어.  
> 커밋을 취소하거나, 스테이징 영역에서 파일을 제거하거나, 변경 사항을 되돌리기 위해 사용.


## 기본 역할

- `git reset`은 현재 브랜치의 **HEAD 포인터**를 지정한 커밋으로 이동 시킨다.
여러 옵션이 존재하는데 해당 옵션에 따라 staging area와 working diectory가 어떻게 변경되는지가 달라진다.

## git reset의 세가지 주요 옵션


### —soft


> `git reset --soft [commit ID]`  
>   
> 최근 커밋을 취소하지만, staging 된 파일과 working diectory의 변경 사항은 그대로 유지 한다.


✅ 최근 커밋을 잘못했지만, 스테이징된 파일을 다시 커밋하고 싶을 때 유용함.


### —mixed(기본값)


> `git reset —mixed [commit ID]`   
>   
> HEAD 포인터를 지정 커밋으로 이동시키고, 스테이징 영역을 비움.  
> 그러나 working diectory의 변경사항은 유지 됨.


✅ 스테이징 영역에 있던 파일을 다시 스테이징 하지 않고, 변경 사항만 남긴 채로 되돌리고 싶을 때 사용


### —hard


> `git reset —hard [commit ID]`   
>   
> HEAD를 이동시키고, 스테이징 영역과 워킹 디렉토리 모두를 지정된 커밋 상태로 되돌림


✅ 현재 작업 중인 모든 변경 사항과 스테이징된 파일을 포함해 완전히 이전 상태로 되돌리고 싶을 때 사용.


⚠️ 해당 명령어를 실행 시 이전으로 되돌릴 수 없으니 주의해서 사용하자.

- `git reset —hard HEAD~1` : 마지막 커밋을 완전히 삭제 후, 워킹 디렉토리와 스테이징 영역을 이전 커밋 상태로 되돌리는 명령어

## git reset의 다른 옵션


### `git reset HEAD [파일명]`

- 특정 파일을 스테이징 영역에서만 제거하고, 워킹 디렉토리 변경사항은 유지.
- 해당 방법은 파일을 커밋하지 않도록 스테이징에서만 제외하고 싶을 때 유용하다.

### 사용 예시

1. 마지막 커밋을 수정하고 싶을 때:

```shell
git reset --soft HEAD~1
```


▶️ 최근 커밋을 취소하고, 변경 사항을 그대로 남겨둔 상태에서 다시 커밋 할 수 있다.

1. 스테이징 영역을 초기화하고 변경 사항을 유지할 때:

```shell
git reset --mixed HEAD~2
```


▶️ 두번째 이전 커밋으로 돌아가고, 스테이징 영역을 비우지만 워킹 디렉토리의 변경 사항은 유지 한다.

1. 현재 작업 중인 모든 것을 취소하고 이전 상태로 되돌릴 때:

```shell
git reset --hard HEAD~1
```


▶️ 마지막 커밋을 삭제하고, 스테이징 된 파일과 변경된 파일을 모두 원래 상태로 되돌림.


### 표 정리


| 옵션     | 스테이징 영역 변경   | 워킹 디렉토리 변경     |
| ------ | ------------ | -------------- |
| —soft  | 변경 없음        | 변경 없음          |
| —mixed | 초기화(스테이징 해제) | 변경 없음          |
| —hard  | 초기화(스테이징 해제) | 변경됨(이전 커밋 상태로) |


# pull, push, clone


> 위 명령어들은 **원격 레포지토리(remote repository)**와 **로컬 레포지토리(local repository)** 간의 상호작용에 중요한 역할을 하는 명령어들 이다.


## git pull


```shell
git pull origin [브랜치 명]
```


`git pull`은 원격 저장소의 변경 사항을 가져와서 로컬 저장소에 병합하는 명령어이다.
즉, **원격 저장소에서** <u>**최신 변경 사항을 가져오고**</u> **이를** <u>**현재 브랜치에 반영**</u>하는 데 사용됩니다.


🤔 **git pull의 동작 방식**

1. `git fetch` : 원격 저장소의 최신 상태를 로컬로 가져옴. 이때 변경사항이 바로 병합되지는 않음
2. `git merge` : 가져온 변경 사항을 현재 로컬 브랜치에 병합

### git fetch



`git fetch`는 Git에서 원격 저장소(Remote Repository)의 변경 사항을 가져오는 명령어.
이 명령어는 원격 저장소에 있는 최신 커밋, 브랜치, 태그 등의 정보를 로컬로 업데이트하지만, 이 정보를 <u>**로컬 브랜치에 자동으로 병합하지는 않습니다.**</u>


💡 `git fetch`의 주요 특징

1. **원격 변경 사항 확인**: 원격 저장소에 새로운 커밋이나 브랜치가 추가되었는지 확인할 수 있습니다.
2. **로컬 브랜치에 영향을 주지 않음**: `git fetch`는 원격 저장소의 변경 사항을 가져오기만 하고, 로컬 브랜치에 적용하지 않습니다. 병합(Merge) 작업은 수동으로 해야 합니다.
3. **작업 안전성**: 작업 중이던 내용에 영향을 주지 않고 원격 저장소의 최신 상태를 가져오기 때문에 안전하게 사용할 수 있습니다.

 `git fetch` 사용 방법


```shell
git fetch <원격저장소명>
```


보통 원격 저장소 이름은 `origin`으로 설정되어 있습니다.


**예시**:


```shell
git fetch origin
```


이 명령어는 원격 저장소 `origin`에서 모든 최신 변경 사항을 가져와 로컬로 업데이트합니다.


### `git fetch`와 `git pull`의 차이

- **`git fetch`**: 원격 저장소의 변경 사항을 로컬로 가져오지만, <u>자동으로 병합하지 않습니다</u>.
- **`git pull`**: 원격 저장소의 변경 사항을 가져오고, <u>자동으로 현재 로컬 브랜치에 병합</u>합니다. (`git fetch + git merge`를 함께 실행하는 명령어)

### `git fetch` 후 병합


`git fetch`로 원격 저장소의 변경 사항을 가져온 후, <u>**수동으로 병합해야 로컬 브랜치에 반영됩니다.**</u> 병합은 `git merge` 명령어를 통해 이루어집니다.


**예시**:


```shell
git fetch origin
git merge origin/main
```


이 명령어는 원격 `origin`의 `main` 브랜치에서 변경 사항을 가져와 로컬 브랜치에 병합합니다.


## git push


```shell
git push origin [브랜치 명]
```


`git push`는 로컬에서 작업한 변경 사항을 원격 저장소로 업로드하는 명령어이다.
즉, <u>**로컬 저장소의 커밋을 원격 저장소로 전송하여 원격 저장소를 최신 상태로 갱신**</u>합니다.


🤔 **git push의 주의사항**


⚠️ 원격 저장소에서 특정 브랜치에 변경 사항이 생긴 경우, git push 전에 git pull을 해야 충돌을 방지할 수 있습니다.


### git push의 추가 옵션


```shell
git push -u origin [branch]
```


▶️ 원격 저장소와 로컬 브랜치를 연결하고, 이후부터는 git push만으로 간편하게 푸시할 수 있다.


## git clone


```shell
git clone https://github.com/username/repository.git
```

- `git clone`은 원격 저장소를 <u>**로컬로 복제하는 명령어**</u>입니다.
- 이 명령어는 <u>**원격 저장소의 전체 내용을 복사하여 로컬에 새로 저장소를 생성**</u>합니다.
- `git clone`은 원격 저장소의 모든 커밋과 브랜치, 파일을 포함한 완전한 복사본을 로컬에 만듭니다.

# 커밋(commit)


```shell
git commit -m "Fix bug in feature X"
```

- 커밋(Commit)은 **프로젝트의 변경 사항을 저장**하는 스냅샷입니다.
- 커밋은 Git이 <u>**파일의 변경 이력을 관리하는 기본 단위**</u>이며, 프로젝트의 특정 시점에서 어떤 변경 사항이 있었는지를 기록합니다.

### 커밋의 메세지 작성 원칙

- **커밋 메시지는 간결하고 명확하게 작성**해야 합니다.
- **제목은 50자 이내**로, **현재 시제**로 작성합니다.
- 필요할 경우 **본문과 푸터**를 추가해 더 상세한 정보를 제공합니다.
- **하나의 커밋은 하나의 작업**에만 집중합니다.

```css
feat : 새로운 기능에 대한 커밋
fix : 버그 수정
chore : 각종 세팅 관련 커밋
docs : 문서 수정에 대한 커밋
style : 코드 스타일(줄 바꿈) 혹은 포맷 등에 관한 커밋
refactor : 코드 리팩토링에 대한 커밋
test : 테스트 코드 수정에 대한 커밋
ci : CI 관련 설정 수정에 대한 커밋
perf: 성능 향상을 위한 변경사항에 대한 커밋
build: 빌드 관련 설정 변경에 대한 커밋
revert: 이전 커밋을 되돌리는 커밋
merge: 브랜치 병합에 대한 커밋
release: 새로운 릴리스 또는 버전에 대한 커밋
deps: 종속성 또는 패키지 업데이트에 대한 커밋
security: 보안 관련 변경사항에 대한 커밋
```


### 커밋 과정

1. **파일 수정**
2. **스테이징**: `git add`로 수정한 파일을 스테이징 영역에 추가합니다.
3. **커밋**: `git commit` 명령어로 스테이징된 파일을 커밋하고, 해당 변경 사항을 기록합니다.

### `git log --pretty=oneline`


```shell
git log --pretty=oneline
```


`git log`는 Git <u>히스토리를 확인할 때 사용하는 명령어</u>입니다.


기본적으로 커밋 이력에 대한 정보를 자세히 보여주지만, `--pretty=oneline` 옵션을 사용하면 각 커밋을 한 줄씩 요약하여 보여줍니다.


### `git config alias.[별명] [커맨드]`


```shell
git config --global alias.history 'log --pretty=oneline'
```


 자주 사용하는 명령어에 별명을 붙여 더 짧게 사용할 수 있는 기능.


🔍 위 명령어는 `git log --pretty=oneline` 명령어에 `history`라는 별칭을 붙이는 예시입니다. 이제 `git history`라고만 입력해도 `git log --pretty=oneline` 명령어가 실행됩니다.


## `git diff`


`git diff`는 Git에서 <u>파일의 변경 사항을 비교할 때 사용하는 명령어</u>입니다.


현재 워킹 디렉토리와 스테이징된 파일, 또는 두 커밋 사이의 차이를 시각적으로 보여줍니다.

1. **현재 변경 사항 비교** : 현재 수정된 파일과 마지막 커밋된 버전 간의 차이를 보여줍니다.

    ```shell
    git diff
    ```

2. **스테이징된 변경 사항 비교** : 스테이징된 파일과 마지막 커밋된 버전 간의 차이를 보여줍니다.

    ```shell
    git diff --staged
    ```

3. **두 커밋 간의 차이 비교**: 두 커밋 사이의 변경 사항을 비교합니다.

    ```shell
    git diff <커밋ID1> <커밋ID2>
    ```


## `git tag`


`git tag`는 프로젝트의 <u>**특정 커밋에 태그를 붙여 명확히 표시할 때 사용하는 명령어**</u>입니다.


보통 릴리스 버전을 관리할 때 많이 사용됩니다.


태그는 커밋에 고정된 "라벨"을 붙여 나중에 쉽게 참조할 수 있게 합니다.

1. **새로운 태그 생성**: 이 명령어는 현재 커밋에 `v1.0.0`이라는 태그를 생성합니다.

    ```shell
    git tag v1.0.0
    ```

2. **특정 커밋에 태그 추가**:  `<커밋ID>`는 해당 커밋의 해시입니다.

    ```shell
    git tag v1.0.0 <커밋ID>
    ```

3. **태그 확인**: 현재 리포지토리에서 설정된 모든 태그를 나열합니다.

    ```shell
    git tag
    ```

4. **태그를 원격 저장소에 푸시**:

    ```shell
    git push origin v1.0.0
    ```

5. **모든 태그를 원격 저장소로 푸시**:

    ```shell
    git push origin --tags
    ```

## **`git blame`**


```shell
git blame <파일명>
```


`git blame`은 파일의 각 줄이 언제, 누구에 의해 마지막으로 수정되었는지를 보여주는 명령어입니다.


##  **`git revert`**


```shell
git revert <커밋ID>
```


`git revert`는 특정 커밋의 변경 사항을 **되돌리는(undo)** 명령어입니다.


하지만, 단순히 변경 사항을 삭제하는 것이 아니라, 해당 커밋의 **반대 작업**을 새로운 커밋으로 기록하여 되돌립니다. 이 때문에 <u>**이전 커밋 기록은 유지**</u><u>하면서 해당 커밋을 취소할 수 있습니다.</u>


### `git revert`의 특징:

- 기존의 커밋 기록은 그대로 유지됩니다.
- 커밋을 취소하는 새로운 커밋이 생성됩니다.
- 협업 중 실수로 잘못된 커밋을 되돌릴 때 매우 유용합니다.
- **`git revert --no-commit`**: 변경 사항을 바로 커밋하지 않고 스테이징 영역에만 적용합니다. 이 후 원하는 대로 수정한 후에 커밋할 수 있습니다.

### `git revert`와 `git reset`의 차이:

- **`git revert`**: 특정 커밋의 변경 사항을 <u>**취소하면서도 기존의 커밋 기록은 유지**</u>합니다. 협업 중에 커밋을 되돌리기에 안전한 방법입니다.
- **`git reset`**: 커밋을 되돌리거나 아예 삭제할 수 있습니다. 하지만 <u>**기존 커밋 기록이 변경되기 때문에 히스토리가 수정**</u>됩니다. 협업 시 주의가 필요합니다.

##  **`git reflog`**


```shell
git reflog
```


**HEAD**(현재 브랜치의 커밋 위치)의 모든 이동 기록을 추적하는 명령어입니다.


**커밋, 체크아웃, 리셋, 병합 등의 모든 활동을 기록**하여, 특정 커밋이나 상태로 쉽게 돌아갈 수 있게 도와줍니다.

- **모든 활동 추적**: 브랜치 이동, 커밋, 병합, 리셋 등의 모든 히스토리를 로컬에서 추적합니다.
- **복구 기능**: 실수로 커밋을 삭제하거나, 브랜치를 잘못 변경한 경우에도 `git reflog`를 통해 과거로 돌아가 복구할 수 있습니다.
- **로컬 기록**: `git reflog`는 로컬에서만 관리되며, 원격 저장소에는 영향을 미치지 않습니다.

### 복구 예시


만약 최근 작업을 실수로 잘못된 브랜치에서 했다면, `git reflog`를 통해 HEAD가 이동했던 시점으로 되돌아갈 수 있습니다.


```shell
git reset --hard HEAD@{2}
```


## **`git cherry-pick`**


```shell
git cherry-pick <커밋ID>
```


 Git에서 특정 커밋을 선택적으로 다른 브랜치에 적용하는 명령어입니다.


<u>**특정 커밋만 가져와서 현재 브랜치에 반영하고 싶을 때**</u> 매우 유용합니다. 


여러 커밋 중에서 중요한 변경 사항 하나만 다른 브랜치에 적용할 때 사용됩니다.

- **선택적 병합**: 병합을 하지 않고도 특정 커밋을 현재 브랜치에 반영할 수 있습니다.
- **커밋 유지**: 원본 커밋의 변경 사항을 새로운 커밋으로 복사하여 반영하므로, 커밋 기록이 그대로 유지됩니다.

### 여러 커밋을 한꺼번에 적용


```shell
git cherry-pick <커밋ID1> <커밋ID2>
```


여러 커밋을 한꺼번에 적용할 수 있습니다.


### 범위를 지정해서 여러 커밋 적용


```shell
git cherry-pick <커밋ID1>..<커밋ID2>
```


이 명령어는 `커밋ID1`부터 `커밋ID2`까지의 커밋을 순차적으로 현재 브랜치에 적용합니다.


### 충돌 처리


`git cherry-pick` 도중 충돌이 발생할 수 있습니다. 충돌이 발생하면 Git이 충돌된 파일을 보여주고, 충돌을 해결한 후에 커밋을 완료해야 합니다.


```shell
git cherry-pick --continue
```


충돌 해결 후 위 명령어를 사용해 cherry-pick 작업을 계속 진행할 수 있습니다.


### cherry-pick 중단


cherry-pick 도중 작업을 취소하려면 다음 명령어를 사용합니다.


```shell
git cherry-pick --abort
```


---


### **`git cherry-pick`** **vs** **`git merge`**

- **`git cherry-pick`**: 특정 커밋을 선택적으로 가져와서 현재 브랜치에 적용합니다. 여러 커밋 중 일부만 반영하고 싶을 때 유용합니다.
- **`git merge`**: 한 브랜치의 모든 변경 사항을 다른 브랜치에 병합합니다. 선택적 적용은 불가능합니다.

## `git commit --amend` 

- 최근 커밋의 메세지를 수정하거나, 스테이징 된 변경사항을 추가하는 데 사용.

# 브랜치(Branch)


브랜치(Branch)는 프로젝트의 독립적인 작업 공간을 의미합니다.


## 1. 브랜치의 기본 개념


브랜치는 Git에서 **커밋의 흐름을 분리**하는 역할을 합니다.


🔍  브랜치를 사용하면 기존 코드에 영향을 주지 않고 새로운 기능 개발, 버그 수정, 실험 등을 진행할 수 있습니다. 이 작업이 완료되면 브랜치를 병합(Merge)하여 메인 브랜치로 합칠 수 있습니다.


🔍 Git에서 기본 브랜치는 보통 `main`(또는 `master`)이라고 불리며, 프로젝트의 주요 코드가 이 브랜치에서 관리됩니다.


🔍  개발자는 이 `main` 브랜치에서 새로운 브랜치를 생성하여 독립적으로 작업한 후, 다시 `main` 브랜치에 병합할 수 있습니다.


## 2. 브랜치 관련 명령어


### 1. 브랜치 생성


브랜치를 생성하면 현재 커밋 상태를 기준으로 새로운 작업 공간이 만들어집니다. 이는 기존 브랜치의 커밋 기록을 모두 포함하고 있습니다.


```shell
git branch <브랜치명>
```


### 2. 브랜치 이동


새로 생성한 브랜치로 이동하려면 `git checkout` 또는 `git switch` 명령어를 사용합니다.


```shell
git checkout <브랜치명>
```


또는


```shell
git switch <브랜치명>
```


### 3. 브랜치 확인


현재 저장소에 존재하는 모든 브랜치를 확인하려면 `git branch` 명령어를 사용합니다.


```shell
git branch
```


### 4. 브랜치 병합


브랜치에서 작업한 내용을 다른 브랜치에 병합할 때는 `git merge` 명령어를 사용합니다.


```shell
git checkout main
git merge <병합할 브랜치명>
```


### 5. 브랜치 삭제


작업이 완료되어 더 이상 필요 없는 브랜치는 삭제할 수 있습니다.


```shell
git branch -d <브랜치명>
```


💡 만약 브랜치가 병합되지 않은 상태에서 강제로 삭제하려면 `-D` 옵션을 사용합니다.


### 6. 브랜치 목록과 현재 브랜치 확인


현재 존재하는 모든 브랜치와 현재 체크아웃된 브랜치를 확인하려면 `git branch`를 사용합니다.


```shell
git branch
```


출력 예시:


```shell
main
* feature/new-feature
```


현재 선택된 브랜치는 `*`로 표시됩니다.


## 3. 브랜치 전략


### 1. Git Flow


Git Flow는 기능 개발, 릴리스, 버그 수정을 관리하는 데 사용되는 브랜칭 전략입니다. 보통 다음과 같은 브랜치들이 존재합니다:

- **main**: 최종 배포된 코드를 관리하는 브랜치.
- **develop**: 개발 중인 기능이 모여있는 브랜치.
- **feature/**: 기능 개발을 위한 브랜치.
- **release/**: 릴리스를 준비하기 위한 브랜치.
- **hotfix/**: 긴급 버그 수정을 위한 브랜치.

## 4. 브랜치 병합 


### 1. git merge

- `merge`는 두 개 이상의 브랜치에서 발생한 변경 사항을 하나로 합치는 명령어입니다.
1. **두 브랜치 병합**: 두 브랜치의 변경 사항을 하나로 합칩니다.
2. **새로운 기능 통합**: 기능 개발이나 버그 수정 작업이 완료되면 이를 메인 브랜치로 병합해 프로젝트에 반영합니다.

💡 `git merge` 사용 방법

1. **병합하려는 브랜치로 이동**:

    ```shell
    git checkout main
    ```

2. **다른 브랜치를 병합**:
위 명령어는 `feature/new-feature` 브랜치를 현재 체크아웃된 `main` 브랜치에 병합합니다.

    ```shell
    git merge feature/new-feature
    ```


### 2. Rebase


Rebase는 병합 과정에서 발생하는 복잡한 이력을 피하기 위해 사용하는 방법입니다. 한 브랜치의 커밋을 다른 브랜치의 최신 상태 위에 "다시 쌓는" 방식으로, 깔끔한 이력 관리를 가능하게 합니다.


```shell
git checkout feature/new-feature
git rebase main
```


위 명령어는 `main` 브랜치의 최신 커밋을 기반으로 `feature/new-feature` 브랜치의 커밋을 다시 쌓습니다.

- 실무에서는 해당 rebase 명령어를 vscode에서 머지와 함께 브랜치 삭제를 할 수 있도록 설정해서 사용하면 편리하게 사용 할 수 있다.

## 5. 브랜치의 병합 충돌 시


### `git merge --abort`


`git merge --abort`는 **병합(Merge)** 과정에서 문제가 발생하거나 충돌이 생겼을 때 병합을 중단하고, 병합 이전의 상태로 되돌리기 위해 사용하는 명령어입니다.


병합 도중에 변경된 내용이 마음에 들지 않거나 충돌을 해결하기 어렵다고 판단되면 이 명령어로 병합을 취소할 수 있습니다.


### 동작 방식

- 병합 도중에 변경된 스테이징된 파일들과 워킹 디렉토리의 상태를 병합 전의 상태로 복구합니다.
- 병합을 완전히 중단하며, <u>**병합을 시도하기 전 상태**</u><u>로 돌아갑니다.</u>
- ⚠️ `git merge --abort`는 병합 도중 <u>**수정되지 않은 모든 파일을 병합 전 상태**</u>로 되돌리지만, 병합 중 <u>**수정된 파일**</u><u>은 복구되지 않기 때문에 이미 편집된 파일은 수동으로 수정해야 할 수 있습니다.</u>

### `git merge --abort`가 사용되는 상황

1. 병합 충돌(Merge Conflict)이 발생했을 때: 병합 도중 두 브랜치에서 같은 파일의 동일한 부분을 다르게 수정했을 때 충돌이 발생합니다. 이 충돌을 해결할 수 없다면, `git merge --abort`를 사용하여 병합을 취소할 수 있습니다.
2. **병합 결과가 원치 않을 때**: 병합 도중 예상치 못한 결과가 나왔거나 실수로 병합을 시도한 경우에도 `git merge --abort`로 병합을 취소할 수 있습니다.

### `git merge --abort`와 유사한 명령어

- **`git reset --hard`**: `git merge --abort`는 병합 중에만 사용할 수 있지만, `git reset --hard`는 병합 여부와 상관없이 워킹 디렉토리와 스테이징된 파일을 특정 커밋으로 되돌릴 수 있습니다. 병합 과정 외에도 되돌리기가 필요할 때 유용합니다.

# stash


```shell
git stash
```

- Git에서 현재 작업 중인 변경 사항을 임시로 저장하고, **워킹 디렉토리**를 깨끗한 상태로 되돌리는 명령어입니다.
- 이 기능을 사용하면 로컬에서 작업하던 내용을 커밋하지 않고도 **브랜치를 전환**하거나 **다른 작업을 우선 처리**할 수 있습니다. 나중에 저장한 변경 사항을 다시 불러와 이어서 작업할 수 있는 기능입니다.
- **스테이징된 파일과 스테이징되지 않은 파일** 모두 임시로 저장되고, 워킹 디렉토리는 마지막 커밋 상태로 초기화됩니다.

### `git stash`의 주요 특징

1. **작업 내용 임시 저장**: 변경 사항을 임시로 저장하고 워킹 디렉토리를 깨끗하게 초기화합니다.
2. **변경 사항 복원**: 필요할 때 저장해둔 변경 사항을 다시 불러와서 작업을 이어갈 수 있습니다.
3. **작업 환경 유지**: 작업 도중 커밋하지 않은 상태로 브랜치를 전환하거나 다른 작업을 처리할 수 있게 도와줍니다.

### 1. `git stash list`로 저장된 변경 사항 목록 확인


```shell
git stash list
```


저장한 스태시의 목록을 확인할 수 있습니다. 각 스태시는 고유한 이름(`stash@{n}`)을 가지며, 이를 통해 여러 스태시 중 원하는 스태시를 선택할 수 있습니다.


### 3. `git stash apply`로 저장된 변경 사항 복원


```shell
git stash apply
```


가장 최근에 저장된 스태시를 현재 워킹 디렉토리로 적용하여, 변경 사항을 다시 복원합니다. **하지만 스태시 목록에서 해당 항목은 삭제되지 않습니다**.


특정 스태시를 복원하려면 스태시 이름을 명시할 수 있습니다:


```shell
git stash apply stash@{1}
```


### 4. `git stash pop`으로 스태시 복원 및 삭제


```shell
git stash pop
```


스태시를 복원하는 동시에 스태시 목록에서 해당 항목을 삭제합니다. 즉, 한 번 사용하면 더 이상 해당 스태시는 남아 있지 않습니다.


### 5. `git stash drop`으로 특정 스태시 삭제


```shell
git stash drop stash@{0}
```


스태시 목록에서 특정 스태시를 삭제합니다. 스태시를 복원하지 않고 삭제할 수 있습니다.


### 6. `git stash clear`로 모든 스태시 삭제


```shell
git stash clear
```


모든 스태시를 한 번에 삭제할 수 있습니다.

### `git stash`의 주요 사용 사례

1. **브랜치 전환 시 임시 저장**
작업 도중 브랜치를 전환해야 하지만 아직 커밋할 준비가 되지 않았을 때 `git stash`를 사용하여 변경 사항을 임시로 저장한 후, 다른 브랜치로 전환할 수 있습니다.

    ```shell
    git stash
    git checkout <다른 브랜치>
    ```

2. **긴급한 버그 수정 처리**
새로운 기능을 개발 중에 긴급하게 버그를 수정해야 할 때, 작업 중인 변경 사항을 스태시로 임시 저장하고, 버그 수정 후 다시 작업을 이어갈 수 있습니다.
3. **작업 내용을 여러 번 스태시로 저장**
여러 작업을 하던 중, 각기 다른 변경 사항을 여러 번 스태시로 저장해두고, 필요할 때 선택적으로 복원할 수 있습니다.

### `git stash` 옵션

1. **스테이징된 변경 사항만 스태시 저장**:

    ```shell
    git stash --keep-index
    ```


    스테이징된 변경 사항을 그대로 유지하면서 스테이징되지 않은 변경 사항만 스태시로 저장합니다.

2. **새 파일까지 포함해 스태시 저장**:
기본적으로 새로 추가된 파일은 스태시로 저장되지 않습니다. 그러나 `u` 옵션을 사용하면 추적되지 않은 파일까지 스태시로 저장할 수 있습니다.

    ```shell
    git stash -u
    ```

3. *모든 파일(추적된 파일 + 추적되지 않은 파일)**을 스태시로 저장:

    ```shell
    git stash -a
    ```
