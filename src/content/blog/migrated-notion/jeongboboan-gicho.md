# **컴퓨터 구조와 운영체제의 기본 원리**


이 챕터의 목표는 다음 두 가지입니다:

---


# **1. 컴퓨터의 구성요소(하드웨어 기본)**


컴퓨터는 내부적으로 크게 “4개의 핵심 부품”으로 구성됩니다.


## **1) CPU**


**Central Processing Unit**, 즉 “연산 담당자”.


쉽게 말하면 **두뇌**입니다.


모든 계산, 로직, 프로그램 실행을 담당합니다.


예:

- 코드에서 1 + 1 계산
- “이 파일을 열어라”, “이것을 네트워크로 보내라” 같은 명령 실행
- 프로그램이 어떤 순서로 실행될지 결정

---


## **2) 메모리(RAM)**


**Random Access Memory**.


프로그램이 “잠깐 사용하는 임시 작업 공간”.


비유하면 **책상 위 작업 공간**이에요.


작업 중인 파일, 변수, 요청, 실행 중인 프로그램들이 잠깐 저장됩니다.


특징:

- 컴퓨터 꺼지면 내용 사라짐 (휘발성)
- 빠르지만 비쌈 → 용량 제한 있음

---


## **3) 저장장치(SSD/HDD)**


비유하면 **서랍장**.


전원이 꺼져도 남아있어야 하는 정보가 저장되는 곳입니다.


예:

- 사진
- 게임 파일
- DB 데이터
- 운영체제(OS) 자체

---


## **4) 입출력 장치(I/O)**


**Input/Output**.


예: 키보드, 마우스, 모니터, 네트워크 카드, 마이크, 웹캠


---


# **2. 운영체제(OS)의 역할**


OS(운영체제)는 **이 모든 하드웨어를 효율적으로 관리하고 프로그램이 충돌 없이 실행되도록 도와주는 소프트웨어**입니다.


OS가 없다면?

- 브라우저를 띄우려면

    직접 CPU에게 “이 파일을 메모리에 올려” 같은 명령을 계속 보내야 합니다.

- 프로그램 두 개가 동시에 같은 메모리를 쓰면 충돌하고 컴퓨터 다운

OS는 이런 문제들을 모두 자동으로 해결합니다.


---


운영체제가 하는 4가지 핵심 업무를 기억하세요.


## **1) 프로세스 관리**


프로그램을 실행하면 “프로세스”라는 단위로 관리합니다.


OS는 다음을 관리합니다:

- 어떤 프로세스를 먼저 실행할지
- 어떤 프로세스는 CPU 시간을 얼마나 가져갈지
- 충돌 없이 여러 개를 동시에 돌릴 수 있도록 조율

예)


유튜브 보면서 디스코드 켜고, 크롬 여러 탭 켜고 → 이걸 모두 OS가 조율함


---


## **2) 메모리 관리**


프로그램은 RAM을 필요로 합니다.


OS는:

- 어떤 프로세스에 RAM을 얼마나 할당할지
- 다른 프로세스가 맞는 공간을 침범하지 않도록 보호

보안에서 이 영역은 매우 중요함 (메모리 오염, 버퍼 오버플로 등)


---


## **3) 파일 시스템 관리**


SSD/HDD에 저장된 파일을 보호하고 정리하는 역할.


예:

- 디렉토리 구조 생성
- 파일 읽기/쓰기 권한 관리
- 삭제/복사/이동 같은 기능 제공

나중에 시스템 해킹에서 “파일 권한 구조”가 아주 중요함.


---


## **4) 입출력 장치 관리**

- 키보드 입력
- 네트워크 카드 동작
- 모니터 출력
- USB 연결

모두 OS가 드라이버를 통해 관리합니다.


---


# **3. 프로그램이 실행된다는 것의 의미**


이 부분을 정확히 이해하면 나중에 **프로세스, 포트, 서버, 네트워크 통신** 등을 훨씬 쉽게 이해하게 됩니다.


## **프로그램 실행 과정 (쉽게 설명)**

1. 저장장치(SSD)에 프로그램 파일이 있다.
2. OS가 이 파일을 읽어 RAM으로 올린다.
3. CPU가 명령어를 하나하나 읽으면서 실행한다.
4. 실행 중 필요한 데이터를 계속 RAM에서 주고받는다.

이 단계를 통해 프로그램이 실행되고, 이를 OS가 관리하면서 여러 프로그램이 동시에 돌아가는 것이다.

# **4. 정보보안에서 이 챕터가 왜 중요한가?**


웹 취약점 분석가라도 **운영체제 지식은 필수**입니다.


이유는:

1. 공격을 하면 결국 OS의 동작을 눈으로 봐야 한다.
- 로그
- 프로세스
- 파일 생성 여부
- 권한 상승 여부

    전부 OS 관점에서 확인해야 함

1. 웹 서버는 99%가 Linux 위에서 돌아간다

    즉, OS 기본기가 없으면 서버 해킹 분석 불가

2. 공격을 하든 방어를 하든

    “프로그램이 어떻게 실행되고 메모리·파일을 어떻게 쓰는지” 이해해야


    이상 행동을 발견할 수 있다.

# **5. 이 챕터에서 배운 ‘핵심 요약’**

1. 컴퓨터는 CPU, 메모리, 저장장치, I/O 장치로 구성
2. 운영체제(OS)는
    - 프로세스

    - 파일 시스템

3. 프로그램 실행은 “저장장치 → 메모리 → CPU” 흐름
4. 정보보안에서 OS 이해는 필수 기반

# **2. 리눅스 파일 구조를 먼저 이해하자**


리눅스는 “드라이브 C:, D:” 같은 개념이 없습니다.


모든 파일은 **루트(/)** 아래에 트리 구조로 존재합니다.


아래 구조는 반드시 기억해야 하는 필수 디렉토리들입니다.


## **`/`**


루트. 모든 경로의 시작점.


## **`/home`**


일반 사용자들의 개인 폴더


예: /home/junghwa


윈도우의 “내 문서”에 해당함.


## **`/root`**


리눅스의 최상위 관리자 계정(root)의 홈 디렉토리.


일반 사용자 홈과는 다릅니다.


## **`/etc`**


리눅스의 모든 **설정 파일** 모음.


초보자들이 가장 많이 건드리는 경로.


예:

- /etc/passwd (계정 정보)
- /etc/ssh/sshd_config (SSH 설정)

## **`/bin`****,****`/sbin`**


bin = binary → 운영에 필요한 기본 명령어들


예: ls, cat, cp, mv


sbin = system binary → 관리자용 명령어


## **`/usr`**


사용자가 설치한 프로그램들이 들어감


리눅스의 “Program Files” 느낌


## **`/var`**


변화(variable)라는 의미.


로그 파일, 웹 서버 파일, DB 파일 등 변화가 많은 데이터 저장.


예:

- /var/log/ → 시스템 로그
- /var/www/ → 웹 서버 루트 디렉토리(NGINX/Apache)

## **`/tmp`**


일시적 파일. 재부팅되면 사라질 수 있음.


웹 취약점에서 자주 악용되는 경로.

# **4. 리눅스 권한 구조 (입문자 수준 정리)**


리눅스에서는 “누가 어떤 파일에 접근할 수 있는가?”가 아주 중요합니다.


파일을 보면 이런 식으로 나옵니다:


```plain text
drwxr-xr--
```


이걸 3칸씩 나눠서 읽습니다:

- 첫 번째 3칸 = owner(파일 주인)
- 두 번째 3칸 = group(같은 그룹 사용자들)
- 세 번째 3칸 = other(그 외 모두)

r = 읽기


w = 쓰기


x = 실행


chmod로 권한 수정 가능:


```plain text
chmod 755 filename
chmod 644 filename
```


chmod는 뒤에서 더 깊게 다루지만 지금은 “읽기/쓰기/실행 권한”이 존재한다는 개념만 이해하면 됩니다.

# **5. 파일 권한을 확인하고 해석하는 법**


권한은 나중에 “권한 상승 공격”, “웹셸 업로드 공격”과 연결되므로


지금 기초를 정확히 잡는 것이 중요합니다.


```bash
ls -l
```


출력 예:


```bash
-rw-r--r-- 1 root root  4096 Jan 12 config.cfg
```


해석 방법:

- rw-r--r-- = 권한
- 첫 문자 -는 일반 파일
- rw- = owner(소유자) 권한 (읽기+쓰기)
- r-- = group 권한 (읽기)
- r-- = other 권한 (읽기)
- root root = 소유자, 그룹

# **1.** **OSI 7계층****을 왜 알아야 할까?**


보안 전문가가 OSI 7계층을 반드시 아는 이유는 간단합니다:

- 공격이 어디 계층에서 일어나는지 정확하게 말할 수 있어야 함
- 패킷 분석을 할 때 어느 계층 흐름인지 해석할 수 있어야 함
- 방화벽/게이트웨이/웹서버가 각각 어느 계층에서 동작하는지 이해해야 함
<details>
<summary>게이트웨이란?</summary>

**게이트웨이(Gateway)**는 서로 다른 네트워크 간을 연결해주는 장치 또는 시스템을 말합니다.


쉽게 비유하면:

- 두 나라 사이의 "국경 검문소" 같은 역할
- 서로 다른 프로토콜이나 네트워크 구조를 번역하고 연결해줌

**주요 기능:**

- 내부 네트워크와 외부 인터넷을 연결
- 서로 다른 통신 규약(프로토콜)을 변환
- 패킷의 출입을 제어하고 라우팅

**OSI 계층 관점:**


게이트웨이는 여러 계층에서 동작할 수 있지만, 일반적으로 **L3(네트워크 계층)** 이상에서 작동합니다. 가장 흔한 예는 홈 라우터인데, 이것이 바로 집 안 네트워크와 ISP(인터넷 서비스 제공자) 네트워크를 연결하는 게이트웨이입니다.


**보안 관점:**


게이트웨이는 네트워크 경계에 위치하기 때문에 방화벽, 침입 탐지 시스템(IDS) 등 보안 장비가 함께 배치되는 중요한 지점입니다.


</details>


예를 들어,

- DDoS 공격 → L3, L4
- TCP SYN Flood → L4
- XSS, SQL Injection → L7 (애플리케이션 계층)
- SSL/TLS 문제 → L6

따라서 OSI는 **공격·방어를 설명하는 언어**입니다.

# **2. OSI 7계층을 쉽게 이해하는 방법 (비유 기반)**


OSI 계층은 총 7단계입니다.


아래 비유는 인터넷을 “편지를 보내는 과정”으로 생각한 것입니다.


---


## **L1. 물리 계층 (Physical Layer)**


실제로 전기 신호가 흐르는 케이블 레벨입니다.


비유: 편지를 들고 뛰어다니는 사람은 신발·도로 기반이 있어야 한다. 이 레벨이 인프라.


예: LAN 케이블, Wi-Fi 전파


---


## **L2. 데이터 링크 계층**


같은 네트워크 안에서 “장치 간” 통신을 담당.


MAC 주소가 등장합니다.


비유: 집 안에서 방 ↔ 방, 층 ↔ 층 사이의 배달.


예: 스위치, MAC 주소, ARP


---


## **L3. 네트워크 계층**


멀리 떨어진 네트워크까지 가는 길을 찾는 계층.


**라우팅**, **IP 주소**가 나온다.


비유: 편지를 보내면 우편집배원이 어느 도시로 보내야 할지 길을 찾는 단계.


예: 라우터, IP 주소


---


## **L4. 전송 계층**


전달의 “신뢰성”을 관리.


여기서 **TCP**와 **UDP**가 등장합니다.


비유: 택배를 보낼 때

- 안전하게 보낼지(TCP)
- 빠르게 보낼지(UDP) 결정하는 단계.

---


## **L5. 세션 계층**


통신을 “지속적으로 유지”하거나 “끊는” 역할.


요즘은 L4/L7 개념에 통합되어 설명하는 경우도 많음.


비유: 전화 통화를 연결하고 끊는 기능.


---


## **L6. 표현 계층**


데이터를 특정 형식으로 변환.


암호화, 압축 등이 포함.


비유: 문서를 영어 → 한국어로 번역하거나, 압축하는 단계.


예: TLS/SSL 일부


---


## **L7. 애플리케이션 계층 (가장 중요한 계층)**


우리 눈에 보이는 모든 인터넷 서비스가 여기 있습니다.


예: HTTP, HTTPS, DNS, FTP, SMTP, API 요청 등등


비유: 실제로 메일을 작성하고 열어보는 단계.


---


### 스토리텔링으로 이해하기

- **편지를 작성한다.** (**L7 응용 계층**)
- 한국인 친구인 줄 알았는데 미국인이었다. 한국어로 쓴 것을 **영어로 다시 번역하자**. (**L6 표현 계층**)
- "나 편지 너한테 보낼 거니까 우리 대화 창(세션) 열어줘, 중간에 끊지 마."라고 **대화 연결을 설정하고 관리**한다. (**L5 세션 계층**)
- **네비게이션(TCP)을 찍고 갈까**, 아니면 보장 없이 빠른 **지름길(UDP)로 갈까** 배달 방식을 결정하고 편지 크기를 조절한다. (**L4 전송 계층**)
- 편지에 **최종 목적지(IP)**를 적고, 그곳에 가기 위한 **다음 거쳐갈 우체국 주소(Next Hop)**를 포함한 **경로를 설정**한다. (**L3 네트워크 계층**)
- 다음 우체국 주소(IP)에서 **ARP 시스템**을 이용하여 **실제 상세 주소(MAC 주소)**를 찾아 포장한다. (**L2 데이터 링크 계층**)
- 마지막으로 포장된 묶음(프레임)을 트럭(물리 매체)에 실어 **전기 신호**로 변환하여 도로(네트워크)로 내보낸다. (**L1 물리 계층**)

# **3. TCP/IP 모델(4계층)과 OSI 계층의 연결**


실제로 인터넷은 OSI 7계층을 그대로 사용하지 않습니다.


실무에서는 **TCP/IP 4계층 모델**이 쓰입니다.


| **TCP/IP 계층**  | **OSI 대응**              |
| -------------- | ----------------------- |
| Network Access | L1 (물리)+ L2 (데이터링크)     |
| Internet       | L3 (네트워크)               |
| Transport      | L4 (전송)                 |
| Application    | L5 + L6 + L7 (세션,표현,응용) |


즉, OSI는 “교과서 모델”, TCP/IP는 “실제 인터넷 모델”입니다.


보안 분석에서는 **L3~L7만 주로 다룹니다.**

# **1. TCP vs UDP 간단 복습**


먼저 큰 그림을 만들기 위해 정리합니다.


## **TCP**

- 신뢰성 보장
- 패킷 순서 보장
- 누락된 데이터 재전송
- 연결형(Connect) 통신
- 예: 웹 브라우징(HTTP/HTTPS), 로그인, 파일 다운로드

## **UDP**

- 신뢰보장 없음(빠름)
- 순서 보장 없음
- 연결 과정 없음
- 예: 게임, 음성채팅(VoIP), 스트리밍

TCP는 “정확하게 전달해야 하는 데이터”에 사용됩니다.


그래서 연결을 **먼저** 만들고 시작합니다.


---


# **2. TCP 3-Way Handshake란?**


TCP 연결을 만들기 위한 **3단계 절차**입니다.


순서만 외우면 안 되고, 각 단계에서 “무엇을 하려는 것인지” 이해하는 것이 중요합니다.


---


# **3. 3단계 흐름 요약**


서버: LISTEN 상태


클라이언트: 연결을 만들고 싶다


## **1단계 — SYN**


클라이언트 → 서버


“연결하고 싶습니다. 제 초기 Sequence 번호는 X입니다.”


예:


```bash
SYN, SEQ=1000
```


## **2단계 — SYN/ACK**


서버 → 클라이언트


“좋습니다. 요청 승인합니다. 그리고 제 초기 Sequence는 Y입니다. 그리고 당신의 Syn도 잘 받았습니다.”


예:


```bash
SYN, ACK, SEQ=3000, ACK=1001
```


## **3단계 — ACK**


클라이언트 → 서버


“서버의 Syn도 잘 받았습니다. 연결 시작합시다.”


예:


```bash
ACK, SEQ=1001, ACK=3001
```


이 3단계가 끝나면 **서버와 클라이언트는 서로를 신뢰하고, 데이터 송수신을 시작할 준비가 완료된 상태**가 됩니다.


---


# **4. 왜 굳이 이런 절차가 필요할까?**


TCP는 “신뢰성”이 목적이기 때문입니다.


서버는 반드시 확실히 해야 합니다:

1. 상대가 진짜 존재하는지
2. 경로에 문제가 없는지
3. 내가 보낸 요청을 상대도 정상적으로 받을 수 있는지
4. 초기 SEQ 번호를 서로 공유해 패킷 순서를 관리할 수 있는지

TCP는 그냥 “연결!” 하고 바로 통신하지 않습니다.


항상 다음을 확인해야 합니다.

- 나 알아?
- 너 알아?
- 우리가 같은 숫자를 기준으로 통신할 준비 됐어?

이 과정을 통해 안정적인 데이터 송수신이 가능해집니다.


---


# **5. 실전 패킷 예시 (Wireshark에서 보는 형태)**


Wireshark를 켜고 브라우저에서 example.com을 접속하면


항상 이런 3개 패킷이 가장 먼저 보입니다:


```plain text
1. SYN     Source Port → Destination Port
2. SYN,ACK Destination Port → Source Port
3. ACK     Source Port → Destination Port
```


3개의 패킷이 거의 동시에 보이기 때문에


많은 신입들이 이걸 “한 번에” 보는데, 실제로는 정확히 3단계입니다.


---


# **6. 보안 공격과의 연결**


TCP 연결 생성 과정은 공격자들이 자주 노립니다.


## **1) SYN Flood Attack**


정상적 SYN 패킷만 보내고


그 다음 ACK를 보내지 않아서


서버를 **반개방 상태(half-open)**로 계속 유지시키는 공격.


서버는 “언제 ACK 오지?” 하면서 계속 기다린 채


자원을 소모하게 됨 → 결국 서버 다운.


## **2) 포트 스캔(Nmap)**


포트가 열려있는지 확인하기 위해


SYN → (SYN/ACK) → RST 형태로


응답을 보고 포트 상태를 판단합니다.


이해되면 다음 챕터에서 상세히 다룹니다.


## **3) 방화벽 정책**


L4 방화벽은


“어떤 포트로 SYN이 들어오는가?”를 감시합니다.


---


# **7. TCP 3-Way Handshake 핵심 요약**

- TCP는 연결형 프로토콜 → 먼저 연결을 만든다
- 연결 생성 절차는 **SYN → SYN/ACK → ACK**
- SEQ/ACK 번호를 교환하여 순서를 맞춘다
- 포트 스캔, DDoS, 세션 관리 등의 기초가 된다
- Wireshark로 보면 가장 먼저 보이는 3개의 패킷

## 1단계 6챕터 : IP, 포트, NAT, 서브넷 — 웹 보안의 필수 네트워크 기본

# **1. IP 주소가 정확히 뭘까?**


IP는 “Internet Protocol Address”의 줄임말로,


**인터넷에서 기계를 식별하기 위한 주소**입니다.


### **비유**


현실 세계에서 집 주소가 있어야 택배가 오듯,


인터넷에서도 **IP 주소가 있어야 패킷을 주고받을 수 있습니다.**


---


## **1) IPv4 주소 구조**


예: 192.168.0.10

- 0~255 범위의 숫자 4개
- 32bit(4 byte)로 구성
- 이 주소는 네트워크에서 “누구인지”를 구별하는 역할

---


# **2. 공인 IP / 사설 IP**


인터넷에서는 두 종류의 IP가 존재합니다.


## **1) 공인 IP (Public IP)**

- 전 세계에서 유일한 IP
- 인터넷 어디서나 접근 가능
- 웹 서버, VPN 서버 등이 쓰는 주소

예:


13.124.xx.xx (AWS EC2), 54.xxx.xxx.xxx


---


## **2) 사설 IP (Private IP)**

- 내부 네트워크용
- 집/회사/Wi-Fi 내부에서만 사용
- 외부에서는 직접 접근 불가
- 보통 공유기가 자동 부여

예:


192.168.x.x


10.x.x.x


172.16.x.x ~ 172.31.x.x


---


## **중요한 결론**


**일반 PC, 핸드폰, 노트북은 모두 사설 IP를 가지고 있다.**


외부(인터넷)와 통신할 때는 NAT를 통해 공인 IP로 바뀐다.


이걸 이해하는 게 바로 다음 주제입니다.


---


# **3. 포트(Port)**


포트는 **하나의 IP가 여러 서비스를 구분하기 위한 번호**입니다.


### **비유**


한 집 주소(=IP)에 문이 여러 개(=포트) 있는 것과 같다.


예:

- 80: HTTP
- 443: HTTPS
- 22: SSH
- 3306: MySQL
- 8080, 3000, 5000, 8000: 개발 서버에서 자주 쓰는 포트

### **핵심**


IP = 집 주소


Port = 방/문 번호


그래서 한 서버가 여러 프로그램을 실행할 수 있는 이유가 바로 포트입니다.


---


# **4. NAT(Network Address Translation) — 꼭 이해해야 하는 핵심**


당신의 컴퓨터는 사설 IP를 쓰고,


인터넷 전체는 공인 IP로 통신 없이 동작합니다.


그럼 이런 의문이 생깁니다:


**“내 컴퓨터는 사설 IP인데, 어떻게 인터넷(공인 IP)과 통신하지?”**


그 답이 바로 **NAT**입니다.


---


## **NAT의 역할**


라우터(공유기)가:

1. 내부 PC의 사설 IP와 포트 → 공인 IP와 특정 포트로 **변환**
2. 외부에서 온 응답을 다시 내부 PC로 **되돌려줌**

## **NAT 동작 예시**


### **내부 PC**


192.168.0.10:45000


(브라우저 포트는 랜덤 포트가 사용됨)


### **NAT → 변환**


112.221.xxx.xxx:53120


(공인 IP + 변환된 포트)


### **외부 서버**


이 IP/포트로 응답을 보내면


NAT가 다시 내부 PC에게 전달.


---


## **왜 NAT가 필요할까?**


이유는 2가지입니다.


### **1) IPv4 주소가 부족해서**


전 세계의 모든 기기에 공인 IP를 줄 수 없습니다.


### **2) 보안 측면**


사설 IP는 외부에서 직접 접근할 수 없기 때문에


기본적인 보호막 역할도 함.


---


# **5. 서브넷(Subnet)**


서브넷은 네트워크를 “그룹”으로 나누는 개념입니다.


예:


192.168.0.10/24


여기서 /24는


**앞 24bit는 네트워크 주소, 뒤 8bit는 호스트 주소**라는 의미.


쉽게 말하면:

- /24 → 한 네트워크 안에 256개의 주소
- /16 → 더 큰 네트워크
- /8 → 훨씬 큰 네트워크

보안에서 서브넷은:

- 내부망 구성
- 라우팅
- 스캔 범위 설정
- ACL 설정
- 방화벽 정책 적용

등에서 매우 중요합니다.


하지만 지금은 “숫자가 네트워크 범위를 의미한다” 정도만 이해하면 됩니다.


---


# **6. 전체 개념 정리 (한 번에 요약)**

- **IP 주소:** 인터넷에서 기계를 식별하는 주소
- **공인 IP:** 전 세계에서 유일, 외부 접근 가능
- **사설 IP:** 내부망 전용, 외부에서 접근 불가
- **포트:** 하나의 IP에서 프로그램/서비스를 구분
- **NAT:** 사설 IP ↔ 공인 IP 변환
- **서브넷:** 네트워크 범위를 나누는 방식 (예: /24)

이 6개는 앞으로 보안 분석의 전체 기반입니다.

# **1. DNS란 무엇인가?**


DNS는


**도메인 이름을 IP 주소로 변환해주는 서비스**입니다.


예:


```plain text
https://example.com → 93.184.216.34
```


### **비유**


전화번호부 역할을 한다.


우리는 “엄마”라고 연락처를 저장해두지만


실제로는 “010-xxxx-xxxx”로 전화가 가는 것처럼


브라우저도 “naver.com”이라고 쓰면


내부에서는 “xxx.xxx.xxx.xxx”라는 IP로 변환해서 접속합니다.


---


# **2. DNS 요청 과정 (한 번에 이해되는 버전)**


브라우저 주소창에 example.com 입력 시 다음 과정이 일어납니다:


### **1) 브라우저 캐시 확인**


“이 사이트의 IP 내가 전에 알고 있었나?”


### **2) OS 캐시 확인**


운영체제(OS)가 최근 30초~몇 분 동안 유지하는 DNS 캐시.


### **3) 로컬 DNS 서버(보통 ISP DNS)에게 질의**


KT, SK, LG 인터넷이 제공하는 DNS 서버


### **4) 만약 로컬 DNS도 모르면 → 루트 DNS로 이동**


루트 DNS는 전 세계 13개 존재.


도메인 최상위 정보만 알고 있음.


### **5) TLD DNS로 이동 (.com, .net 담당)**


예: .com 도메인들의 DNS가 모여있음.


### **6) 최종 authoritative DNS(도메인 주인 DNS)**


여기서 “example.com의 IP는 이것입니다”라고 알려줌.


### **7) OS ↔ 브라우저가 캐싱**


다음 요청부터 훨씬 빠르게 처리됨.


---


# **3. DNS를 공격하는 방식**


웹 취약점 분석가·모의해커 관점에서는


DNS는 공격자에게 매우 매력적인 타깃입니다.


### **DNS Cache Poisoning**

- 공격자가 가짜 IP를 DNS 캐시에 주입

    → 사용자가 정상 서버 대신 악성 서버로 접속됨


### **DNS Spoofing**

- 네트워크 내부에서 DNS 응답을 가로채
- 공격자가 임의의 IP를 돌려주는 방식

### **악용 예시**


피싱 페이지로 유도, 악성 파일 다운, 세션 탈취 등


웹 공격은 대부분 7계층(L7) 공격이지만


DNS 공격은 예외적으로 **계층 사이에서 발생하는 트릭 공격**입니다.


---


# **4. 라우팅(Routing)이란?**


DNS가 주소를 찾는 과정이었다면,


라우팅은


**그 주소(IP)까지 실제로 패킷을 운반하는 과정입니다.**


### **비유**


택배 기사님이 실제로 도로를 따라 이동하는 과정.


라우터는 패킷을 보면 항상 다음을 결정합니다:

- 이 패킷은 어느 방향으로 보내야 하는가?
- 어느 라우터를 다음 경유지로 정해야 하는가?

---


# **5. 패킷이 실제로 이동하는 구조**


예: 당신 컴퓨터 → 네이버 접속

1. 브라우저에서 요청 생성
2. DNS 조회 → 네이버 IP 획득
3. TCP 3-Way Handshake
4. 패킷이 NAT를 통과하여 공인 IP로 변환
5. ISP 라우터로 이동
6. 여러 지역 라우터를 거침
7. 네이버 엣지 서버(가장 가까운 지역 서버)에 도착
8. 서버가 응답을 되돌려 보냄
9. NAT가 다시 사설 IP로 변환
10. 브라우저가 HTML 렌더링

---


# **6. Traceroute(Windows: tracert)의 의미**


Traceroute는


“패킷이 통과하는 모든 라우터 경로를 보여주는 명령어”입니다.


예:


```plain text
tracert google.com
```


결과:

- 몇 번째 라우터를 지나갔는지
- 어느 지역에 있는지
- 어디서 지연이 발생하는지

보안 실무에서는


경로 분석, DDOS 침입 지역 추적 등에 활용됩니다.


---


# **7. 이 챕터 핵심 요약**

- DNS는 도메인 → IP 변환 역할
- DNS는 계층 사이에 존재하는 독특한 공격 포인트
- 라우팅은 IP 기반으로 패킷을 목적지까지 이동
- NAT는 내부 사설 IP를 공인 IP로 변환
- Traceroute는 라우터 경로를 보여주는 명령어
- 웹 요청은 “DNS → NAT → 라우팅 → 서버 응답” 구조로 흐름

## 1단계 8챕터 — HTTP / HTTPS / 쿠키·세션 / 웹 요청 구조

좋아.


그럼 지금부터 1단계 1번 파트의 마지막 챕터,


웹 취약점 분석가가 반드시 이해해야 하는 핵심인


# **1단계 8챕터 — HTTP / HTTPS / 쿠키·세션 / 웹 요청 구조**


를 시작하겠다.


이 챕터는 뒤에서 배울:

- XSS
- CSRF
- 세션 하이재킹
- 인증 우회
- CORS
- SameSite
- SSRF
- Host Header Attack

등 모든 웹 취약점의 뿌리가 되는 이론이다.


이번 파트는 당신의 웹 보안 실력이 폭발적으로 성장하는 지점이다.


# **1. HTTP란 무엇인가?**


HTTP(HyperText Transfer Protocol)는

- *클라이언트(브라우저)와 웹 서버가 데이터를 주고받는 규칙(프로토콜)**이다.

즉,

- 당신이 클릭할 때
- 서버가 페이지를 보내줄 때
- 로그인할 때
- API로 JSON을 받을 때

모두 HTTP가 사용된다.


# **2. HTTP는 “상태가 없는(stateless)” 프로토콜이다**


이 말은:

- 한 요청과 다음 요청 사이에 기억이 없다
- 서버는 이 사람이 누군지 기억하지 않는다

예를 들어:

1. 첫 요청: “안녕하세요, 로그인 페이지 주세요”
2. 두 번째 요청: “나 로그인을 했습니다”
3. 세 번째 요청: “마이페이지 주세요”

서버 입장에서 보면:

- 1번 요청 보낸 사람
- 2번 요청 보낸 사람
- 3번 요청 보낸 사람

전부 서로 다른 사람으로 보인다.


그래서 “세션(Session)”이라는 개념이 등장한다.


이걸 이해해야 이후 세션 취약점, 쿠키 탈취, CSRF, **세션 고정(Session Fixation)**을 이해할 수 있다.


# **3. HTTP 요청(Request) 구조**


브라우저가 서버에 보내는 요청은 아래처럼 생긴다.


GET /login HTTP/1.1


Host: example.com


User-Agent: Chrome/120.0


Accept: text/html


Cookie: sessionId=abcd1234


요청은 3개로 구성된다:


# **1) Request Line**


GET /login HTTP/1.1

- 메서드(GET, POST, PUT, DELETE…)
- 경로(/login, /api/user)
- 프로토콜 버전

# **2) Request Header**


Host: example.com


User-Agent: Chrome


Accept: text/html


Cookie: sessionId=abcd1234


웹 취약점 분석가가 제일 많이 보는 부분.


여기서 발생하는 취약점이 많다.


예:

- Host 헤더 변조 → 서버 오인
- User-Agent 기반 필터 우회
- Accept-Language 이용한 정보수집
- Cookie 탈취 및 변조
- Origin/Referer 기반 CSRF 우회
- Authorization 헤더 변조 → 인증 우회

# **3) Request Body**


POST, PUT 같은 요청에서 사용됨.


예:


username=admin&password=1234


또는 JSON:


{


"email": "test@example.com",


"pw": "1234"


}


# **4. HTTP 응답(Response) 구조**


서버가 브라우저에게 보내는 응답:


HTTP/1.1 200 OK


Content-Type: text/html


Set-Cookie: sessionId=abcd1234; HttpOnly; Secure


<html> ... </html>


응답도 3파트다.

- Status Line (200 OK, 404 Not Found, 500 Internal Server Error)
- Header (Set-Cookie, Content-Type 등)
- Body (HTML, JSON, 이미지)

# **5. 쿠키(Cookie)**


HTTP는 stateless라서 “기억 기능”이 없기 때문에


서버는 쿠키를 이용해 사용자를 식별한다.


쿠키는 브라우저에 저장되는 작은 데이터 조각이다.


예:


sessionId=abcd1234


브라우저는 자동으로 모든 요청마다 쿠키를 함께 보낸다.


이 시스템 때문에:

- 로그인 상태 유지
- 장바구니 유지
- 자동 로그인
- 사용자 트래킹
이 가능하다.

그리고 동시에…

- 세션 탈취
- CSRF
- XSS를 통한 쿠키 탈취
- 세션 고정 공격
의 취약점이 발생한다.

# **6. 세션(Session)**


서버가 쿠키를 이용해 “사용자별 저장 공간”을 만들어둔 것이 세션이다.


브라우저는:


Cookie: sessionId=abcd1234


서버는:


세션 저장소에서 "abcd1234"를 찾아서


→ 거기에 로그인 정보, 장바구니 정보 등 저장


정화의 FE 백엔드 지식으로 비유하면:

- FE에서 쿠키는 “키(key)”만 들고다님
- BE는 쿠키의 값(sessionId)을 이용해 “유저 상태”를 찾아서 복원함

이 개념이 정확히 자리잡아야


이후에 배울 세션 하이재킹, 세션 탈취, 세션 고정 공격이 완벽히 이해된다.


# **7. HTTPS란?**


HTTPS는 HTTP + TLS 암호화.


즉,


내용을 암호화해서 중간에서 보더라도 해석할 수 없게 만든 것.


예:

- 로그인 정보
- 쿠키(sessionId)
- 민감한 요청
이 모두 안전하게 암호화됨.

# **8. 이 챕터 핵심 요약**

- HTTP는 stateless → 세션/쿠키 필요
- Request Line + Header + Body 구조
- Response도 Status Line + Header + Body
- 쿠키 = 브라우저가 저장하는 사용자 식별 데이터
- 세션 = 서버의 사용자 상태 저장소
- HTTPS = HTTP + 암호화(TLS)

여기까지 이해하면


웹 취약점 분석의 모든 기초가 완성된다.

## 1단계 9챕터: HTTPS & TLS 심화 — 웹 취약점 분석의 기반 암호화 구조

좋아.


이제 HTTP의 상위 단계이자 웹 보안의 핵심 중 핵심,


HTTPS / TLS 암호화 구조를 본격적으로 들어간다.


이 단계를 정확히 이해하면 이후에 나오는:

- 세션 하이재킹
- 쿠키 탈취
- SSLStrip
- MITM 공격
- CORS, SameSite 보안
- 인증/인가 취약점
- Host Header Attack

이 모든 것이 완전히 명확해진다.


# **1단계 9챕터: HTTPS & TLS 심화 — 웹 취약점 분석의 기반 암호화 구조**


# **1. HTTPS란 무엇인가?**


HTTPS는:


HTTP + TLS(암호화 계층)


즉,


“웹 요청·응답을 전부 암호화해서 중간에서 보더라도 해석할 수 없도록 만든 것.”


# **2. HTTPS가 등장한 이유**


HTTP는 평문(plaintext) 통신이다.


즉,

- 로그인 정보
- 쿠키(sessionId)
- Request Body
- Authorization 헤더

모두 그대로 노출된다.


중간에 패킷을 캡처하면 전부 다 보인다.


그래서 등장한 것이 TLS이다.


# **3. TLS(Transport Layer Security)란?**


TLS는


브라우저 ↔ 서버 간 통신을 암호화하는 프로토콜이다.


TLS는 두 가지 역할을 한다:

1. 암호화(Encryption)
→ 중간자가 데이터를 봐도 해석 못함
2. 무결성(Integrity)
→ 데이터가 중간에서 바뀌면 바로 감지됨
3. 서버 인증(Authentication)
→ “내가 진짜 네이버인지 증명”

HTTPS에서의 “패드락 자물쇠 아이콘”이 바로 이것이다.


# **4. TLS Handshake 과정 (쉬운 버전)**


브라우저가 https://example.com 접속하면 다음 4단계를 거친다.


# **1) Client Hello**


브라우저 → 서버

- “나 TLS 사용하고 싶어!”
- “내가 지원하는 암호화 방식은 이거이거야.”

# **2) Server Hello**


서버 → 브라우저

- “좋아. 우리 둘이 사용할 암호화 방식은 이걸로 정하자.”

그리고 **서버 인증서(SSL 인증서)**를 브라우저에 준다.


# **3) 브라우저가 인증서 검증**

- 이 인증서가 CA(인증기관)에 의해 발급된 게 맞는지
- 도메인이 일치하는지
- 인증서가 만료되지 않았는지

여기서 실패하면 브라우저는 ‘주의 요함’, ‘안전하지 않음’으로 표시한다.


# **4) 세션키 공유 (암호화 채널 생성)**


브라우저와 서버는 대칭키(빠른 암호화에 사용)를 공유하고


이후 모든 통신은 완전히 암호화된다.


# **5. 왜 대칭키와 비대칭키를 둘 다 사용할까?**


초보자들이 가장 자주 헷갈리는 부분이다.


# **비대칭키(RSA/ECDSA)의 특징**

- 매우 안전함
- 하지만 엄청 느림
- 인증서 검증에 사용

# **대칭키(AES)의 특징**

- 매우 빠름
- 대량 데이터 암호화에 적합

그래서 이렇게 사용한다:

1. 핸드쉐이크에서는 비대칭키(서버 인증 + 세션키 생성)
2. 이후 데이터 암호화는 대칭키(AES 등)

즉, TLS는


비대칭키로 문을 열고, 대칭키로 대량의 데이터를 안전하게 주고받는 구조다.


# **6. HTTPS가 깨지는 순간들**


HTTPS는 안전하지만, 다음 상황에서는 취약해질 수 있다:


# **1) MITM(중간자 공격)**

- 공격자가 인증서를 위조해 끼워넣는 경우
(→ 실제론 매우 어렵고, 브라우저 경고 뜸)

# **2) SSLStrip 공격**

- HTTP로 강제 다운그레이드 시키는 공격
- 오래된 사이트나 리디렉션 문제에서 발생

# **3) HSTS 미설정**

- 브라우저가 HTTP로 접속할 수 있는 상태

# **4) 인증서 만료 또는 불일치**

- 브라우저가 “안전하지 않음” 표시
- 공격 표면 증가

# **7. 쿠키 보안 옵션 3가지 — 이것이 웹 보안의 핵심**


# **1) HttpOnly**


→ JS로 접근 불가 → XSS로 쿠키 탈취 방지


# **2) Secure**


→ HTTPS인 경우만 전송 → http downgrade 공격 방지


# **3) SameSite**


→ CSRF 방지


(Strict / Lax / None)


이 3개는 웹 취약점 분석가라면 반드시 숙지해야 하는 프런트라인 방어 기능이다.


# **8. 인증서 구조 (CA → Intermediate → Server Cert)**


인증서는 계층 구조로 이루어져 있다:


Root CA


→ Intermediate CA


→ Server Certificate (example.com)


브라우저는 전 세계 Root CA를 내장하고 있고,


Root CA → Intermediate → Server Cert


이 체인이 유효해야 “신뢰할 수 있음(🔒)”이 표시된다.


# **9. HTTPS 핵심 요약**

- HTTPS = HTTP + TLS 암호화
- TLS는 비대칭키로 인증 + 대칭키로 암호화
- 인증서 검증으로 MITM 차단
- 쿠키 보안 옵션(HttpOnly/Secure/SameSite) 필수
- HTTP는 평문 → HTTPS는 암호화
- HSTS는 HTTPS 강제 고정용 정책

여기까지 이해하면 이후 웹 공격 대부분을 훨씬 직관적으로 해석할 수 있다.

웹 해킹할 때 대부분 **Kali Linux**나 기타 리눅스 배포판 사용하니까,


리눅스 명령어는 거의 “모국어 수준”으로 익숙해지는 게 좋습니다.

- 기본 명령어
    - ls, cd, pwd, mkdir, rm, cp, mv
    - cat, less, head, tail, grep, find
    - ps, top, htop, kill
    - chmod, chown, ln
- 패키지 관리
    - apt update/upgrade/install
- 파일 권한 / 소유권
    - rwx, 755 / 644 의미
- 쉘 기초
    - bash에서 파이프(|), 리다이렉션(>, >>, <)
    - 간단한 alias, history 활용

**실습 아이디어**

- Kali (또는 Ubuntu) 가상머신 설치해서:
    - 홈 디렉토리·기타 디렉토리 탐색
    - 특정 문자열이 포함된 파일을 grep으로 찾아보기
    - 프로세스 목록에서 특정 프로그램 찾아 kill 해보기

## **2단계. 네트워크 (이론 + 패킷 분석)**


여기가 “1. 네트워크” 파트입니다.


웹 취약점 분석가는 **패킷 흐름**을 이해하고 있어야, 이상한 동작을 눈으로 보고 이해할 수 있습니다.


### **2-1. OSI 7계층 & TCP/IP 모델**

- OSI 7계층: L1~L7 역할 정리
- TCP/IP 4계층 (링크 / 인터넷 / 전송 / 응용)
- 캡슐화/역캡슐화 개념

**핵심 포인트**


“HTTP 요청 하나 보낼 때, 실제로 L7→L4→L3→L2로 내려가서 패킷이 어떻게 나가는지 머릿속에서 시뮬레이션 가능한 수준”


---


### **2-2. IP/TCP/UDP/포트**

- IP
    - IP 주소 / 서브넷 / 게이트웨이 개념
    - 라우팅이란?
- TCP
    - 3-way handshake (SYN, SYN/ACK, ACK)
    - 연결 종료 (FIN, ACK)
    - 시퀀스 번호 / ACK 번호
- UDP
    - 연결 없는 프로토콜 특징
- 포트
    - well-known port, ephemeral port
    - 하나의 IP에서 여러 서비스가 동작하는 방식

**실습**

- Wireshark로 TCP 3-way handshake 직접 캡처해서 패킷 구조 보기
    - 필터: tcp.flags.syn == 1 등 사용
- ping, traceroute, netstat/ss 사용해보기

---


### **2-3. HTTP/HTTPS 심화**


웹 해킹에서는 **HTTP/HTTPS**가 가장 중요합니다.

- HTTP
    - Request/Response 구조, 메서드 (GET/POST/PUT/DELETE 등)
    - 상태 코드 (2xx/3xx/4xx/5xx)
    - 헤더 (Host, Cookie, Referer, User-Agent, Authorization 등)
    - 바디 / 쿼리스트링 / 폼데이터
- HTTPS
    - TLS Handshake 개념
    - 대칭키/비대칭키 아주 기초
    - 인증서 컨셉

**실습**

- 브라우저 개발자도구 Network 탭으로 HTTP 요청 뜯어보기
- curl로 직접 요청 보내보기
    - curl -v, curl -X POST -d 등
- Burp Suite, OWASP ZAP으로 프록시 잡아서 패킷 가로채보기

---


### **2-4. 네트워크 도구 사용 연습**

- ping, traceroute, nslookup, dig
- nmap 스캔
    - 포트 스캔: nmap -sS, -sV, -O
    - 서비스 버전, OS 탐지

**목표**

- “이 서버에서 어떤 포트가 열려 있는지, 어떤 서비스가 도는지” 기본적인 정보는 nmap과 Netstat로 직접 볼 수 있는 수준

---


## **3단계. 시스템(리눅스/윈도우) & 서버 이해**


여기가 “2. 시스템”에 해당하는 부분입니다.


웹 취약점을 찾아도, 결국 **시스템에 어떤 영향이 있는지** 이해해야 제대로 된 리포트를 쓸 수 있습니다.


### **3-1. 리눅스 시스템 심화**

- 사용자/그룹/권한 구조
    - /etc/passwd, /etc/shadow 개념
- 프로세스 / 서비스
    - systemd / service 관리 (systemctl)
- 로그
    - /var/log 구조
    - 웹서버 로그 (access.log, error.log) 형식 이해

**실습**

- 웹 서버 설치하고 로그 확인
    - Nginx 또는 Apache 설치
    - 간단한 정적 페이지 서빙
    - 브라우저에서 접속 후 access.log / error.log 읽어보기

---


### **3-2. 윈도우 기본 이해 (선택적이지만 추천)**


웹 취약점 분석가는 리눅스만 알아도 시작은 가능하지만,


현업에선 윈도우 서버/클라이언트도 여전히 많습니다.

- 윈도우 프로세스 / 서비스 / 이벤트 로그 개념
- 레지스트리 구조 (대략)
- 작업 관리자 / 이벤트 뷰어 보는 법

---


### **3-3. 가상화 환경 / 보안 실습 환경 만들기**

- 가상머신 도구
    - VMware / VirtualBox / Parallels 등
- Kali Linux 설치
- 취약한 웹 애플리케이션 설치
    - DVWA (Damn Vulnerable Web Application)
    - OWASP Juice Shop
    - WebGoat

**목표**

- “내 컴퓨터에서 마음껏 공격해도 되는 실습 환경”을 직접 만들 수 있을 것
- 앞으로의 모든 실습은 여기 위에서 진행

---


## **4단계. 웹 구조 & 개발 관점 이해 (이미 베이스 있음 → 보안 관점 추가)**


사용자는 이미 프론트엔드 개발자이기 때문에, 이 단계는 “보안 관점으로 재정리” 느낌으로 가면 됩니다.


### **4-1. 웹 애플리케이션 구조**

- 클라이언트 / 서버 / DB 구조
- SPA vs SSR vs CSR
- REST API / GraphQL 개념
- 세션 / 쿠키 / 토큰 (JWT) 정리
    - 세션 ID가 서버/DB에 어떻게 저장되고, 요청마다 어떻게 매칭되는지
    - JWT 장단점, 만료/재발급 로직

---


### **4-2. 인증 / 인가 / 세션 관리**


이 파트는 보안에서 매우 중요합니다.

- 로그인 로직 흐름
    - ID/PW → 서버에서 검증 → 세션 or 토큰 발급
- 세션 고정(Session Fixation), 세션 하이재킹 개념
- 비밀번호 저장
    - 해시 (bcrypt, scrypt, PBKDF2)
    - 솔트 개념

**실습**

- 간단한 로그인 기능이 있는 예제 (직접 만든 것 or DVWA)에서:
    - 세션 쿠키 값을 일부러 복사해서 다른 브라우저에 붙여넣기 → 세션 하이재킹 체험
    - 만료/로그아웃 동작 확인

---


## **5단계. 보안 기초 이론 (전반적인 보안 마인드)**


이제부터 “보안” 여러 분야에 공통으로 쓰이는 기초 개념들입니다.


### **5-1. 보안 3요소 / 위협 모델**

- CIA Triad
    - Confidentiality / Integrity / Availability
- 인증/인가/감사 (AAA)
- 위협 모델링 간단 개념
    - STRIDE(위변조/위장/정보유출 등) 정도만 이름과 개념

---


### **5-2. 취약점 분류와 기준**

- CVE / CVSS 개념
- 취약점 종류 (RCE, LFI/RFI, SQL injection, XSS, CSRF, SSRF 등)
- 0-day / 1-day / N-day 개념

---


### **5-3. 암호학 기초**


고급 수학까지는 필요 없고, “웹 보안에서 필요한 만큼만” 이해하면 됩니다.

- 대칭키 / 비대칭키
- 해시 vs 암호화의 차이
- SSL/TLS에서의 활용 (공개키로 암호화, 개인키로 복호화 등 개략)

---


## **6단계. 웹 취약점 이론 (OWASP Top 10 중심)**


여기가 이제 “3. 웹”의 핵심입니다.


목표는 최소한 **OWASP Top 10 전 항목을 예제와 함께 이해**하는 것.


대표적인 취약점들을 순서대로:

1. 인젝션 (Injection)
    - SQL Injection
    - Command Injection
    - LDAP, NoSQL 인젝션 간략히
2. 인증/세션 취약점
    - Broken Authentication & Session Management
3. 접근통제 취약점
    - IDOR, Broken Access Control
4. XSS (Cross-Site Scripting)
    - Reflected / Stored / DOM-based
5. CSRF (Cross-Site Request Forgery)
6. 파일 업로드 취약점
    - 업로드 우회, 확장자 우회, 웹셸 업로드
7. SSRF (Server-Side Request Forgery)
8. XXE (XML External Entity)
9. 보안 설정 미비 (Security Misconfiguration)
10. 민감정보 노출 (Sensitive Data Exposure)

### **각 취약점 공부 루틴 (패턴)**


각 항목별로 다음 패턴으로 공부하면 좋습니다:

1. 개념
    - 취약점이 왜 발생하는지 (개발자가 뭘 잘못해서?)
    - 어떤 상황에서 자주 나오는지
2. 공격 시나리오
    - 공격자가 구체적으로 어떤 입력을 보내는지
    - 성공하면 어떤 일이 벌어지는지 (정보 노출, 권한 상승 등)
3. 방어 방법
    - 입력 검증 / 인코딩 / 필터링
    - 인증/인가 로직 설계 원칙
    - 보안 설정/헤더 설정 등
4. 실습
    - DVWA / Juice Shop / WebGoat 등에서 해당 취약점 레벨 깨보기
    - Burp Suite 사용해서 패킷 수정, 반복 공격 연습

---


## **7단계. 웹 해킹 도구 & 자동화**


웹 취약점 분석가는 “툴 숙련도 + 직접 스크립트로 보조” 두 가지가 중요합니다.


### **7-1. 필수 도구**

1. Burp Suite
    - 프록시 설정 (브라우저와 Burp 연결)
    - Repeater, Intruder, Decoder, Comparer, Sequencer
    - 간단한 스캐닝 기능 사용법
2. OWASP ZAP
    - 자동 스캔, 스파이더링
3. 브라우저 개발자 도구
    - Network / Application / Console 탭 활용

**실습**

- 브라우저 → Burp → 타겟 사이트 구조 살펴보기
- 쿠키/헤더 수정하면서 로그인 우회 실험
- 요청 재전송/Replaying, Brute force 시뮬레이션

---


### **7-2. 스크립트/자동화 (Python 추천)**


웹 취약점 분석가는 “반복 업무를 스크립트로 줄이는 능력”이 큰 강점입니다.

- Python 기초 (변수, 조건문, 반복문, 함수, 모듈)
- requests, httpx로 HTTP 요청 자동화
- BeautifulSoup / lxml로 HTML 파싱
- 간단한 포트 스캐너, 디렉토리 브루트포싱 도구 만들어보기

**예시 미니 프로젝트**

- 로그인 폼에 ID/PW 리스트 넣어서 로그인 성공 여부 체크하는 스크립트
- 특정 파라미터에 SQL 인젝션 페이로드를 자동으로 넣어보는 간단한 PoC 스크립트

---


## **8단계. 실전 모의해킹 흐름 배우기**


지금까지는 “기술” 하나하나였다면,


여기서는 “실제 업무에서 어떻게 진행하는지” 흐름을 익히는 단계입니다.


### **8-1. 모의해킹 전체 프로세스**

1. 범위 정의 (Scope 정의)
    - 테스트 대상 도메인, IP, 기능
    - 허용/금지 공격 유형
2. 정보 수집 (Reconnaissance)
    - 도메인, 서브도메인, IP 범위 수집
    - 서비스/포트 스캔
    - 기술 스택 파악
3. 취약점 분석
    - 자동 스캐너 (Burp/ZAP, 기타 상용 도구)
    - 수동 점검 (OWASP Top 10 체크리스트 기반)
4. 공격 및 권한 상승 (Exploit / Privilege Escalation)
    - PoC 작성
    - 내부망 접근, 다른 계정 탈취 등
5. 리포트 작성
    - 취약점 상세 설명, 재현 방법, 영향도, 보안 권고사항

---


### **8-2. 버그바운티/CTF 플랫폼 활용**


실전 감각을 기르려면 “실제 또는 유사 실전 환경”이 필요합니다.

- 웹 해킹 중심 CTF 플랫폼
    - Hack The Box (Web 카테고리)
    - TryHackMe (Web Fundamentals, OWASP Top 10 등 트랙)
- Bug Bounty Program
    - HackerOne, Bugcrowd, YesWeHack 등 (영문)

**목표**

- 최소 몇 개의 웹 취약점 문제를 직접 풀어서,

    “이런 식으로 취약점을 발견하고, PoC를 만들고, 리포트를 쓰는구나” 감 잡기


---


## **9단계. 문서화·리포트·포트폴리오**


웹 취약점 분석가는 **“잘 적는 사람”이 강합니다.**


기술보다 리포트 퀄리티로 평가받는 경우가 많습니다.


### **9-1. 리포트 작성 연습**


각 실습에서:

- 대상 설명
- 발견된 취약점 이름 (예: Stored XSS)
- 발생 위치 (URL, 파라미터명)
- 재현 절차 (Step by step)
- 영향도 (사용자 데이터 탈취, 계정 탈취 등)
- 권고사항 (개발팀이 뭐를 수정해야 하는지)

형태로 정리해서 **Notion or Markdown**으로 남겨두면 좋습니다.


---


### **9-2. 포트폴리오/이력 세팅**


나중에 실제 취업/프리랜서로 갈 때 사용할 수 있도록:

- “웹 취약점 분석 실습 기록” 페이지
    - 사용한 환경 (Kali, DVWA 등)
    - 각 취약점별 공격 PoC와 방어안 요약
- “개발자 + 보안 관점” 둘 다 보여주는 프로젝트
    - 예: 본인이 만든 작은 웹 서비스에 대해 스스로 모의해킹 후 리포트 작성

---


## **10단계. 추천 공부 순서 요약 (체크리스트 버전)**


위 내용을 “순서대로 체크하면서” 진행할 수 있게 정리해보면:

1. 리눅스 기본 명령어·파일권한·서비스·로그 익숙해지기
2. OSI 7계층 / TCP/IP 모델 / TCP 3-way handshake 완전히 이해
3. HTTP/HTTPS 구조 & 주요 헤더·메서드·상태코드 숙지
4. Kali Linux + 취약한 웹앱(DVWA/Juice Shop/WebGoat) 설치
5. 웹 애플리케이션 구조, 세션/쿠키/JWT, 로그인/권한 로직 복습 (개발자 관점 → 보안 관점)
6. 보안 기초 개념 (CIA, 암호화/해시, 취약점/위협모델) 정리
7. OWASP Top 10 각 항목에 대해:
    - 개념 정리
    - 실습 (DVWA/PortSwigger Lab 등)
    - 방어 방법까지 정리
8. Burp Suite / ZAP 툴 숙련:
    - Proxy, Repeater, Intruder 활용
    - 자동 스캔 + 수동 분석
9. Python으로 간단한 HTTP 공격/탐지 스크립트 작성
10. HackTheBox / TryHackMe / CTF 문제로 실전 감각 익히기
11. 모든 실습을 리포트 형식으로 문서화 → 포트폴리오화
