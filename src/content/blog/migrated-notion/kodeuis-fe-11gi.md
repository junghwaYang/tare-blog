> 👉🏻  **CSS의 Cascading에 대해 설명해 주세요.**


## <u>**Cascading 이란?**</u>

>   
> 동일한 요소에 여러 스타일을 적용 시킬 때,  
> 어떤 스타일이 **우선적으로 적용**이 되는지 결정하는 방식을 말한다.  
> 

### 적용 순서(우선 순위가 높은 순)


> 1️⃣ 인라인 스타일


> 2️⃣ id 선택자


> 3️⃣ class, 속성, 가상 선택자


> 4️⃣ 태그 선택자


###  소스 순서에 따른 우선순위


```css
.box{
	background-color: blue;
}

.box{
	background-color: red;
}
```


위 css를 작성 했을때, css 코드의 동작은 ‘위에서부터 아래로’ 동작 하기 때문에


결과적으로 `background-color: red;` 가 적용이 된다.


### CSS의 상속


일부 CSS 스타일은 부모 요소로 부터 자식 요소에게 상속되는 속성들이 있다.


`color`, `font-family`, `font-size` 와 같은 텍스트 요소에 관한 속성들은


부모로 부터 자식 요소가 상속 받아 적용되지만,


`margin`, `padding` 과 같은 박스 속성은 상속이 되지 않는다.


여기서 부모로 부터 상속 받은 자식요소에 정의된 스타일이 없을 경우,


부모 요소로 부터 상속받은 스타일이 적용되지만,


자식 요소에 정의된 스타일이 작성 된 경우에는 자식 요소에 정의된 스타일로 적용 된다.


### !important


`!important` 는 위 CSS 우선 순위 규칙을 **모두 무시한 채 가장 최우선으로 적용** 되는 속성이다.


```css
.box{
	background-color: blue !important;
}

.box{
	background-color: red;
}
```


 `!important` 는 위 처럼 사용되는데,


해당 스타일링의 최종 스타일은 `background-color: blue !important;` 가 적용 된다.


⚠️ `!important` 는 모든 규칙을 무시하고 최우선으로 적용되기 때문에,


협업 시 스타일이 동작하지 않는다고 무작정 사용하면 추후 큰 화를 입을 수 있다.


정말 필요할 때를 잘 구분해서 사용하는게 좋다.


### 브라우저 스타일


기본적으로 html 요소에는 브라우저 스타일이 적용 되어있다.


이는 사용자 정의 스타일이 없을 때 적용되며,


사용자 정의 스타일이 있을때는 브라우저 스타일 위에 사용자 정의 스타일로 덮어 씌우게 된다.


 


⭐️ 프로젝트 스타일링 전 reset.css 나 normalize.css 등을 통해, 브라우저 스타일을 지우고 시작한다.


🔍reset.css :

- [https://meyerweb.com/eric/tools/css/reset/](https://meyerweb.com/eric/tools/css/reset/)
- [https://elad2412.github.io/the-new-css-reset/](https://elad2412.github.io/the-new-css-reset/)

🔍normalize.css : 

- [https://necolas.github.io/normalize.css/](https://necolas.github.io/normalize.css/)

？ **reset.css**와 **normalize.css**의 차이


1️⃣ reset.css

- reset.css의 경우 브라우저 스타일을 초기화 해주는 스타일 시트 이다.
- 사용자 정의 스타일로 처음부터 웹 페이지를 스타일링 할 수 있다.

2️⃣ normalize.css

- normalize.css는 reset.css 처럼 초기화 해주진 않지만, 가능한 브라우저 스타일을 최대한 건들지 않는 선에서 브라우저 간에 상이한 부분만 스타일을 통일시켜주는 시트라고 생각하면 된다.

✅ 고로, 모든 `font-size`와 `margin`, `padding`을 초기화 하고 싶으면 reset.css를


일부 브라우저 스타일을 사용하여, 스타일링을 하고 싶다면 normalize.css를 추천한다.

# 시맨틱 태그(Semantic Tags) 란?

- 태그 자체가 해당 요소의 의미나 목적을 명확히 표현해주는 태그이다.

## 시맨틱 태그를 사용하면 좋은 점

1. 웹 페이지의 구조가 명확해진다.
    - <header>, <article>, <section> 등 각 요소의 역할이 명확해지기 때문에 구조와 내용에 대한 빠른 이해가 가능하다.
2. 검색 엔진 최적화(SEO)
    - 시맨틱 태그를 사용하게 될 경우 검색 엔진이 해당 페이지의 콘텐츠를 더 잘 이해하기때문에, SEO가 향상되고, 검색 순위에서 우선 노출이 될 가능성이 높다.
3. 접근성
    - 시맨틱 태그의 경우 태그 자체의 역할이 분명하기에 스크린 리더와 같은 기술이 사용되었을 때, 웹 페이지를 더 잘 이해할 수 있게 도와줍니다.

## 의미를 명확히 해주는 시맨틱 태그들

- header
    - 페이지의 머릿글로 로고, 제목, 내비게이션 등이 포함 된다.
- nav
    - 내비게이션 메뉴를 나타낸다.
- main
    - 해당 페이지의 주 컨텐츠를 나타낸다.
- article
    - 독립적인 컨텐츠를 뜻할때 나타낸다. 예) 뉴스 기사, 블로그 게시물
- section
    - 주제 별 컨텐츠를 그룹화 할때 사용한다.
- aside
    - 메인 컨텐츠와는 별개의 정보가 포함 되거나, 사이드바를 뜻한다.
- footer
    - 페이지의 하단부에 위치하는 정보를 뜻한다. 주로 저작권 정보, 연락처, 약관 등이 포함된다.

> 👉🏻 **사용자가 브라우저 font-size를 변경하였을 때, 이에 대응해서 웹 화면의 요소가 크거나 작아지도록 하려면 어떻게 스타일링 하면 좋을까요?**


우선 css 스타일링의 단위에는 `px, rem, em, %, vw` 가 있습니다.


그 중 `px`을 제외한 `rem, em, %, vw`는 **상대 단위** 입니다.


즉, `px`은 고정 단위이기 때문에 `rem, em, %, vw`를 사용하여, 사용자 브라우저 크기에 따른 유연한 표현이 가능합니다.


# 1. `rem`


`rem` 단위는 <u>**html 루트 요소의 글꼴 크기를 기준으로 상대 크기를 설정하는 단위**</u> 입니다.


별도에 html 글꼴 크기를 지정하지 않았다면, 기본 폰트 크기는 16px 이므로, 1rem은 16px로 설정 됩니다


💡브라우저에서 사용자가 기본 글꼴 크기를 18px로 변경하면, 모든 rem 단위의 크기도 자동으로 맞춰 조정 됩니다.


```css
html {
  font-size: 16px; /* 기본 설정 */
}

h1 {
  font-size: 2rem; /* 32px */
}

p {
  font-size: 1rem; /* 16px */
}

.container {
  padding: 1.5rem; /* 24px */
}
```


## 2. `em` 


**`em`**은 해당 요소의 <u>**상위 요소 글꼴 크기를 기준으로 크기를 설정하는 단위**</u>입니다.


부모 요소의 폰트 크기를 기준으로 상대적인 크기를 적용하고자 할 때 유용합니다.


```css
.container {
font-size: 1rem; /* 기본 글꼴 크기 16px */
}

p {
font-size: 1.5em; /* 부모 폰트 크기(1rem) * 1.5 = 24px */
}
```


## 3. `%(퍼센트)`


`%(퍼센트)` 단위는 <u>**부모 요소의 크기를 기준으로 비율을 적용할 때 사용**</u>합니다.


💡 폰트 크기뿐만 아니라 요소의 크기나 레이아웃에 대한 상대적 비율을 설정할 때 유용합니다.


```css
html {
  font-size: 100%; /* 기본적으로 16px */
}

.container {
  font-size: 150%; /* 16px * 1.5 = 24px */
}
```


## (번외) 🤔 `vw(viewport width)`는 정답이 아닌가요?


`vw(viewport width)` 단위의 경우 상대 단위는 맞지만, 말그대로 웹 페이지가 렌더링 되는 화면 영역을 기준으로 하기때문에 질문에서의 <u>**‘font-size 의 변경’**</u> 이라는 부분이 질문의 의도에 부합하지 않다고 생각했습니다.

- vw의 경우 주로 반응협 웹 디자인에서 화면 크기에 따라 요소의 크기가 동적으로 조정 되도록 할 때 유용하게 사용 됩니다.
- 해당 질문이 **‘font-size를 변경했을 때’**가 아닌, **‘사용자의 브라우저의 크기가 줄어들거나, 커졌을 때’** 였다면, vw 단위도 포함 시켜야 합니다.

> 👉🏻 **Git에서 branch merge 방법들과 각 방법의 특징을 설명해 주세요.**

- Git에서 브랜치를 병합하는 방법 : **`merge`**, **`rebase`**, **`cherry-pick`**, **`squash`**

### 1. `git merge`

- 두 브랜치를 병합하면 새로운 커밋이 생성 된다.
이 커밋은 병합된 브랜치의 히스토리를 유지한다.

```shell
git checkout main
git merge feature-branch
```

- 히스토리가 그대로 유지되므로 나중에 작업 이력을 추적하기 쉽다.
- 팀워크 시 히스토리를 보존하여 작업 내용을 명확히 이해할 수 있다.
- 하지만 병합 커밋이 추가되어 히스토리가 복잡해질 수 있다.

### 2. `git rebase`

- 브랜치의 히스토리를 재작성하여 최신 커밋 위로 변경 사항을 이동시킨다.
결과적으로 병합 커밋이 생기지 않는다.

```shell
git checkout feature-branch
git rebase main
```

- 히스토리가 깔끔하고 이해하기 쉬운 일렬의 커밋으로 유지됨.
- 히스토리를 변경하기 때문에 이미 공유된 브랜치에 대해 사용하는 경우 주의가 필요하며,
잦은 충돌이 발생 할 수 있다.

### 3. `git cherry-pick`

- 다른 브랜치의 특정 커밋만을 선택적으로 현재 브랜치에 적용할 수 있다.
- 전체 브랜치를 병합하지 않고 필요한 커밋만 선택적으로 적용할 수 있다.

```shell
git checkout main
git cherry-pick <commit-hash>
```

- 특정 커밋만 선택하여 적용할 수 있어 유연하게 사용이 가능하다.

### 4. `git merge --squash`

- 브랜치의 모든 변경 사항을 단일 커밋으로 현재 브랜치에 적용된다.

```shell
git checkout main
git merge --squash feature-branch
git commit -m "Squashed commit message"
```

- 커밋 히스토리가 간결해지고, 복잡한 병합 히스토리가 없어진다.
단, 여러개의 히스토리를 하나로 합치는 것이기에 개별 커밋이 사라지므로 세부 히스토리 추적이 어렵다.

> 👉🏻 **Git Flow 브랜치 전략에 대해 설명해 주세요.**

- Git Flow는 Git을 사용하는 프로젝트에서 브랜치를 관리하는 방법론 중 하나이다.
- 이 전략은 브랜치와 작업 흐름을 체계화하여 협업과 배포를 보다 효율적으로 만드는 데 도움을 준다.

### Git Flow 브랜치 종류

1. **`master`**  : 배포 가능한 상태를 유지
2. **`develop`**  : 개발 중인 코드가 포함된 브랜치(기능 + 통합 작업)
3. **`feature`**  **:** 특정 기능을 개발하는 브랜치. `develop` 브랜치에서 분기하여 생성 됨
4. **`release`**  **:** 배포 준비가 완료된 상태의 브랜치 (일명 QA 브랜치)
5. **`hotfix`**  : 긴급한 버그 수정 브랜치(`master`에서 분기하여 생성)

### Git Flow 사용 흐름

1. **기능 개발**: `develop` 브랜치에서 `feature` 브랜치를 생성하여 기능 개발을 진행
⇒ 완료 후 `develop` 브랜치에 merge
2. **릴리스 준비**: `develop` 브랜치에서 `release` 브랜치를 생성하여 배포 준비
⇒ 완료 후 `master`와 `develop` 브랜치에 merge
3. **긴급 수정**: `master` 브랜치에서 `hotfix` 브랜치를 생성하여 긴급한 버그를 수정
⇒ 완료 후 `master`와 `develop` 브랜치에 merge

### Git Flow의 장점

- 각 브랜치의 역할과 사용 시점이 명확하여 관리가 쉽다.
- `master` 브랜치에 항상 안정된 코드가 유지됩니다.
- 기능 개발과 버그 수정이 분리되어 팀원들이 동시에 작업할 수 있습니다.

### Git Flow의 단점

- 브랜치가 많아지면서 관리가 복잡할 수 있습니다.
- Git Flow의 규칙과 흐름을 이해하고 적용하는 데 시간이 걸릴 수 있습니다.

## 🔍 또다른 브랜치 전략법


### 트렁크 브랜치 전략(Trunk-Based Development)

    - 트렁크 브랜치 전략(Trunk-Based Development)은 Git Flow와 같은 복잡한 브랜치 전략과는 달리, 더 간단하고 빠른 개발 방식을 지향하는 전략이다.
    - 이 방법은 브랜치를 최소화하고, 지속적으로 메인 브랜치(트렁크)로 통합하여 코드의 일관성과 배포의 신속성을 강조하기 위해 사용된다.

    ### 트렁크 브랜치 전략의 개념

    1. **트렁크 브랜치 (Trunk)**
        - 트렁크 브랜치는 메인 브랜치로, 모든 개발 작업이 통합되는 브랜치.
         보통 `main`이나 `master`라는 이름으로 불립니다.
    2. **짧은 라이프사이클의 브랜치**
        - 기능 개발이 필요한 경우, 짧은 기간 동안 기능 브랜치를 만들 수 있지만,
        이 브랜치는 대부분 빠르게 트렁크 브랜치에 병합 되어야 한다.
        - 기능 개발 후에는 자주, 짧은 간격으로 트렁크 브랜치에 병합한다.
        이는 개발 중 코드의 통합을 지속적으로 진행하여 충돌을 줄이고 일관성을 유지하기 위함이다.
    3. **통합과 배포의 빈도**
        - 코드 변경 사항이 자주 통합되어야 한다. 그 이유는 일관성을 유지하기 위함이다.

    ### 트렁크 브랜치 전략의 사용 예시

    - 개발자는 트렁크 브랜치에서 직접 작업하거나, 기능 브랜치를 짧은 기간 동안 생성하여 작업 후 즉시 병합해서 사용한다.
    - 병합 후, 자동화된 테스트 및 배포 프로세스를 통해 코드를 검증하고 배포 진행

    ### 트렁크 브랜치 전략의 장점

    - 브랜치가 최소화되어 관리가 쉽다.
    - 코드 변경 사항이 빨리 통합되고 배포되므로 피드백을 빠르게 받을 수 있다.
    - 모든 변경 사항이 트렁크 브랜치에서 관리되므로 코드베이스의 일관성을 유지할 수 있다.

    ### 트렁크 브랜치 전략의 단점

    - 기능 개발을 자주 트렁크 브랜치에 통합해야 하므로, 개발자에게 부담이 될 수 있음.
    - 대규모 프로젝트에서는 트렁크 브랜치만으로 모든 변경 사항을 통합하는 것이 어려울 수 있다.
    이 경우, 자동화된 배포와 테스트 시스템이 별도로 요구 될 수 있다.

# **1. 이벤트 버블링 (Event Bubbling)**

- 이벤트가 발생한 요소에서 시작하여, 상위 요소로 전파되는 과정.
이벤트는 최상위 부모 요소까지 올라가면서 각 요소에서 처리될 수 있다.
- 이벤트 버블링을 막기 위해 `event.stopPropagation()` 메서드를 사용하여 이벤트의 전파를 중단할 수 있다.
- 이벤트가 하위 요소에서 상위 요소로 전파됨

# **2. 이벤트 캡처링 (Event Capturing)**

- 이벤트가 최상위 부모 요소에서 시작하여, 하위 요소로 전파되는 과정.
이벤트는 부모 요소부터 시작하여 자식 요소로 내려갑니다.
- 이벤트가 상위 요소에서 하위 요소로 전파됨

# **3. 이벤트 위임 (Event Delegation)**

- 특정 요소가 아니라 그 부모 요소에 이벤트 리스너를 설정하여, 자식 요소에서 발생한 이벤트를 처리하는 기법
이렇게 하면 자식 요소가 동적으로 추가되거나 제거될 때에도 이벤트 리스너가 유지된다.
- 많은 자식 요소에게 개발적으로 이벤트 리스너를 추가 할 필요가 없어서 성능이 향상된다.
- 부모 요소에 이벤트 리스너를 설정하여 자식 요소에서 발생한 이벤트를 처리하는 기법
- `event.target.closest('tag')`는 클릭한 요소에서 시작하여 가장 가까운 td 조상 요소를 찾아 반환합니다.

# 캡처링과 위임의 차이


| 특성     | 이벤트 캡쳐링                | **이벤트 위임**                     |
| ------ | ---------------------- | ------------------------------ |
| **정의** | 상위 요소에서 하위 요소로 이벤트 전파  | 부모 요소에 리스너를 설정하여 자식 요소의 이벤트 처리 |
| 사용 목적  | 이벤트 전파 방식을 제어          | 동적으로 추가된 요소에서의 이벤트 처리          |
| 동작 방식  | true 플래그로 설정된 리스너에서 발생 | 부모 요소에서 발생한 이벤트를 자식 요소가 처리     |
| 예시     | 특정 이벤트의 처리 순서를 제어      | 이벤트를 효율적으로 처리하기 위해 부모에 리스너 추가  |

# **렉시컬 스코프란?**


렉시컬 스코프(Lexical Scope)는 프로그래밍 언어에서 <u>**변수의 유효 범위를 결정**</u>하는 중요한 개념이다.


렉스컬 스코프 자체는 <u>**함수 선언 위치에 따라 결정되는 상위 스코프**</u> 라고 할 수 있다.

- 함수가 <u>**선언된 위치**</u>에 따라 변수가 접근 가능한지를 정의한다.
- 함수의 스코프는 그 함수가 작성된 위치에 의해 결정된다.
- javasciprt는 렉시컬 스코프 방식만 따른다.

## **렉시컬 스코프의 작동 방식**


렉시컬 스코프의 작동 방식은 우선, 함수가 선언된 위치에서 스코프가 결정된다.


이때 변수는 함수 내에서뿐만 아니라 외부 스코프에서도 접근할 수 있다.

- **스코프 체인**
    - 함수가 호출될 때, 해당 함수의 스코프 내에서 변수 값을 찾고, 만약 그 변수가 존재하지 않으면 외부 스코프에서 검색하는데 이를 _**스코프 체인**_이라고 한다.

### 예시 코드


```javascript
function outFunction() {
  const outVariable = '외부 변수';

  function inFunction() {
    console.log(outVariable); // outFun의 스코프에서 outVariable을 참조함.
  }

  inFunction(); // '외부 변수' 출력
}

outFunction();
```


위 예제코드를 보면 `inFunction` 함수는 `outFunction` 함수 내부에 선언 되었다.


그리고 `inFunction` 함수는  `inFunction` 함수의 스코프인 `outVariable` 변수를 참조하여, ‘외부 변수’를 출력하고 있다.


👉🏻 이처럼 _**함수가 선언된 위치에 따라 외부 스코프의 변수에 접근할 수 있다는 점**_은 <u>**렉시컬 스코프의 핵심**</u>이라고 할 수 있다.

# HTTP 메소드란?

- HTTP 메소드는 클라이언트가 서버에 요청을 보낼 때 사용하는 방법을 정의하는 일련의 동작이다.
- 각 메소드는 특정한 목적을 가지고 있으며, 웹 애플리케이션에서 자원을 요청하거나 수정할 때 사용된다.

## 주요 HTTP 메소드

1. **`GET`** **:** 서버에서 자원을 요청할 때 사용 ( 가져오기 )
    - 데이터 조회에 사용되며, 서버의 상태를 변경하지 않는다.
    - URL에 쿼리 문자열을 추가하여 요청할 수 있는 데이터를 전달할 수 있다.
    - 요청된 데이터는 서버에 저장되지 않으며, 브라우저의 캐시에 저장될 수 있다.
1. **`POST`** **:** 서버에 데이터를 제출하거나 자원을 생성할 때 사용 ( 데이터 제출 및 생성 )
    - 요청 본문(body)에 데이터를 포함시켜 서버에 전송한다.
    - 서버의 상태를 변경하거나 새로운 자원을 생성할 수 있다.
1. **`PUT`** **:** 서버의 특정 자원을 업데이트하거나 새 자원을 생성할 때 사용 ( 업데이트 )
    - 요청 본문에 포함된 데이터로 자원을 완전히 대체 할 수 있다.
    - 자원이 존재하지 않으면 새로운 자원을 생성한다.
1. **`PATCH`** **:** 서버의 특정 자원을 부분적으로 수정할 때 사용 ( 일부 업데이트 )
    - PUT 메소드와 달리 전체 자원을 대체하지 않고, 필요한 부분만 수정한다.
1. **`DELETE`** **:** 서버에서 특정 자원을 삭제할 때 사용 ( 삭제 )
    - 요청된 자원을 서버에서 삭제하는 작업을 수행합니다.
1. **`HEAD`** **:** 서버에서 자원의 헤더 정보만 요청할 때 사용 ( 헤더 정보 요청 )
    - 서버의 상태를 확인하거나 캐시된 데이터의 유효성을 검사하는 데 사용된다.
    - 응답 본문 없이, 상태 코드와 헤더 정보만 포함되어있다.
1. **`OPTIONS`** **:** 서버에서 지원하는 HTTP 메소드에 대한 정보를 요청할 때 사용 ( 메소드 정보 요청 )
    - 특정 자원에 대해 클라이언트가 사용할 수 있는 메소드 목록을 가져오는 데 사용된다.

# AJAX에 대해서..


## AJAX란 무엇인가?

> 웹 애플리케이션이 **페이지를 다시 로드하지 않고도 서버와 데이터를 주고받을 수 있게** 해주는 기술
- 웹 페이지가 **동적이고 대화형**으로 동작하도록 도와주며, 백그라운드에서 서버와 통신하는 방식으로 동작한다.
- 이로써 사용자 경험이 크게 향상된다는 장점이 존재한다.

## 주요 특징

- **비동기적으로 동작한다.** ⇒ 요청을 보내고나서 응답을 기다리지않고, 그 사이 다른 작업을 처리 할 수 있다.
- **JSON을 사용한다.** ⇒ 기존에는 XML 데이터로 주고받는 방식이지만, 좀 더 다루기 쉬운 JSON으로 대체 되었다.
- **페이지를 새로고침 없이 데이터를 갱신한다.** ⇒ 필요한 데이터만 렌더링하여, **페이지의 일부만 업데이트가 가능**하다.

## 구성 요소

1. HTML/CSS ⇒ 페이지 콘텐츠 및 스타일 정의
2. JavaScript ⇒ AJAX 요청 생성 & 응답 처리 역할
3. XMLHttpRequest 객체 ⇒ 서버와 데이터를 주고받기위한 주요 매커니즘. JS에서 비동기 요청을 보낼 수 있게 해주는 역할
4. server ⇒ request 처리와 response 반환
5. JSON/XML ⇒ 서버와 클라이언트 간에 데이터를 주고 받을 때 사용되는 형식

## 동작 과정

1. 사용자 액션 수행
2. XMLHttpRequest 객체 생성 후 서버에 비동기적으로 request 요청
3. 서버가 요청 처리 후 데이트를 클라이언트로 response 반환
4. 클라이언트는 서버로부터 받은 데이터를 처리해 웹 페이지 일부를 업데이트 함

## AJAX를 사용하는 이유?


### AJAX 이전 전통적인 DOM 동작 방식

- **서버와의 데이터 교환이 필요할 때 전체 페이지를 다시 로드하는 방식**을 사용
    1. 동작 수행 시, 브라우저는 서버에 HTTP 요청
    2. 서버에서 요청 처리 후, 전체 HTML 페이지를 반환
    3. 브라우저는 기존 페이지를 제거하고, 서버로 부터 받은 새로운 HTML 페이지를 다시 렌더링

위 처럼 전통적인 DOM의 방식을 생각해보면, DOM 동작은 **새로운 HTML 페이지를 전체적으로 다시 렌더링** 하게 된다.


### 전통적인 DOM 동작 방식의 단점

- 페이지 전체가 새로고침 되면서, 페이지의 모든 컨텐츠를 다시 로드하고 렌더링 해야하기 때문에 (리렌더링,리페인팅) 응답 시간이 길어지게 된다.
- 단순 데이터 변경이나 일부 UI 업데이트를 위해 전체 페이지를 다시 렌더링 하기 때문에 불필요한 데이터 전송이 발생해 서버의 부하를 일으킨다.

**AJAX**를 사용하는 이유는 웹 페이지의 **동적인 사용자 경험**을 개선하고, **효율적인 데이터 갱신**을 통해 **응답성과 대화형 인터페이스**를 제공하기 위해서입니다. 이를 이해하기 위해 AJAX를 사용하지 않았던 **이전의 전통적인 DOM 동작 방식**과 비교해 설명해볼게요.


### **전통적인 방식과 AJAX 방식의 비교**


| **항목**       | **전통적인 방식 (페이지 새로고침)**    | **AJAX 방식 (비동기 요청)**        |
| ------------ | ------------------------- | --------------------------- |
| **페이지 업데이트** | 페이지 전체가 다시 로드됨            | 필요한 부분만 선택적으로 업데이트          |
| **사용자 경험**   | 페이지가 새로고침되어 **깜박임** 발생    | 페이지가 **매끄럽게 유지**되며 동적임      |
| **서버 요청**    | 매 요청마다 전체 HTML을 요청        | 필요한 **데이터만 요청**하고 최소한의 응답   |
| **응답 시간**    | 페이지 전체를 다시 로드하므로 **느림**   | 필요한 데이터만 로드하여 **빠름**        |
| **성능**       | **서버 부하가 큼**, 불필요한 리소스 전송 | **서버 부하 감소**, 네트워크 비용 절감    |
| **사용 사례**    | 전통적인 **폼 제출**이나 **링크 클릭** | **동적 콘텐츠 갱신** (예: 검색 자동 완성) |


## 결론

1. **향상된 사용자 경험**: AJAX는 페이지 새로고침 없이 필요한 부분만 **갱신**할 수 있기 때문에 사용자 경험이 **매끄럽고 반응성**이 좋다.
2. **빠른 응답성**: 전통적인 방식에서는 사용자가 어떤 요청을 할 때마다 페이지 전체를 다시 로드해야 했지만, AJAX는 필요한 데이터만 주고받으므로 **빠른 응답**을 제공한다. 이로 인해 서버와의 통신이 **효율적**으로 이루어진다.
3. **효율적인 서버 사용**: AJAX는 서버에 **불필요한 전체 페이지 요청**을 보내지 않기 때문에, 서버 자원을 보다 **효율적**으로 사용할 수 있다. 단순히 특정 데이터를 업데이트하는 데 전체 HTML을 로드하는 것보다, 필요한 데이터만 요청하여 서버의 부하를 줄이기 때문이다.
4. **대화형 인터페이스 제공**: AJAX는 **대화형 웹 애플리케이션**을 가능하게 만든다. 예를 들어, 사용자가 입력한 텍스트를 실시간으로 서버에 보내어 **자동 완성** 기능을 제공하거나, 페이지의 특정 요소만 즉각적으로 업데이트할 수 있기 때문이다.

# 자바스크립트의 실행 컨텍스트


## 실행 컨텍스트란?

> 코드가 실행되는 환경
- JS는 싱글 스레드 기반 동기적 언어로, 코드를 실행하기 전에 필요한 변수, 함수, 객체 등을 준비하는 과정을 거친다.

## 실행 컨텍스트의 역할

1. 변수나 함수의 선언과 초기화를 관리 한다.
2. 코드의 실행 순서를 제어한다.
3. 실행 중인 스코프와 this 값을 관리한다.

## 구성 요소

- **변수 객체** : **변수, 함수 선언** 및 **함수 인자** 등을 저장하는 객체. 실행 컨텍스트가 생성되면 먼저 변수 객체가 생성되고, 이 안에 **변수와 함수**가 저장된다.
- **스코프 체인** : 스코프 체인은 **변수나 함수를 참조**할 때 접근 가능한 스코프의 목록이다. 스코프 체인을 통해 **중첩된 함수**에서도 외부 함수나 전역에서 선언된 변수에 접근할 수 있다.
- **this 바인딩** : 실행 컨텍스트가 생성될 때마다 **this** 값이 결정된다. **this**는 함수가 **어디서 호출되었는지**에 따라 값이 달라지는데, 전역 컨텍스트에서 **this**는 전역 객체(window)를 가리키고, 객체의 메서드 내부에서는 그 **객체**를 가리킨다.

## 동작 과정

1. **생성 단계** : **변수, 함수 선언**이 변수 객체에 등록됩니다. 이때 변수는 `undefined`로 초기화되고, 함수는 실제 함수로 초기화됩니다. 이 과정을 호이스팅(Hoisting)이라고 합니다.
2. **실행 단계** : 코드가 한 줄씩 실행되며, 변수에 값이 할당되고 함수가 호출됩니다.

## 실행 컨텍스트의 예시 코드


```javascript
function outer() {
  var x = 10;
  function inner() {
    console.log(x); // 10
  }
  inner();
}
outer();
```


이 코드에서는 **`outer`** 함수가 호출되면 새로운 실행 컨텍스트가 생성됩니다. 그 후 `inner` 함수가 호출되면 `inner` 함수만의 **새로운 실행 컨텍스트**가 생성되지만, **스코프 체인** 덕분에 `outer` 함수의 변수인 `x`에 접근할 수 있습니다.


### 실행 컨텍스트 스택

> 실행 컨텍스트는 **스택** 형태로 관리됩니다. 자바스크립트 코드가 실행될 때마다 새로운 컨텍스트가 생성되고, 이를 "실행 컨텍스트 스택(Execution Context Stack)"에 쌓아 올립니다. 가장 마지막에 쌓인 컨텍스트가 **현재 실행 중인 컨텍스트**입니다.

```javascript
function a() {
  console.log('a');
}
function b() {
  a();
  console.log('b');
}
b();
```

- `b()`가 호출되면 `b`의 실행 컨텍스트가 생성되어 스택에 추가됩니다.
- 그 후 `a()`가 호출되면 `a`의 실행 컨텍스트가 스택에 추가되고, `a()` 실행이 완료되면 해당 컨텍스트는 스택에서 제거됩니다.
- 마지막으로 `b()`의 실행이 끝나면, `b`의 실행 컨텍스트도 스택에서 제거됩니다.

## 비동기 처리 과정에서의 실행 컨텍스트


## 비동기 처리의 실행 컨텍스트

# 구성

- **콜 스택(Call Stack)**: 자바스크립트가 **함수를 호출**할 때, 해당 함수의 실행 컨텍스트가 콜 스택에 추가됩니다. 함수가 종료되면 실행 컨텍스트가 콜 스택에서 제거됩니다.
- **Web APIs**: 브라우저 환경에서 비동기 작업(예: 타이머, HTTP 요청 등)을 처리하기 위해 사용되는 API들입니다. 이들은 자바스크립트 엔진이 아닌 **브라우저**가 제공하는 기능입니다.
- **콜백 큐(Callback Queue)**: 비동기 작업이 완료되면 해당 콜백 함수가 **콜백 큐**에 추가됩니다. 콜백 큐에 있는 함수는 콜 스택이 비워졌을 때 이벤트 루프에 의해 콜 스택으로 이동하여 실행됩니다.
- **이벤트 루프(Event Loop)**: 이벤트 루프는 **콜 스택이 비어 있는지 확인**하고, 비어 있다면 콜백 큐에 있는 작업을 콜 스택으로 가져와 실행합니다. 이를 통해 비동기 작업이 실행됩니다.

# **비동기 작업에서 실행 컨텍스트의 흐름**


비동기 작업이 실행될 때 **실행 컨텍스트가 어떻게 동작**하는지 알아보겠습니다. 자바스크립트는 비동기 작업(예: `setTimeout`, `fetch`)을 실행할 때 **콜 스택, Web APIs, 콜백 큐, 이벤트 루프**를 활용합니다. 아래에서는 `setTimeout`의 예시를 통해 실행 컨텍스트가 어떻게 동작하는지 설명합니다.


### 예시 코드:


```javascript
console.log('Start');

setTimeout(() => {
  console.log('Callback');
}, 2000);

console.log('End');
```


### 동작 과정:

1. **동기 코드 실행**:
    - 첫 번째로 `console.log('Start')`가 호출됩니다.
    - **`console.log('Start')`****의 실행 컨텍스트**가 생성되고 **콜 스택**에 추가됩니다.
    - `'Start'`가 출력된 후, **`console.log`****의 실행 컨텍스트**는 **콜 스택에서 제거**됩니다.
2. **비동기 함수 호출 (****`setTimeout`****)**:
    - `setTimeout` 함수가 호출되면, **`setTimeout`****의 실행 컨텍스트**가 생성되어 **콜 스택**에 추가됩니다.
    - `setTimeout`은 **비동기 작업**으로, 2초 후에 실행해야 할 콜백 함수를 **Web APIs**에 등록합니다.
    - 이후 **`setTimeout`****의 실행 컨텍스트**는 **콜 스택에서 제거**됩니다. 이 시점에서 2초 타이머가 시작됩니다.
3. **동기 코드 실행 (****`console.log('End')`****)**:
    - 다음으로 `console.log('End')`가 호출되며, **`console.log('End')`****의 실행 컨텍스트**가 생성되어 **콜 스택**에 추가됩니다.
    - `'End'`가 출력된 후, **`console.log`****의 실행 컨텍스트**는 **콜 스택에서 제거**됩니다.
4. **비동기 작업 완료**:
    - **2초 후**, `setTimeout`의 콜백 함수가 **Web APIs**에서 완료되고 **콜백 큐**로 이동합니다.
    - 이벤트 루프는 **콜 스택이 비어 있는지** 확인합니다. 현재 콜 스택이 비어 있으므로, **콜백 큐에 있는 콜백 함수**가 **콜 스택으로 이동**하여 실행됩니다.
5. **콜백 함수 실행**:
    - 콜백 함수(`console.log('Callback')`)가 콜 스택에 추가되고 실행됩니다.
    - **`console.log('Callback')`****의 실행 컨텍스트**가 생성되어 **콜 스택**에 추가됩니다.
    - `'Callback'`이 출력된 후, **`console.log`****의 실행 컨텍스트**는 **콜 스택에서 제거**됩니다.

### 출력 결과:


```plain text
Start
End
Callback
```


# **실행 컨텍스트의 역할**


비동기 작업에서 실행 컨텍스트는 주로 **콜 스택**에 쌓여서 **현재 실행 중인 함수의 환경**을 관리합니다. **콜백 함수가 언제 실행되는지**를 결정하는 것은 **이벤트 루프와 콜백 큐**이며, 비동기 작업이 완료되면 **실행 컨텍스트가 콜 스택에 추가**되어 비동기 코드를 실행하게 됩니다.

- **콜 스택**은 현재 실행 중인 동기 코드(실행 컨텍스트)를 관리합니다.
- **비동기 작업**은 콜 스택에서 실행되지 않고, **Web APIs**를 통해 관리됩니다.
- **콜백 큐**와 **이벤트 루프**는 비동기 작업이 완료되면 해당 콜백을 콜 스택에 추가하여 실행합니다.

# **정리**

- **실행 컨텍스트**는 **콜 스택**에 쌓이며, 코드 실행의 환경을 정의합니다.
- **비동기 작업**은 실행 컨텍스트가 콜 스택에서 바로 처리되지 않고, **Web APIs**에 의해 비동기로 관리되며, 이후 **콜백 큐**와 **이벤트 루프**를 통해 실행됩니다.
- **이벤트 루프**는 **콜 스택이 비어 있는지** 확인하고, 비어 있으면 **콜백 큐에 있는 함수**를 콜 스택으로 이동시켜 **실행 컨텍스트**로 만들어 실행합니다.

> 리액트에서 Virtual Dom이 무엇이고, 이를 사용하는 이유에 대해서 설명하세요.

# Virtual DOM이란?

- 실제 DOM 트리를 가상으로 복제하여 추상화한 JS 객체이다.

## 사용하는 이유?


실제 DOM의 렌더링 과정은 무척 복잡하다.


이미 그려낸 웹 브라우저에서 특정 요소가 바뀐다면, 다시 처음부터 렌더링을 시작하여, 화면을 그려내야한다.


만약 많은 양의 Node를 포함 할 경우 DOM 생성 과정이 오래걸리고 그만큼 성능 저하가 발생할 수 있다.


이렇게 특정 요소의 변화가 일어났을때 다시 화면을 그려내는 것을 _**리플로우(레이아웃 변화로 인해 요소들의 위치와 크기를 다시 계산하는 과정)**_와 _**리페인팅(화면의 시각적요소가 변경되면 그 변경된 시각적요소를 다시 그리는 과정)**_이라고하는데,


버츄얼 돔은 이 과정들을 최소화하여, 성능 향상에 도움을 주기 위해 나타났다.


우선, 버츄얼 돔을 사용하는 리액트의 경우를 보면, 처음 DOM이 그려지고 변화가 일어날때, 해당 DOM의 가상 복제본을 두개를 생성하게 된다. 이름 그대로 **가상의 복제 DOM을 데이터에 저장**해놓는다.


첫번째 가상돔은 변경되기 전의 원본 DOM 복제본, 두번째 가상돔은 변경된 후의 DOM 복제본이다.


이중에 변경된 사항만 체크하여, 변경한 부분만 실제 DOM에 전달하게 된다.


해당 가상돔을 이용하게 될 경우, 노드의 변화가 일어났을때 처음부터 다시 그리는 방식이 아닌 변경된 부분만 최소화하여, 렌더링 하게 되니 불필요한 리플로우와 리페인팅을 줄일 수 있다.


이로 인해 가져오는 이점은 웹 페이지 성능이 효율적으로 향상 시킬 수 있다는 이점이 있다.

> 리액트에서 배열을 렌더링 할때 key를 써야하는 이유에 대해 설명해주세요.

# 배열 렌더링 시 key를 써야하는 이유?


key는 각 요소를 명확하게 식별할 수 있는 고유한 값을 나타낸다.


배열의 각 요소에 key를 지정함으로써 요소들이 추가,변경,삭제 되었을때, 특정 고유 키 값만 확인하여, 해당 변경사항에 대해 빠르게 렌더링 할 수 있다.


key를 지정하지않게된다면 리액트가 처음부터 해당 배열을 순회하면서 변화되는 요소를 찾아서 수정하는 동작을 거치게 되는데, 이렇게 동작할 경우 고유한 값이 없기때문에 개발자가 의도한대로 동작하지 않을 수도 있으며, 해당 배열의 전체를 순회하며 렌더링 되기때문에 불필요한 렌더링이 발생하게 된다. 이는 곧 성능저하의 원인이 된다.


따라서 key를 사용하게 되면 불필요한 렌더링을 방지해주며, 효율적이게 리렌더링을 할 수 있다.


주의해야할 사항으로는 key의 값을 index로 사용하는 경우가 있는데 index는 데이터가 추가되거나, 삭제가 될 경우에는 해당 값이 변경 될 경우 되도록 ‘고유한 값’으로 key를 지정해주는게 좋다.


## 8주차 위클리페이퍼

# **리액트 생명주기 (Life Cycle)**


리액트 컴포넌트는 생성되고, 업데이트되고, 제거되는 과정에서 일련의 생명주기 단계(Life Cycle)를 거칩니다. 리액트의 생명주기는 크게 3가지 단계로 나뉩니다.


# **마운팅(Mounting)**

> 컴포넌트가 처음 DOM에 추가되는 단계입니다.
- `constructor`: 컴포넌트가 처음 생성될 때 호출되며, 초기 상태를 설정할 수 있습니다.
- `getDerivedStateFromProps`: `props`에서 `state`를 설정하거나 갱신할 때 사용됩니다.
- `render`: JSX를 반환하며 DOM에 실제로 렌더링되는 부분입니다.
- `componentDidMount`: 컴포넌트가 렌더링된 후에 호출됩니다. _**네트워크 요청이나 DOM 접근이 필요한 작업에 주로 사용됩니다.**_

# **업데이트(Updating)**

> 컴포넌트의 상태나 `props`가 변경되어 다시 렌더링되는 단계입니다.
- `getDerivedStateFromProps`: _**마운트 때와 동일하게 호출**_되며, `props`가 변경될 때 `state`를 갱신할 수 있습니다.
- `shouldComponentUpdate`: 컴포넌트가 다시 렌더링될지 여부를 결정하는 메서드입니다. 성능 최적화를 위해 사용됩니다.
- `render`: 상태나 `props`가 변경된 내용을 기반으로 _**다시 렌더링**_됩니다.
- `getSnapshotBeforeUpdate`: _**업데이트 직전에 호출되며, DOM 변화를 기록할 수 있습니다.**_
- `componentDidUpdate`: 컴포넌트가 업데이트된 후 호출되며, 이전 `props`나 상태와 비교해서 후속 작업을 할 수 있습니다.

# **언마운팅(Unmounting)**

> 컴포넌트가 DOM에서 제거되는 단계입니다.
- `componentWillUnmount`: 컴포넌트가 DOM에서 제거되기 직전에 호출되며, 리소스를 정리하는 작업을 할 수 있습니다.

## **리액트 생명주기 요약**


리액트 컴포넌트는 _**생성(Mount), 업데이트(Update), 제거(Unmount)**_라는 주기를 겪으며, 각 단계에서 특정 메서드가 실행됩니다. 리액트에서는 `useEffect`와 같은 훅을 사용하여 생명주기 관련 작업을 처리할 수 있습니다.


# **웹 페이지 렌더링 방식: CSR, SSR, SSG**


## **1. CSR (Client-Side Rendering)**


Client-Side Rendering은 웹 페이지가 브라우저에서 렌더링되는 방식입니다. 서버에서 초기 HTML 파일만 보내고, JavaScript 파일이 다운로드된 후에 클라이언트(브라우저)에서 나머지 페이지가 렌더링됩니다. 리액트의 기본 렌더링 방식입니다.

- **초기 로딩 속도**: 초기 로딩 시간이 길어질 수 있습니다. 서버에서 빈 HTML 파일을 전달하고, 클라이언트에서 JavaScript를 로드한 후 렌더링을 진행하기 때문에 첫 화면을 보여주는 시간이 느릴 수 있습니다.
- **빠른 사용자 경험**: 한 번 페이지가 로드된 이후에는 SPA(Single Page Application)처럼 동작하여 페이지 전환이 빠르고, 사용자 경험이 부드럽습니다.
- **SEO에 불리함**: 초기에는 빈 HTML이 전달되므로, SEO(Search Engine Optimization)에 약할 수 있습니다. 구글봇은 JavaScript를 처리할 수 있지만, 대부분의 검색 엔진 크롤러는 그렇지 않기 때문입니다.
- **사용 예시**: 대부분의 대시보드, 내부 웹 애플리케이션, 즉각적인 페이지 로딩보다는 사용자 경험이 중요한 경우에 사용됩니다.

## **2. SSR (Server-Side Rendering)**


Server-Side Rendering은 서버에서 미리 HTML을 렌더링해서 클라이언트로 보내는 방식입니다. 서버가 요청을 받을 때마다 HTML을 생성하고 이를 브라우저에 전달합니다. React에서는 Next.js 같은 프레임워크를 사용해 SSR을 구현할 수 있습니다.

- **빠른 첫 화면**: 서버에서 HTML을 완성한 상태로 보내기 때문에, 초기 화면 로딩 속도가 빠릅니다.
- **SEO에 유리**: 검색 엔진이 완성된 HTML을 받을 수 있어, SEO 최적화에 좋습니다.
- **서버 부하**: 매 요청마다 HTML을 생성해야 하므로, 서버에 부하가 클 수 있습니다.
- **사용 예시**: SEO가 중요한 블로그, 마케팅 사이트, 이커머스 사이트 등에서 사용됩니다.

## **3. SSG (Static Site Generation)**


Static Site Generation은 빌드 타임에 모든 페이지를 미리 렌더링하여 정적인 HTML 파일을 생성하는 방식입니다. 이 방식은 초기 페이지 로딩이 빠르면서도 SSR처럼 검색 엔진 친화적입니다.

- **최상의 성능**: 빌드 타임에 미리 생성된 정적 파일을 제공하기 때문에, 서버에서 동적 렌더링이 필요 없고, 매우 빠른 로딩 속도를 자랑합니다.
- **SEO에 유리**: 서버에서 정적 HTML을 제공하므로, SEO 최적화에 유리합니다.
- **변경에 대한 한계**: 모든 페이지를 빌드 타임에 미리 생성하므로, 데이터가 자주 변경되는 페이지에는 적합하지 않습니다. 하지만 이를 보완하기 위해 Next.js에서는 ISR(Incremental Static Regeneration) 같은 방식이 제공됩니다.
- **사용 예시**: 블로그, 포트폴리오 사이트, 문서 사이트 등 자주 업데이트가 필요하지 않은 페이지에 적합합니다.

- Next는 [빌드 타임⁽¹⁾](/1464ef56099480b3be82e68a2c9c5a62#1464ef5609948074b13fcc05478b4bf1)에 HTML을 생성을 하여, **성능**과 **SEO를 극대화** 시킬 수 있습니다.

# 서버 사이드 렌더링 실행 과정

1. 사용자가 브라우저에서 특정 URL로 요청을 보냅니다.
2. Next 서버는 해당 요청을 처리하고, 페이지의 리액트 컴포넌트를 실행해 HTML을 생성합니다.
3. 서버에서 생성된 HTML이 브라우저로 반환 됩니다.
4. 브라우저는 받은 HTML을 즉시 렌더링하여 초기 화면을 빠르게 볼 수 있도록 합니다.

# Hydration

> Hydration은 SSR로 서버에서 생성된 HTML에 리액트의 인터랙티브 기능을 연결하는 과정이다.

## 작동 원리


### 1. 서버 렌더링

- 리액트 컴포넌트 실행 후 완성된 HTML 생성
- 해당 생성된 HTML은 브라우저로 전달되어 즉시 렌더링 됨

### 2. 클라이언트에서 리액트 초기화

- 브라우저는 서버에 전달된 HTML을 파싱해 화면에 렌더링 한다.
- 이후, 클라이언트는 JS 번들을 로드한다.

### 3. 리액트와 서버 HTML 결합

- 리액트는 클라이언트에서 서버 렌더링된 HTML을 재분석하고 리액트의 컴포넌트 상태와 이벤트 핸들러를 연결한다. (핸들러 추가)

### 4. 완료

- 서버에서 제공된 HTML은 리액트에 의해 활성화되고 동적으로 작동하는 SPA(Single Page Application)로 전환된다.

## 중요한 개념


### 1. DOM 동기화

- 서버에서 생성된 HTML과 클라이언트에서 React가 생성하는 가상 DOM이 동일해야 한다.

### 2. JS 번들 크기

- 클라이언트가 Hydration을 완료하려면 서버에서 전달된 HTML 외에 React 애플리케이션의 JavaScript 번들도 로드해야하는데 번들이 너무 크면 Hydration 시간이 길어 질 수 있다.

### 3. Hydration 지연

- JS가 로드되기 전까지는 페이지가 인터랙티브하지않아 해당 시간동안에는 버튼 클릭과 같은 동작이 무시된다.

### 4. Lazy Hydration

- Next.js는 필요에 따라 Hydration을 지연시킬 수 있다. 예를 들어, 사용자가 뷰포트에 들어오지 않은 컴포넌트는 나중에 Hydration될 수 있다.

## 번외(styled components)


styled components는 css-in-js 방식으로 `className`이 동적으로 생성되게 된다.


그렇기때문에 Next에서 (SSR) styled components를 사용할때 주의해야 할 점이 있다.


### 문제


별도의 설정을 진행해주지않으면 `Hydration` 과정에서 스타일 깨짐 등의 이슈가 발생하게 된다.


이는 ‘스타일 미스매치’ 때문이라고 할 수 있는데


SSR에서 생성된 CSS와 클라이언트에서 생성된 CSS가 일치하지않아서 리액트에서는 경고를 표기하며, UI가 깨질 수있다.


그래서 Next에서 styled components를 사용하게 될 경우 `_document.tsx`에서 아래와 같이 별도의 설정을 해줘야한다.

- 예제

```typescript
import Document, { DocumentContext, Html, Head, Main, NextScript } from 'next/document';
import { ServerStyleSheet } from 'styled-components';

class MyDocument extends Document {
  static async getInitialProps(ctx: DocumentContext) {
    const sheet = new ServerStyleSheet();
    const originalRenderPage = ctx.renderPage;

    try {
      ctx.renderPage = () =>
        originalRenderPage({
          enhanceApp: (App) => (props) => sheet.collectStyles(<App {...props} />),
        });

      const initialProps = await Document.getInitialProps(ctx);
      return {
        ...initialProps,
        styles: (
          <>
            {initialProps.styles}
            {sheet.getStyleElement()}
          </>
        ),
      };
    } finally {
      sheet.seal();
    }
  }

  render() {
    return (
      <Html>
        <Head />
        <body>
          <Main />
          <NextScript />
        </body>
      </Html>
    );
  }
}

export default MyDocument;
```


또는 지연 로딩 or `dynamic`을 이용해서 스타일을 유지하는 방법도 있다.

- 예제

```typescript
import dynamic from 'next/dynamic';

const DynamicComponent = dynamic(() => import('../components/StyledComponent'), {
  ssr: true,
});

function Page() {
  return <DynamicComponent />;
}

export default Page;
```


styled components를 사용할때 이런 스타일 미스 매치 또는 FOUC 문제가 발생하긴 하지만 위 방식을 활용하면 FOUC 문제를 해결 할 수 있으며, styled components 만의 이점도 가져갈 수있다.


### 장점과 이점

- **SSR에서의 장점**:
    - 서버에서 스타일을 미리 렌더링하여 초기 로딩 속도가 빨라진다.
    - 스타일이 HTML과 함께 제공되므로, SEO 최적화에 유리해진다.
- **Hydration의 이점**:
    - 서버에서 제공된 스타일을 그대로 사용하므로, 클라이언트가 스타일을 다시 생성할 필요가 없어 성능이 최적화된다.
    - SSR과 Hydration이 제대로 작동하면, CSS가 재생성되지 않으므로 스타일 미스매치 경고와 UI 깨짐을 방지할 수 있다.

# CORS 에러와 해결방법


> **🤔 CORS 에러에 대해 설명하고, 어떻게 해결하면 될지 설명해 주세요.**


## 들어가며


---


### CORS란 무엇인가?


CORS(Cross-Origin Resource Sharing)는 웹 브라우저의 보안 기능으로, **다른 출처(Domain, Protocol, Port)의 리소스에 접근**하려고 할 때 발생할 수 있는 문제를 해결하기 위해 고안되었다.


## 본문


---


### CORS 에러란?


CORS 에러는 브라우저가 특정 요청을 보낼 때 서버로부터 적절한 허용 정책을 받지 못했을 때 발생합니다.


예를 들어:

- 클라이언트: `http://localhost:3000`
- API 서버: `https://api.example.com`

이 경우, `http://localhost:3000`에서 `https://api.example.com`으로 요청을 보내면 **출처가 다름으로** 브라우저는 요청을 차단하거나, 서버가 적절히 응답하지 않는 경우 에러를 발생시킵니다.


### 해결방법

1. <u>**서버에서 허용 헤더 추가**</u>
서버는 요청을 보낸 출저를 허용하기 위해 응답 헤더에 다음을 포함해야 한다.
    - `Access-Control-Allow-Origin: http://localhost:3000`
2. <u>**사전 요청(Preflight Request) 처리**</u>
브라우저는 `OPTIONS` 요청을 먼저 보내서 서버가 요청을 허용하는지 확인할 수 있다. 이 경우 서버는 다음과 같은 추가 헤더를 반환해야한다.

    ```shell
    Access-Control-Allow-Methods: GET, POST, PUT
    Access-Control-Allow-Headers: Content-Type
    Access-Control-Allow-Origin: http://localhost:3000
    ```

3. <u>**프록시 사용**</u>
개발 환경에서 CORS 문제를 간단히 해결하려면 클라이언트와 서버 간에 프록시를 사용할 수 있다.
    > 프록시(Proxy)는 클라이언트와 서버 사이에 중개 역할을 하는 서버를 뜻한다.  
    > 클라이언트가 요청을 보내면, 이 요청을 대신 받아 실제 서버에 전달하고 응답을 다시 클라이언트로 전달하는 방식으로 작동한다.  
    > 동작 원리나 종류에 대해서는 SW 지식이 필요하니 본문에서는 다루지 않았다.
    - _**React에서 프록시 설정(Vite & CRA)**_
        - V**ite**
        Vite의 경우, `vite.config.ts` 파일을 수정하여 프록시 설정이 가능하다.
            - `/api` 로 시작하는 요청이 `https://api.example.com`로 프록시됨
            - 예시: `fetch(’/api/data’)`는 실제로 `https://api.example.com/data`로 요청된다.

        ```typescript
        import { defineConfig } from 'vite';
        import react from '@vitejs/plugin-react';
        
        export default defineConfig({
          plugins: [react()],
          server: {
            proxy: {
              '/api': {
                target: 'https://api.example.com', // 실제 API 서버
                changeOrigin: true, // Origin 헤더를 대상 서버로 변경
                rewrite: (path) => path.replace(/^\/api/, ''), // '/api' 제거
              },
            },
          },
        });
        ```

        - **CRA**
        CRA에서는 `package.json`에 프록시를 설정할 수 있다.
            - 클라이언트에서 `/api/data`로 요청하면 자동으로 `https://api.example.com/api/data`로 전달된다. 하지만 이런 방식은 세부적인 경로를 지원하지 않는다.

        ```json
        // package.json
        "proxy": "https://api.example.com"
        ```

        - CRA에서 커스텀 미들웨어 사용
        더 복잡한 프록시 설정이 필요하다면, `http-proxy-middleware` 라이브러리를 사용할 수 있다.
            - 설치 : `npm install http-proxy-middleware --save`
            - 설정

            ```javascript
            // src/setupProxy.js
            const { createProxyMiddleware } = require('http-proxy-middleware');
            
            module.exports = function (app) {
              app.use(
                '/api',
                createProxyMiddleware({
                  target: 'https://api.example.com',
                  changeOrigin: true,
                  pathRewrite: { '^/api': '' }, // '/api' 제거
                })
              );
            };
            ```

    - _**Next.js에서 프록시 설정(API 라우트 & Rewrite)**_
        - API 라우트로 프록시 구현

        ```javascript
        // pages/api/propxy.js
        import { NextApiRequest, NextApiResponse } from 'next';
        
        export default async function handler(req: NextApiRequest, res: NextApiResponse) {
          const response = await fetch('https://api.example.com' + req.url, {
            method: req.method,
            headers: {
              'Content-Type': 'application/json',
              ...req.headers, // 클라이언트의 요청 헤더를 그대로 전달
            },
            body: req.body,
          });
        
          const data = await response.json();
          res.status(response.status).json(data);
        }
        
        // 클라이언트에서는 /api/proxy 로 요청
        const fetchData = async () => {
          const res = await fetch('/api/proxy/data');
          const data = await res.json();
          console.log(data);
        };
        ```

        - Rewrite 설정

        ```javascript
        // next.config.js
        module.exports = {
          async rewrites() {
            return [
              {
                source: '/api/:path*',
                destination: 'https://api.example.com/:path*', // 실제 API 서버
              },
            ];
          },
        };
        ```

4. <u>**CORS 해결을 위해 브라우저 확장 프로그램 사용(비추천)**</u>
개발 중이라면 Chrome 등의 브라우저에서 CORS 관련 확장을 설치해 임시적으로 우회할 수도 있습니다. 하지만 이는 **임시적인 해결책**이며, 프로덕션 환경에서는 사용하지 않는 것이 좋습니다.

## 마치며


---


위에 CORS 에러 해결 방법에 대해 여러가지 작성을 했는데 사실 FE 개발자로써 할 수 있는 최선의 방법은 프록시 설정을 하는것이 최선일 것 같다.


그 외에 다른 방법들은 서버 설정을 변경해야하는 부분이라서 알아두는 정도로만 인식하면 좋을듯 싶다.


# SEO와 개선 방법


> **🤔 SEO가 무엇인지 설명하고, 개선을 위해 어떤 작업을 할 수 있을지 설명해 주세요.**


## 들어가며


---


### SEO란?


SEO(Search Engine Optimization)는 **검색 엔진 최적화**를 뜻하며, 웹사이트가 검색 엔진에서 더 잘 노출되고 높은 순위를 차지하도록 최적화하는 작업을 의미한다.


이를 통해 더 많은 **트래픽**을 유도하고, 사용자 경험(UX)을 개선하며, 전환율(Conversion Rate)을 높이는 데 목적을 둔다.

> SEO 종류와 개선 방안에는 여러가지가 있겠지만 FE 개발자로써 SEO 개선을 위한 주요 작업과 기술적인 접근 방식에 대해서만 다루겠다.

## 본문


---


### 개선 방법

1. <u>**HTML 구조 및 메타데이터 최적화**</u>
검색 엔진은 HTML 구조를 기반으로 페이지의 내용을 분석합니다. 잘 짜인 HTML은 크롤링과 인덱싱에 유리합니다
    - _**적절한 태그 사용**_
        - **헤딩 태그**: `<h1>`은 페이지의 핵심 제목에만 사용하고, `<h2>`, `<h3>`를 계층적으로 사용합니다.
        - **목록 태그**: `<ul>`, `<ol>`을 활용하여 정보 계층 구조를 명확히 합니다.
    - _**메타데이터 추가**_
        - `<title>`: 페이지의 주요 내용을 간결하게 기술.
        - `<meta name="description">`: 페이지 내용을 요약하여 검색 엔진 결과에 표시.
        - `<meta name="robots">`: 페이지의 크롤링 및 인덱싱 여부를 제어.

## <meta name=”robots”> 태그란?

### **`<meta name="robots">`** **태그란?**


`<meta name="robots">` 태그는 검색 엔진 크롤러(robots)가 특정 웹 페이지를 **크롤링하거나 인덱싱하는 방식**을 제어하는 메타 태그입니다.


이 태그는 웹사이트의 개별 페이지에 대해 **검색 엔진이 수행해야 할 작업**을 지정하는 데 사용됩니다.


---


### **주요 속성값과 의미**


`robots` 태그는 **하나 이상의 속성값**을 설정하여 검색 엔진의 동작을 지정합니다. 주요 값은 다음과 같습니다:


| 속성값                | 설명                                          |
| ------------------ | ------------------------------------------- |
| **`index`**        | 페이지를 **검색 결과에 포함**(인덱싱)합니다. (기본값: 포함)       |
| **`noindex`**      | 페이지를 검색 결과에 **포함하지 않음**.                    |
| **`follow`**       | 페이지 내 링크를 따라 크롤링합니다. (기본값: 링크 따라감)          |
| **`nofollow`**     | 페이지 내 링크를 **따라가지 않음**.                      |
| **`noarchive`**    | 페이지의 **캐시된 버전을 저장하지 않음**.                   |
| **`nosnippet`**    | 검색 결과에서 **페이지 요약(snippet)**을 표시하지 않음.       |
| **`noimageindex`** | 페이지에 있는 이미지를 검색 엔진의 **이미지 검색 결과에 포함하지 않음**. |
| **`nocache`**      | 페이지를 **캐싱하지 않음**. (일부 검색 엔진에서만 지원)          |


---


### **예제**


### **1. 기본 사용**


페이지를 크롤링하고 인덱싱하도록 허용(기본값):


```html
<meta name="robots" content="index, follow">
```


### **2. 검색 결과에 포함하지 않음**


검색 엔진이 페이지를 크롤링하되, 검색 결과에는 포함하지 않도록 설정:


```html
<meta name="robots" content="noindex, follow">
```


### **3. 링크를 따라가지 않음**


페이지는 인덱싱되지만, 내부 링크를 따라 크롤링하지 않도록 설정:


```html
<meta name="robots" content="index, nofollow">
```


### **4. 검색 결과에 표시하지 않으며, 캐싱도 금지**


```html
<meta name="robots" content="noindex, noarchive">
```


---


### **적용 사례**

1. **공개하지 않아야 하는 페이지**
    - 로그인 페이지, 관리자 대시보드 등과 같이 검색 엔진에 노출될 필요가 없는 페이지:

        ```html
        <meta name="robots" content="noindex, nofollow">
        ```

2. **유사 콘텐츠 중복 방지**
    - 동일한 콘텐츠를 포함한 여러 페이지가 있을 때, 검색 엔진이 중복 콘텐츠를 인덱싱하지 않도록 설정:

        ```html
        <meta name="robots" content="noindex, follow">
        ```

3. **링크 크롤링 제한**
    - 외부 링크나 특정 내부 링크를 검색 엔진이 크롤링하지 않도록 제한:

        ```html
        <meta name="robots" content="index, nofollow">
        ```

4. **미완성 페이지 비노출**
    - 아직 공개 준비가 안 된 페이지를 검색 엔진에 노출하지 않도록 설정:

        ```html
        <meta name="robots" content="noindex, nofollow">
        ```


---


### **유의 사항**

1. **`robots.txt`****와의 차이**
    - `<meta name="robots">`는 **개별 페이지**에 대해 설정하며, 페이지가 로드된 후 동작합니다.
    - 반면, `robots.txt`는 **사이트 전체**의 크롤링 규칙을 정의하고 크롤러가 서버에 접근하기 전에 적용됩니다.
2. **적용 대상**
    - `<meta name="robots">` 태그는 Google, Bing 등 대부분의 검색 엔진에서 지원되지만, 검색 엔진마다 동작 방식이 약간 다를 수 있습니다.
3. **브라우저와 사용자의 영향**
    - `<meta name="robots">`는 브라우저나 사용자에게는 영향을 미치지 않으며, 오직 검색 엔진 크롤러에만 영향을 줍니다.
4. **올바른 사용**
    - 잘못된 설정(예: `noindex`가 필요한 페이지에 `index` 설정)은 SEO에 부정적인 영향을 미칠 수 있으니 주의해야 합니다.

---


### **정리**


`<meta name="robots">` 태그는 검색 엔진 크롤러가 특정 페이지를 **인덱싱하거나 크롤링하는 방식을 제어**하는 중요한 도구입니다.


이를 통해 민감한 페이지를 검색 결과에서 숨기거나, 검색 엔진이 중복 콘텐츠를 처리하는 방식을 조정할 수 있습니다.


SEO 전략에서 각 페이지의 의도와 목적에 맞게 적절히 설정하는 것이 중요합니다.


        예:


        ```html
        <title>React로 제작된 SEO 최적화 웹사이트</title>
        <meta name="description" content="SEO 최적화와 React를 활용한 빠르고 효율적인 웹사이트 개발 방법을 배웁니다.">
        <meta name="robots" content="index, follow">
        ```

    - _**이미지 태그 최적화**_
        - **대체 텍스트(alt)**: 이미지를 설명하는 문구를 작성하여 이미지가 로드되지 않거나, 화면 읽기 도구를 사용할 때 유용하게 만듭니다.
        - 이미지 파일 이름: `seo-friendly-image.jpg`처럼 키워드를 포함.

        예:


        ```html
        <img src="/images/example.jpg" alt="React SEO 최적화 예제 이미지">
        ```

2. <u>**성능 최적화**</u>
    - _**CSS/JS 파일 최적화**_
        - CSS와 JavaScript를 **번들링** 및 **압축**(Minify)하여 파일 크기를 줄입니다.
        - 불필요한 코드를 제거하고 필요한 코드만 로드하는 **코드 스플리팅** 사용.
    - _**Lazy Loading**_
        - 이미지나 비동기 데이터는 사용자가 필요할 때 로드되도록 설정.
        - React의 `React.lazy` 또는 HTML의 `loading="lazy"` 속성 활용.

        ```html
        <img src="/images/large-image.jpg" loading="lazy" alt="SEO 최적화 이미지">
        ```

    - _**캐싱 및 CDN 사용**_
        - 정적 파일을 브라우저 캐시에 저장하여 재요청을 줄입니다.
        - CDN(Content Delivery Network)을 사용하여 사용자와 가까운 위치에서 리소스를 제공합니다.
    - _**Core Web Vitals 개선**_
        - **Largest Contentful Paint(LCP)**: 주요 콘텐츠 로드 시간 최적화.
            - 텍스트 또는 이미지 최적화, 폰트 최적화 ⇒ Next에서 제공되고 있음
        - **First Input Delay(FID)**: 초기 입력 응답성 개선.
            - 클라이언트 요청 시 응답하기까지의 걸리는 시간
        - **Cumulative Layout Shift(CLS)**: 레이아웃 안정성 확보.
            - 페이지 로드 중 발생하는 콘텐츠의 이동량
            - 대체적으로는 CSS 애니메이션 최소화 정도

# 세션 기반 인증과 토큰 기반 인증 비교


## 1. 세션 기반 인증

- 사용자가 로그인하면 서버는 사용자 정보를 기반으로 세션을 생성하고, 해당 세션의 ID(쿠키에 저장)를 클라이언트(브라우저)가 요청할때마다 서버로 전달
- 서버가 세션 데이터를 직접 관리하여, 보안 및 데이터 무결성 보장에 유리
- 단, 확장성에 한계가 있음
- 서버 간 세션 공유를 위해 별도의 설정이 필요함

## 2. 토큰 기반 인증

- 사용자가 로그인하면 서버는 사용자 정보를 기반으로 토큰(JWT 등)을 생성해 클라이언트에 전달
- 이는 세션 기반과는 다르게 해당 토클을 로컬 스토리지 또는 세션 스토리지에 저장한뒤, 요청 시 헤더에 포함시켜 서버로 전달함
- 확장성이 높으며, 토근 자체적으로 인증 정보를 포함하므로, 인증 상태를 서버에 요청하지 않고, 검증 할 수 있다.
- 토큰 크기가 세션 ID보다 크기 때문에 대역폭 부담이 늘어날 수 있다.

### **왜 Axios는 JSON으로 파싱하지 않아도 결과가 출력되는가?**

1. axios는 요청을 보낼 때 Content-Type 헤더를 자동으로 설정하고, 서버의 응답이 JSON 형식일 경우, 응답 데이터를 자동으로 JSON으로 파싱합니다. 따라서 response.data를 통해 이미 파싱된 JSON 데이터를 사용할 수 있습니다.
2. axios는 내부적으로 ***JSON.parse()**를 사용하여 응답 데이터를 자동으로 파싱합니다. 사용자가 직접 파싱을 수행할 필요가 없으므로, 코드가 더 간결하고 사용하기 쉽습니다.
3. axios는 JSON뿐만 아니라 다양한 응답 형식을 지원하며, 요청에 대한 응답의 형식에 따라 적절히 처리합니다. 예를 들어, 이미지나 텍스트와 같은 다른 형식의 응답에 대해서도 적절히 처리할 수 있습니다.

***JSON.parse()** : JSON 문자열을 JavaScript 객체로 변환하는 메서드


### axios의 요청 객체 속성의 종류

1. `data`: 서버에서 반환된 응답 데이터. 응답 데이터를 자동으로 파싱되어 이 속성에 저장됨.
2. `status`: HTTP 응답 상태 코드
    1. 성공적인 요청 = 200
    2. 생성된 리소스 = 201
    3. 클라이언트 오류 = 400
    4. 서버 오류 = 500
3. `statusText`: 응답의 상태 텍스트
    1. 200 OK, 404 Not Found 등의 메시지를 포함한다.
4. `headers`: 서버에서 반환된 응답 헤더. 이 속성은 객체 형태로, 각 헤더의 이름과 값을 포함한다
5. `config`: 요청을 수행할 때 사용된 Axios의 설정 객체. 요청의 **메소드, URL, 헤더, 타임아웃** 등의 설정이 포함되어 있음
6. `request`: 요청을 수행한 XMLHttpRequest 객체. 이 속성은 원시 요청 객체에 대한 정보를 제공한다.

    ```javascript
    import { useState, useEffect } from 'react';
    
    function useWindowWidth(transformFn) {
      const [width, setWidth] = useState(() => transformFn(window.innerWidth));
    
      useEffect(() => {
        const handleResize = () => {
          setWidth(transformFn(window.innerWidth));
        };
    
        window.addEventListener('resize', handleResize);
    
        return () => {
          window.removeEventListener('resize', handleResize);
        };
      }, [transformFn]);
    
      return width;
    }
    
    export default useWindowWidth;
    ```

## css in js와 css in css의 차이를 말하고, 각 장점과 단점을 한개이상 말해주세요.

# **1. CSS-in-CSS**


CSS-in-CSS는 전통적인 방식으로, CSS 파일을 작성하고 HTML 요소에 클래스를 추가하여 스타일을 적용합니다. 스타일은 .css 파일에 저장되며, 이를 HTML 또는 JavaScript 파일에서 링크하여 사용합니다.


## **장점:**

- HTML과 CSS가 분리되어 있어 가독성이 좋습니다.
- 다양한 스타일시트 기능을 활용할 수 있으며, 기존의 CSS 생태계와 잘 맞습니다.
- 성능 측면에서 스타일을 한 번 로드한 후 재사용할 수 있습니다.

# **단점:**

- CSS가 글로벌 스코프를 사용하기 때문에 클래스 이름 충돌의 위험이 있습니다.
- 스타일을 동적으로 변경하거나 조건부로 적용하는 데 제약이 있을 수 있습니다.

# **2. CSS-in-JS**


CSS-in-JS는 JavaScript 파일 내에서 스타일을 작성하고, 컴포넌트별로 스타일을 직접 적용하는 방식입니다. React와 같은 프레임워크에서 많이 사용되며, styled-components, emotion 같은 라이브러리가 대표적입니다.


## **장점:**

- **Scoped Styles:** 스타일이 특정 컴포넌트에 국한되므로 클래스 이름 충돌을 방지할 수 있습니다.
- **동적 스타일링:** JavaScript 변수를 활용하여 조건부로 스타일을 적용하거나 테마에 맞춰 동적으로 스타일을 변경할 수 있습니다.
- **모듈성:** 컴포넌트와 스타일이 한 곳에 있으므로 관리하기가 더 쉽습니다.

## **단점:**

- 성능 측면에서 전통적인 CSS보다 느릴 수 있으며, 런타임에 스타일이 적용되므로 초기 로딩 속도가 느려질 수 있습니다.
- 스타일을 JavaScript에서 다루기 때문에 CSS의 전체 기능을 쉽게 활용하기 어려울 수 있습니다.

# **요약**


•	**CSS-in-CSS**는 전통적인 방식으로 스타일과 로직을 분리하고, 전역 스코프에서 스타일을 관리합니다.


•	**CSS-in-JS**는 스타일을 컴포넌트 안에 넣어 로직과 함께 관리할 수 있고, 동적 스타일링이 용이합니다.


각 방식은 프로젝트의 규모나 요구사항에 따라 선택됩니다. CSS-in-JS는 주로 React와 같은 프레임워크에서 컴포넌트 기반 개발을 할 때 많이 사용되고, CSS-in-CSS는 전통적인 방식이나 간단한 웹사이트에 적합합니다.


## 객체의 키를 반환하는 함수

### 문제


해당 `getProperty`함수는 특정 객체의 키 값을 반환하는 함수 입니다.


하지만 아래 코드는 실행 시 타입 오류가 발생합니다.


타입 오류가 나지 않도록 수정하고, 왜 아래의 함수가 타입 에러를 유발하는지 설명하세요


```typescript
type User = {
  id: number;
  name: string;
  age: number;
};

function getProperty(obj: User, key: string): any {
  return obj[key];
}

const user: User = { id: 1, name: "홍길동", age: 30 };

const name = getProperty(user, "name");
 
// 에러 발생
```

<details>
<summary>정답과 해설</summary>

해당 `getProperty`함수를 보면, `return`문에서 `obj` 객체의 `key`를 통해 값을 반환하고 있습니다.


그러나 `obj`의 타입은 `id`, `name`, `age`만 포함하고 있습니다.


함수에서 `key`의 타입을 `string`으로 지정해 주면, `"id"`, `"name"`, `"age"`뿐만 아니라 `"email"`, `"address"`와 같은 다른 문자열도 전달될 수 있습니다.


이 경우 TypeScript는 <u>**타입의 안전성**</u>을 위해, 존재하지 않는 키를 사용하려고 한다고 판단하여 **타입 오류**를 발생시킵니다.


이 문제를 해결하기 위해서는 **`keyof`** **연산자**를 사용하면 됩니다.


`keyof` 연산자는 **객체의 키 이름을 타입으로 추출**하여, **유니온 타입**(예: `"id" | "name" | "age"`)으로 만듭니다. 이를 통해, 함수 인자에 올바르지 않은 키 이름이 전달될 경우 컴파일 단계에서 오류를 발생시킵니다.


### 수정된 코드:


```typescript
type User = {
  id: number;
  name: string;
  age: number;
};

function getProperty(obj: User, key: keyof User): any {
  return obj[key];
}

const user: User = { id: 1, name: "홍길동", age: 30 };
const name = getProperty(user, "name");
```

## 동적 키 타입을 활용한 값 추출 함수

# 문제


### 요구사항

- `input data`는 객체 배열 형태 입니다.
- 특정 키를 기준으로 **해당 키의 값만** 추출 합니다.
- 동적 키 타입(keyof T)을 활용하여, 매개변수와 반환 타입을 정의하세요.
- 제네릭을 사용하는 이유에 대해서도 설명 해 주세요.

### 코드


```typescript
const users = [
	{id: 1, name: "홍길동", age: 30},
	{id: 2, name: "김철수", age: 25},
	{id: 3, name: "이영희", age: 23},
];

// 특정 키의 값을 추출하는 함수
function extract(data, key){
	return data.map((item) => item[key]);
}

// 결과 출력
console.log(extract(users, "name")); // ["홍길동", "김철수", "이영희"]
console.log(extract(users, "age")); // [30, 25, 23]
```

<details>
<summary>해설 및 풀이</summary>

요구사항에도 말했듯이 `input data`는 객체 배열 형태기이기때문에, `제네릭 타입 T`를 사용합니다.


그리고 동적인 키를 처리하기 위해서 `key`는 `keyof T` 타입으로 정의합니다.


반환값은 `T[keyof T][]` 이며, `key`에 해당하는 값의 배열을 뜻합니다.


```typescript
function extract<T>(data: T[], key: keyof T): T[keyof T][]{
	return data.map((item) => item[key]);
}
```

1. `extract<T>`
    1. `extract<T>`는 제네릭 함수를 뜻합니다. 여기서 `T`는 **타입을 나타내는 변수**라고 생각하면 됩니다.
    2. 사용자가 제공하는 데이터의 타입을 기준으로 함수 내부에서 자동으로 타입을 추론할수 있도록 돕는 역할을 합니다.
    3. 위와 같이 `data`에 들어오는 객체 배열이 `users` 라면, `T`는 아래와 같이 타입이 추론됩니다.

    ```typescript
    {id: number, name: string, age: number}
    ```

2. 제네릭을 사용하는 이유?
    1. **재사용성과 유연성**
        1. 해당 extract 함수는 users라는 데이터 외에도 다른 데이터 구조에서도 사용이 되어야하기 때문입니다.
        2. 제네릭으로 타입을 지정할 경우 타입을 유연하게 추론할 수 있음으로 코드의 재사용성이 높아집니다.
    2. **타입 추론의 편리성**
        1. 재네릭을 사용하면 입력한 값에 따라서 타입을 자동으로 추론 할 수 있습니다.
        2. 위 1번과 같이 `extract(users, "name")` 를 호출하면 컴파일러가 자동으로 `T = {id: number, name: string, age: number}` 로 추론 합니다.
    3. **명확한 타입 정의**
        1. 제네릭은 입력 데이터 구조를 명확히 반영한 타입 정의를 제공합니다.
            1. 예를 들어 `extract(users, "address")` 와 같이, 잘못된 키를 사용하려 한다면, 컴파일 에러가 발생하게 됩니다.

# Record


`Record`은 `TypeScript`에서 제공하는 유틸리티 타입 중 하나로, 객체 타입을 더 간단하고 명확하게 정의할 수 있도록 도와줍니다.


## 1. 기본적인 형태


`Record<K, T>`은 `키(K)`와 `값(T)`의 타입을 명시적으로 지정하여, 객체 타입을 정의합니다.

- `K` : 키의 타입 (보통 string, number, 또는 유니언 타입이 됩니다.)
- `T` : 값의 타입

### 예제


```typescript
type Example = Record<string, number>;
// Example은 모든 키가 string이고, 값은 number인 객체
const obj: Example = {
  a: 1,
  b: 2,
  c: 3,
};
```


## 2. `Record`를 사용하는 이유


기본적으로 객체 타입을 아래처럼 작성 할 순 있습니다.


```typescript
type CustomObject = {
  [key: string]: number;
};

const obj: CustomObject = {
  a: 1,
  b: 2,
  c: 3,
};
```


하지만 이렇게 작성하면, **값의 타입만 지정할 수 있고 키의 타입은 제한되지 않습니다**.


즉, 문자열 외에 다른 타입의 키를 사용해도 오류가 발생하지 않습니다.


`Record`를 사용하면 **키와 값 모두의 타입을 명확히 정의**할 수 있어 더욱 안전하고 간결하게 코드를 작성할 수 있습니다.


## 3. 실용적인 사용 예제

- 고정된 키의 객체를 정의할 때

```typescript
type UserRoles = Record<"admin" | "editor" | "viewer", boolean>;

const roles: UserRoles = {
  admin: true,
  editor: false,
  viewer: true,
};
```

- 동적 키의 객체를 정의할 때

```typescript
type GroupedData<T> = Record<string, T[]>;

const grouped: GroupedData<number> = {
  group1: [1, 2, 3],
  group2: [4, 5, 6],
};
```


## 4. `Record`의 한계와 주의점

- `Record`는 모든 키를 동일한 타입으로 강제
    - 만약 각 키의 값 타입이 서로 다르다면, `Record`는 적합하지 않을 수 있습니다.
    - 이 경우 유니언 타입이나 고정된 객체 타입을 사용하는 것이 더 적합합니다.

    ```typescript
    type MixedValues = {
      id: number;
      name: string;
    };
    
    // 이 경우 Record 사용 불가, 명시적으로 작성해야 함
    const obj: MixedValues = {
      id: 1,
      name: "Alice",
    };
    ```

- `Record`는 모든 키를 강제
    - 특정 키를 필수로 지정하거나 선택적으로 처리하려면 `Partial`, `Pick` 같은 유틸리티 타입과 조합해야 합니다.

    ```typescript
    type PartialRecord<K extends keyof any, T> = Partial<Record<K, T>>;
    
    const obj: PartialRecord<"id" | "name", string> = {
      id: "123", // name은 선택적으로 포함 가능
    };
    ```


## Partial & Pick

TypeScript는 객체 타입을 다룰 때 더 효율적이고 안전한 코드를 작성할 수 있도록 **유틸리티 타입**을 제공합니다.


그중 `Partial`과 `Pick` 특정 상황에서 객체 타입을 변형할 수 있는 중요한 도구입니다.


---


# 1. **`Partial`** **유틸리티 타입**


### **역할**


`Partial<T>`는 **객체 타입** **`T`****의 모든 속성을 선택적(****`optional`****)으로 만드는 타입**입니다.


### **문법**


```typescript
type Partial<T> = {
  [P in keyof T]?: T[P];
};
```


즉, 객체 `T`의 **모든 속성 키를 순회하며(****`keyof T`****)** **`?`****를 추가**해 선택적으로 변환합니다.


### **사용 예제**


```typescript
type User = {
  id: number;
  name: string;
  email: string;
};

type PartialUser = Partial<User>;

const user1: PartialUser = {
  id: 1,
}; // name, email은 선택적
```


### **주요 특징**

- 원래 필수 속성이었던 `id`, `name`, `email`이 이제 선택적 속성으로 바뀌었습니다.
- 즉, 객체의 일부분만 사용할 수 있도록 허용합니다.

---


# 2. **`Pick`** **유틸리티 타입**


### **역할**


`Pick<T, K>`는 **객체 타입** **`T`****에서 특정 속성(****`K`****)만 선택하여 새로운 타입을 만드는 것**입니다.


### **문법**


```typescript
type Pick<T, K extends keyof T> = {
  [P in K]: T[P];
};
```


즉, 타입 `T`의 키들(`keyof T`) 중 **일부(****`K`****)를 선택하여 새 객체 타입**을 만듭니다.


### **사용 예제**


```typescript
type User = {
  id: number;
  name: string;
  email: string;
};

type PickedUser = Pick<User, "id" | "name">;

const user2: PickedUser = {
  id: 2,
  name: "Alice",
}; // email은 포함되지 않음
```


### **주요 특징**

- `User` 타입 중에서 `"id"`와 `"name"`만 포함한 새로운 타입을 만듭니다.
- 특정 속성만 선택적으로 사용하고 싶을 때 매우 유용합니다.

---


# 3. **`Partial`****과** **`Pick`****의 차이점**


| **특징**    | **Partial**             | **Pick**               |
| --------- | ----------------------- | ---------------------- |
| **주요 기능** | 모든 속성을 선택적으로 바꿈         | 특정 속성만 선택              |
| **결과 타입** | 원래 속성이 모두 선택적이 된 객체 타입  | 원래 타입의 일부 속성만 가진 객체 타입 |
| **사용 목적** | 일부분만 설정하거나, 선택적 데이터를 처리 | 필요한 속성만 추출하여 사용        |
| **사용 문법** | `Partial<T>`            | `Pick<T, K>`           |


---


# 4. **실용적인 사용 사례**


### **1)** **`Partial`****: 업데이트용 데이터**


`Partial`은 **업데이트 함수**를 작성할 때 유용합니다.


모든 속성을 업데이트하지 않아도 되기 때문에 선택적으로 데이터 제공이 가능합니다.


```typescript
type User = {
  id: number;
  name: string;
  email: string;
};

function updateUser(id: number, updates: Partial<User>): void {
  console.log(`Updating user ${id} with`, updates);
}

updateUser(1, { name: "Bob" }); // 일부 속성만 업데이트
```

- `updates` 객체는 `User` 타입의 모든 속성을 선택적으로 가질 수 있습니다.

---


### **2)** **`Pick`****: 제한된 속성만 사용**


`Pick`은 **API 응답 타입**에서 필요한 필드만 추출할 때 유용합니다.


```typescript
type User = {
  id: number;
  name: string;
  email: string;
  password: string;
};

type PublicUser = Pick<User, "id" | "name" | "email">;

const user: PublicUser = {
  id: 1,
  name: "Alice",
  email: "alice@example.com",
}; // password는 제외됨
```

- `PublicUser`는 `User` 타입에서 `id`, `name`, `email`만 포함한 새로운 타입입니다.
- 민감한 정보를 제외하고 일부 정보만 공개할 때 유용합니다.

---


### 5. **`Partial`****과** **`Pick`****의 조합**


`Partial`과 `Pick`을 함께 사용하면 더 강력한 타입 조작이 가능합니다.


예를 들어, 특정 속성을 선택하면서 그 속성을 선택적으로 만드는 경우입니다.


### **예제**


```typescript
type User = {
  id: number;
  name: string;
  email: string;
  password: string;
};

type OptionalPublicUser = Partial<Pick<User, "name" | "email">>;

const user: OptionalPublicUser = {
  name: "Alice", // email은 없어도 됨
};
```

- `Pick<User, "name" | "email">`: `User` 타입에서 `name`과 `email` 속성만 선택.
- `Partial<>`: 선택된 속성들을 선택적 속성으로 변환.

---


# 6. **결론**


### **`Partial`**

- 객체 타입의 모든 속성을 선택적으로 변환.
- 주로 **업데이트, 설정 함수**에서 활용.

### **`Pick`**

- 객체 타입에서 특정 속성만 추출하여 새로운 타입 생성.
- **API 응답, 제한된 데이터 처리**에 활용.

### **조합 활용**

- `Partial`과 `Pick`을 조합하면 특정 속성만 선택적으로 처리하는 강력한 타입 변환이 가능.

# 정답 및 해설


### 완성 코드


```typescript
import { GetStaticProps } from 'next';

type Props = {
  title: string;
  body: string;
};

export default function BlogPost({ title, body }: Props) {
  return (
    <div>
      <h1>{title}</h1>
      <p>{body}</p>
    </div>
  );
}

export const getStaticProps: GetStaticProps = async () => {
  const response = await fetch('http://placeholder.com/posts/1');
  const data = await response.json();

  return {
    props: {
      title: data.title,
      body: data.body,
    },
  };
};
```


### SSG 동작 방식

- **빌드 타임에만** 실행됩니다.
- `getStaticProps`는 프로젝트를 빌드할 때 한 번 호출되어 데이터를 가져옵니다.
- 가져온 데이터를 이용해 **정적인 HTML 파일**을 생성하며, 이 HTML 파일은 배포 후 변경되지 않습니다(재생성 제외).

## 궁굼해요


> 🤔 `getStaticProps`의 타입이 왜 `GetStaticprops`인지?


`getStaticProps`는 Next.js에서 미리 정의한 함수로, 타입은 **`GetStaticProps`**로 지정됩니다. 이 타입은 기본적으로 Next.js의 타입 정의 파일에서 제공되며, **반환값과 매개변수**를 정확히 지정해줍니다.

- 타입 정의

```typescript
type GetStaticProps<
  Props extends { [key: string]: any } = { [key: string]: any },
  Params extends ParsedUrlQuery = ParsedUrlQuery
> = (
  context: GetStaticPropsContext<Params>
) => Promise<GetStaticPropsResult<Props>> | GetStaticPropsResult<Props>;
```

- 매개 변수 타입

```typescript
type GetStaticPropsContext<Params extends ParsedUrlQuery = ParsedUrlQuery> = {
  params?: Params; // 동적 라우팅에서 사용되는 파라미터
  preview?: boolean; // 미리보기 모드 여부
  previewData?: any; // 미리보기 데이터
  locale?: string; // i18n 로케일 정보
  locales?: string[]; // 사용 가능한 로케일
  defaultLocale?: string; // 기본 로케일
};
```

- 반환 타입
    - `getStaticProps` 함수의 반환값은 `GetStaticPropsResult`로, `props` 필드를 포함합니다. 이 필드가 컴포넌트로 전달됩니다.

```typescript
type GetStaticPropsResult<Props> = 
  | { props: Props; revalidate?: number | boolean } // 정적 페이지 생성
  | { redirect: Redirect } // 리다이렉트
  | { notFound: true }; // 404 반환
```


> 🤔 `getStaticProps` 함수는 왜 호출하지 않아도 동작하는가?


`getStaticProps`는 Next.js 내부에서 자동으로 호출됩니다. **Next.js는 빌드 타임에** **`getStaticProps`****를 찾아 실행**하고, 그 결과를 사용해 정적 HTML 페이지를 생성합니다.


### **작동 방식**

1. **컴파일 시점:**
    - Next.js는 페이지 파일에서 `export const getStaticProps`를 찾습니다.
    - `getStaticProps` 함수가 있으면 해당 함수를 호출하여 반환된 데이터를 가져옵니다.
2. **정적 페이지 생성:**
    - `getStaticProps`의 반환값에서 `props`를 추출합니다.
    - Next.js는 이 `props`를 이용해 페이지 컴포넌트를 렌더링하고 HTML 파일을 생성합니다.
3. **사용자 요청 시:**
    - 사용자가 페이지에 접근하면, 빌드된 HTML 파일과 JSON 형태의 `props`가 클라이언트에 전달됩니다.
    - React는 전달된 `props`를 사용해 컴포넌트를 렌더링합니다.

> 🤔 `props`는 어떻게 컴포넌트에 전달되는가?


Next.js는 빌드 타임에 `getStaticProps`가 반환한 데이터를 페이지 컴포넌트에 자동으로 전달합니다.


다음 과정을 통해 `props`가 전달됩니다

- **함수 작성:**

```typescript
export const getStaticProps = async () => {
  return {
    props: {
      title: 'Hello, World!',
    },
  };
};
```

- **빌드 타임 처리:**
    - Next.js는 `getStaticProps`를 실행하여 반환된 `props` 데이터를 저장합니다.
    - 빌드 시 생성된 HTML 파일에 `props` 데이터가 포함됩니다.
    - `props` 데이터는 JSON 파일로도 생성되어 클라이언트로 전달될 준비를 합니다.
- **컴포넌트에 전달:**

Next.js는 내부적으로 `props`를 페이지 컴포넌트의 첫 번째 매개변수로 전달합니다.


```typescript
export default function PageComponent({ title }: { title: string }) {
  return <h1>{title}</h1>;
}
```

## Next.js의 코드스플리팅과 동적 임포트

# 문제


Next.js의 코드 스플리팅(Code Splitting)과 동적 임포트(Dynamic Import)에 대해서 설명하고, 각 장점에 대해서 설명 하세요.


## 정답

# 코드 스플리팅(Code Splitting) 이란


애플리케이션의 번들 크기를 줄이기 위해 코드를 여러 개의 작은 청크(chunk)로 분할하는 기술.


이를 통해 초기 로딩 시 불필요한 코드를 다운로드하지 않고도 애플리케이션을 사용할 수 있다.


해당 기술을 사용하는 이유는 필요한 정보만 로드해서 초기 렌더 속도를 개선하고 전체적인 애플리케이션의 성능을 향상 시켜주기때문에 사용한다.(최적화)


# 동적 임포트(Dynamic Import) 란?


코드에서 특정 모듈을 필요할 때까지 로드하지 않고, 런타임에 조건적으로 로드하는 기술

- next/dynamic 모듈을 통해 동적 임포트를 쉽게 사용할 수있게 된다

```javascript
import dynamic from 'next/dynamic';

const DynamicComponent = dynamic(() => import('../components/HeavyComponent'));

function Page() {
  return (
    <div>
      <h1>My Page</h1>
      <DynamicComponent />
    </div>
  );
}

export default Page;
```


## 장점


### 초기 번들 크기 감소

- **불필요한 코드 로딩 방지**: 초기 로딩 시 사용자가 당장 필요로 하지 않는 코드(예: 대형 라이브러리나 컴포넌트)를 제외하여 번들 크기를 줄입니다.
- **네트워크 최적화**: 작은 번들을 로드하므로 네트워크 사용량이 감소하고, 저속 연결에서도 더 나은 성능을 제공합니다.

### 페이지 로딩 속도 향상

- **퍼포먼스 개선**: 초기 로딩 시간이 단축되어 사용자에게 빠른 응답을 제공합니다.
- **지연 로딩을 통한 사용자 경험 향상**: 사용자 상호 작용에 따라 필요한 컴포넌트를 로드하므로 앱의 반응성이 좋아집니다.

### 코드의 모듈화 및 유지 보수성 향상

- **모듈별 로직 분리**: 동적 임포트를 통해 코드 구조를 개선하고, 모듈별로 독립적인 개발이 가능합니다.
- **가독성 향상**: 필요한 부분에서만 모듈을 로드하므로 코드가 간결해집니다.

### SSR과의 호환성 제어:

- **SSR 비활성화 옵션**: 동적 임포트 시 `ssr: false` 옵션을 사용하여 서버 사이드 렌더링 시 특정 모듈이 로드되지 않도록 할 수 있습니다.
- **클라이언트 전용 코드 처리**: 브라우저 환경에서만 동작하는 코드(예: 로컬 스토리지 접근 등)를 안전하게 사용할 수 있습니다.

```javascript
const DynamicComponentWithNoSSR = dynamic(
  () => import('../components/NoSSRComponent'),
  { ssr: false }
);
```


### 조건부 로딩 및 성능 최적화

- **사용자 행동에 따른 로딩**: 특정 이벤트(예: 버튼 클릭)에 따라 모듈을 로드할 수 있습니다.

```javascript
import { useState } from 'react';

function Page() {
  const [Component, setComponent] = useState(null);

  const loadComponent = async () => {
    const mod = await import('../components/HeavyComponent');
    setComponent(() => mod.default);
  };

  return (
    <div>
      <button onClick={loadComponent}>Load Component</button>
      {Component && <Component />}
    </div>
  );
}
```

- **뷰포트 진입 시 로딩**: Intersection Observer API를 사용하여 요소가 뷰포트에 진입할 때 로드하도록 할 수 있습니다

### 브라우저 지원 및 번들 크기 감소

- **폴리필 최적화**: 동적 임포트를 통해 필요할 때만 폴리필을 로드하여 번들 크기를 줄일 수 있습니다.

## **OAuth(Open Authorization)**


OAuth(Open Authorization)는 우리말로 '개방형 접근 권한 표준'이라고 하며, 민감한 정보를 제공하지 않으면서 제3자가 보호된 리소스에 접근할 수 있도록 설계된 **권한 위임 프로토콜**입니다.


OAuth는 사용자가 신뢰하는 서비스(클라이언트)가 다른 플랫폼(리소스 서버)에 저장된 데이터를 안전하게 접근할 수 있도록 **권한을 위임**받는 방식으로 동작합니다. 이를 통해 사용자는 자신의 자격 증명(아이디, 비밀번호)을 직접 제공하지 않아도, OAuth를 지원하는 플랫폼에서 데이터 접근 권한을 안전하게 관리할 수 있습니다.


대표적인 예로, 사용자가 어떤 웹사이트에서 자신의 **구글 드라이브 파일**이나 **페이스북 친구 목록** 등에 접근할 수 있는 권한을 위임하는 경우를 들 수 있습니다.


OAuth의 주요 구성 요소는 다음과 같습니다:

- **리소스 소유자(Resource Owner)**: 보호된 리소스에 대한 소유권을 가진 사용자
- **클라이언트(Client)**: 리소스 소유자를 대신하여 보호된 리소스에 접근하려는 애플리케이션
- **권한 서버(Authorization Server)**: 리소스 소유자로부터 접근 권한을 확인하고, 액세스 토큰(Access Token)을 발급
- **리소스 서버(Resource Server)**: 보호된 데이터를 저장하고, 액세스 토큰을 이용해 요청을 인증하는 서버

OAuth의 장점은 사용자가 자신의 자격 증명을 노출하지 않고도, 신뢰하는 클라이언트 애플리케이션이 필요한 리소스에 접근할 수 있도록 안전한 환경을 제공한다는 점입니다.

## **OIDC(OpenID Connect)**


OIDC(OpenID Connect)는 OAuth의 확장으로, **인증(Authentication)** 기능이 포함된 프로토콜입니다. OIDC는 OAuth 2.0 프로토콜의 상위 계층에서 동작하며, 접근 권한(Authorization)을 넘겨받는 것뿐만 아니라 사용자의 신원까지 인증할 수 있습니다.


즉, OAuth가 **인가(Authorization)**만 제공하는 데 반해, OIDC는 **인증(Authentication)**까지 한 번에 처리할 수 있습니다. 이를 통해 사용자의 신원을 검증하고, 인증된 사용자 정보를 클라이언트 애플리케이션에 전달할 수 있습니다.


### **OIDC의 주요 특징**

1. **OAuth 2.0 기반**: OIDC는 OAuth 2.0을 기반으로 확장된 프로토콜로, OAuth의 기능을 그대로 사용하면서 추가적인 인증 기능을 제공합니다.
2. **ID 토큰 발급**: OIDC는 인증 과정에서 **ID 토큰**을 발급합니다. 이 토큰에는 사용자의 신원을 나타내는 정보(예: 사용자 ID, 이메일, 이름 등)가 포함됩니다.
3. **사용자 인증 편의성**: OIDC는 **소셜 로그인**에서 주로 활용되며, 구글, 카카오톡, 네이버, 페이스북 등의 계정을 이용하여 간편하게 인증을 수행할 수 있습니다.

### **OIDC의 구성 요소**

- **OIDC 프로바이더(Provider)**: 인증 서비스를 제공하는 플랫폼(예: 구글, 카카오톡, 네이버 등)
- **ID 토큰(ID Token)**: JWT(JSON Web Token) 형식으로 발급되며, 사용자 신원을 포함한 인증 정보를 클라이언트에 전달
- **클라이언트(Client)**: 사용자의 인증 정보를 제공받아 애플리케이션에 활용
- **엔드포인트(Endpoints)**: OIDC는 인증, 토큰 발급, 사용자 정보 요청 등 다양한 엔드포인트를 제공

## **OAuth와 OIDC의 차이**

- **OAuth**는 주로 **권한 위임**에 초점이 맞춰져 있으며, 사용자가 신뢰하는 서비스가 특정 리소스에 접근하도록 허용하는 방식입니다.
- **OIDC**는 여기에 더해 **사용자 인증**까지 지원하므로, 사용자의 신원을 확인할 수 있습니다.

---


### 인가와 인증

- 인가(Authorization)는 사용자가 특정 리소스나 기능에 <u>**접근할 수 있는 권한을 확인하고 부여**</u>하는 과정을 말합니다.
- 쉽게 말하면, "이 사람이 이 작업을 해도 되는지 확인하고 허락하는 것"입니다. 인가는 보통 인증(Authentication) 이후에 이루어집니다.
- 인증은 "이 사람이 누구인지 확인"하는 과정이라면, 인가는 "확인된 사람이 무엇을 할 수 있는지 결정"하는 과정입니다.

#  REST API

> REST API(Representational State Transfer API)는 웹 기반의 API 디자인 아키텍처 스타일로, 클라이언트와 서버 간의 통신을 간단하고 일관되게 만들기 위해 **HTTP 프로토콜**을 활용합니다. REST는 리소스를 다루는 방식과 해당 리소스에 대한 행위를 정의하는 여러 가지 원칙과 제약을 포함합니다.
- REST API는 HTTP 프로토콜을 기반으로 하여 리소스를 효과적으로 관리하고 클라이언트와 서버 간의 통신을 간단하게 만들어주는 디자인 패턴이다.
- REST API는 리소스 지향적이며, 상태가 없는 요청을 통해 클라이언트와 서버 간의 분리된 아키텍처를 제공합니다. 이러한 특징 덕분에 REST API는 웹 개발에서 널리 사용되고 있습니다.

## **주요 개념**


### **리소스(Resource)**:

- REST API는 리소스를 기본 단위로 다룹니다. 리소스는 데이터(예: 사용자, 상품, 게시물 등)를 나타내며, 각 리소스는 고유한 URI(Uniform Resource Identifier)를 통해 식별됩니다.

    예: /users, /products/123와 같이 URI로 접근할 수 있습니다.


### **HTTP 메소드**:

- REST API는 CRUD(Create, Read, Update, Delete) 작업을 수행하기 위해 HTTP 메소드를 사용합니다:
    - **GET**: 리소스를 조회합니다.
    - **POST**: 새로운 리소스를 생성합니다.
    - **PUT**: 기존 리소스를 업데이트합니다.
    - **PATCH**: 리소스의 일부를 수정합니다.
    - **DELETE**: 리소스를 삭제합니다.

### **상태(stateless)**:

- REST API는 각 요청이 독립적이어야 하며, 서버가 클라이언트의 상태를 저장하지 않아야 합니다. 즉, 모든 요청은 필요한 정보를 포함해야 하며, 이전 요청의 상태에 의존하지 않아야 합니다.

### **표현(representation)**:

- 클라이언트는 리소스의 표현(예: JSON, XML 등)을 요청할 수 있으며, 서버는 이 표현을 통해 클라이언트에 데이터를 전달합니다.

### **캐시 가능(cachable)**:

- REST API의 응답은 캐시할 수 있어야 하며, 클라이언트는 이전에 받은 응답을 캐시하여 성능을 향상시킬 수 있습니다.

## **REST API의 작동 방식**

1. **클라이언트가 요청을 보냅니다**:
    - 클라이언트는 HTTP 메소드를 사용하여 특정 리소스의 URI로 요청을 보냅니다.
2. **서버가 요청을 처리합니다**:
    - 서버는 요청을 받고, 해당 리소스에 대한 작업을 수행한 후, 결과를 클라이언트에게 응답합니다.
3. **클라이언트가 응답을 처리합니다**:
    - 클라이언트는 서버의 응답을 받아, 필요한 정보를 사용자에게 표시하거나 추가 작업을 수행합니다.

# 이력서 작성하기


## 구성과 순서

- <u>**제목**</u>
    - 이름 or 한줄 자기소개
- <u>**프로필**</u>
    - 연락처, github 링크, blog 링크, 사진
- <u>**자기소개**</u>
    - 나에 대한 전반적인 소개와 업무적 강점을 어필하는 항목
        - 간략하게 내가 어떤 경력/경험을 가진 개발자인지 요약
        - 그동안 어떤 성과를 내왔는지
        - 업무적 강점/관심사/태도(일과 무관한 것 x)
        - 강점/관심사/태도에 대한 근거 (관련 경험이나 노력)
        - 경험 항목과 연결될 수 있는 내용이 들어가는 게 좋음
        - 설득력을 높이기 위해 가능한 숫자로 표현
        - 잘 안써진다면 일단 스킵하고, 이력서를 다 쓰고나서 반복되는 키워드 등을 찾은 후 작성
- <u>**경험(회사 프로젝트 경험)**</u>
    - 필요로 하는 역량을 갖췄음을 증명하는 항목(가장 중요)
        - 그간의 경력과 경험을 최신순으로 작성
        - 어필될 수 있는 핵심적인 경험 위주로 작성
        - 면접 질문 준비할 때 가장 많이 참고하는 항목
        - 회사, 부서/직함, 재직/담당 기간, 사용한 기술, 담당한 일과 성과와 수치 중심의 결과(예: 매출 성장률, 수상 경력, 언론 기사 등)
        - 역량을 보여줄 수 있는 정도로 구체적으로 작성

## 2.2 단락 평가

# 단락 평가(Short-circuit Evaluation)이란?


---


논리 연산식에서 첫번째 피연산자 값 만으로도 해당 연산의 결과를 확정 지을 수 있다면 그때는 두번째 피연산자의 값에는 접근하지 않는 JS의 특징을 말한다.


## 코드


```javascript
let varA = false;
let varB = true;

// AND 연산자
console.log(varA && varB);

// OR 연산자
console.log(varA || varB);
```


## 설명

- AND 연산자의 조건은 왼쪽의 피연산자가 true일때는, 오른쪽 피연산자를 반환한다.
- OR 연산자의 조건은 왼쪽 피연산자가 true일때는, 왼쪽 피연산자를 반환한다.
