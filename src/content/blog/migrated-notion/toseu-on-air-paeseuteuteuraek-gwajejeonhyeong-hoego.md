서류 전형을 거치지 않고 빠르게 지원해볼 수 있는 패스트 트랙에 지원했다.


과제 테스트는 7월 12일 오후 12시부터 오후 10시까지 진행되었다.


학원 수업이 있음에도 불구하고, 과제 전형에 집중하기 위해서 오후 12시가 되자마자 바로 과제를 다운받아 분석을 시작하였다.


우선 토스에서 사용하는 TDS를 과제전형에 맞게 커스텀마이징한 형태로 이커머스 형식의 주문>장바구니>결제 형식의 기능 구현 과제였다.


복잡한 상태관리와 컴포넌트 props 활용 등 다양한 기술 역량을 확인할 수 있는 적절한 과제였다.


다만, TDS 형식을 경험해보지못했던 나는 꽤나 당황했다.


해당 컴포넌트를 살펴볼 필요없이 명확한 props명이 주어졌지만, 이를 어떻게 활용할 것 인가에 대해서 다소 고민이 있었다.


그리고 더구나 하드코딩으로 되어있지만서도 제공된 컴포넌트가 `li` 태그로 구성이 되어있었는데,


시멘틱 마크업 구조상 ‘ul 하위에는 li 태그만 존재해야하며 li를 단독으로 사용할 수 없다’가 원칙이지만


제공된 컴포넌트 구조가 아래와 같았다.


```java
<div>
  <li>...</li>
</div>
```


물론 이것을 토스팀의 실수라고 생각하진 않았다.


단지, 해당 시멘틱 마크업 구조에 대한 기본적인 이해가 있는지 테스트를 하기 위함의 일종의 테스트 라고 생각했다.


그렇게 요구사항과 코드 파악을 위해 사용된 시간만 4시간이 흘러갈 때 쯔음이었다.


더이상의 코드 파악은 시간 낭비일 것이라는 생각이 들어서 REST API 부터 정리하기 시작했다.


제공된 아이템 리스트, 아이템 옵션, 주문 목록 GET/POST와 결과 데이터 가져오기를 별도의 service 폴더에 axios를 활용하여 예외처리를 진행 해주었다.


각 페이지에서 GET 요청을 하여 제공된 컴포넌트 props에 적절한 data를 제공해주고 그렇게 주문 리스트 페이지를 완료했다.


그 다음으로 난황을 겪었던 페이지가 주문 상세 부분이었다.


다양한 아이템 별로 옵션값이 다 달랐는데 보여지는 view가 컴포넌트로 구분이 되어있었다.


switch를 이용하여 해당 컴포넌트 type에 맞게 컴포넌트를 렌더링 하도록 해주었는데 문제는 api에서 문제였다.


기본 아이템에 대한 정보와 그에 따른 옵션값 정보가 별도의 api로 구성이 되어있었다.


이럴때는 두 api 모두 불러온 후 reduce를 활용해 타입에 맞는 아이템을 그룹으로 재 매핑 한 후에 렌더링 시켜주면 되는데


제공된 요구사항의 디자인과 텍스트 요구사항이 상이했다.


실제 프로젝트 업무를 진행할때 텍스트의 요구사항과 제공된 시안이 상이 할 경우 디자이너분께 확인 요청을 드리거나 물어봤을텐데 어디에도 물어볼 곳이 없으니 오로지 내 스스로 명확한 요구사항을 파악 후 작성해야하다보니 많은 고민을 했던 것 같다.


자세히 언급할수는 없어서 간단하게 언급하자면 셀렉드 드롭다운 옵션에서 디자인 상에서는 어떠한 값도 선택되지않은 ‘옵션을 선택해주세요’라는 placeholder가 있다. 하지만 텍스트 요구사항에서는 기본값이 선택된 상태를 요구하고있었다.


사실 옵션을 선택해주세요 라는 placeholder를 띄우고 서버에 post를 할때는 선택된값이 없어도 첫번째 index로 서버에 전달을 하면된다.


하지만..진짜 문제였던것은 제공된 셀렉트옵션 컴포넌트에 있었다.


토스팀 내부에서 착오가 있었는지는 잘 모르겠다.


제공된 컴포넌트는 무조건 default 값이 선택이 되게 되어있다. 그래서 사실상 placeholder가 필요가 없는 컴포넌트였다.


이 부분을 도대체 어떻게 맞추라는건지 고민을 할 때 쯔음 나에게 남은시간은 고작 3시간 남짓이었고, 장바구니 기능과 상태관리는 시작도 못한 상태였다.


이부분은 추후에 문의드리기로 하고 제공된 컴포넌트에 맞추어 진행하기로 하였다.


그리고 추가적인 옵션 최소값, 최대값을 구현하고나니 어느새 마감시간이 1시간도 남지않았다.


이대로는 보낼 수 없어 최대한 리팩토링하고 보내자 라는 생각으로 중복로직(금액에 대한 포멧팅 등)을 utils로 분리하고 view와 비즈니스로직을 분리하여 리팩토링을 끝낸 후 마무리를 하였다.


핵심인 상태관리부분을 미처 하지못해서 너무 아쉬웠다.


차라리 ai를 조금 활용하여 빠르게 작업할걸 그랬나 싶었다.

> 사실 이번 과제전형에서는 ai를 일체 사용하지않았다.  
> 내 스스로에 대한 테스트이기도 했으며, 온전히 내 실력을 보여주자라는 마음으로 임했으나 타임어택의 상황에서는 오히려 ai를 활용해 빠르게 초안작업을 하는것이 적절한 방식이었을지도 모른다.

그렇게 나의 아쉬운 과제전형이 끝났고 결과는 당연하게도 탈락했다.


현재까지 토스에 지원하고 2번이라는 기회가 있었음에도 불구하고 과제전형, 코딩테스트에서 탈락한 것을 보면 확실히 문제가 있는듯했다.


첫번째 문제는 ‘지금 당장 확인할 수 없는 사항에 많은 시간을 소모한다는것’


이것은 이번 타임어택 과제 전형을 진행하면서 느끼게 되었다. 내가 할 수 있는 것, 요구사항이 명확한 것, 내가 변경할 수 없는 것 이 부분을 빠르게 파악하는것이 중요함을 크게 느꼈다.


또한, 설계에 있어서도 어떤 패턴과 어떤 방식을 활용하면 중복없이 클린한 코드를 작성할 수 있을 것인가


이부분은 고민해서 달라질건없을 것같다. 최대한 다양한 경험을 통해서 획득할 수 있는 것이라고 생각한다. 최대한 많이 다양한 케이스를 직접 개발을 진행해봐야한다.


나는 여기서 한가지 의문점에 봉착하게 된다.


정말 구현만 할 줄 알면 되는건가…..? 자료구조, 알고리즘, 개발 이론 정말 몰라도 만들기만 하면 되는건가 싶다


나는 아직 개발을 하면서 커뮤니티 관련 도메인만 다뤄봤지 이커머스나 SaaS 형태의 도메인은 다루어본적이 없다.


신입에게 있어서 다양한 도메인을 경험해보기에는 무척 한정적이지만 개인 프로젝트라도 해당 부분을 메워갈 수 있다고 생각한다.


지금도 꾸준히 토이 프로젝트를 기획하고 디자인하고 개발하고있지만 실제 기업에서 요구하는 다양한 도메인 서비스를 구현한 토이프로젝트를 많이 경험해보는것이 추후 과제전형, 코딩테스트에도 유리하게 작용할 것임을 알게 되었다.


앞으로는 너무 이론에만 취중하지않고, 실무에서 직접 활용해 구현해보는것을 중점적으로 공부해야할 것 같다.
