# DOM 조작 함수는 **왜 순수 함수가 될 수 없는가?**

- DOM 조작 함수는 HTML 요소를 **수정**하거나 **새로운 요소를 추가**하는 등의 작업을 수행하는데, 이는 외부 상태(즉, 웹 페이지의 DOM)를 변경하기 때문에
- 순수 함수는 외부 상태를 변경하지 않지만, DOM 조작 함수는 특정 DOM 요소의 **텍스트, 스타일, 속성 등을 변경**하여 부작용을 발생시킨다.
- 외부 상태(DOM)를 조작하기 때문에, 동일한 입력을 주어도 DOM의 현재 상태에 따라 다른 결과가 나올 수 있다.

## DOM 조작 함수 예시


```javascript
// DOM 조작 함수
function changeElementText(element, newText) {
  element.textContent = newText;
}
```


위 코드로 미루어보았을때, element와 newText에 어떤 값을 받느냐에 따라 DOM에 그려지는 형태가 달라지기 때문에 순수 함수에 포함 되지 않는다.

# 추상화

- 추상화는 복잡한 세부 사항을 감추고 사용자가 더 높은 수준의 코드에만 집중할 수 있도록 하는 기법.

## 함수 추상화

- 함수를 호출한 사람은 함수가 **어떻게 동작하는지**를 몰라도 되고, __**어떤 입력이 필요한지와 어떤 출력이 나오는지**__만 알면 된다.

## 모듈화

- 복잡한 코드를 여러 모듈로 나누어, **각 모듈이 특정 기능을 담당**하게 하고, 이를 다른 모듈에서 __**재사용**__할 수 있게 한다.
- 각 모듈은 내부 구현을 감추고, 외부에 **필요한 함수나 객체만 노출하여 높은 수준의 추상화를 제공**합니다.

## 고차 함수

- 고차 함수는 특정 로직을 캡슐화하고, 나머지는 함수로 전달하여 추상화를 강화한다.

### 고차 함수 예시 코드


```javascript
function withLogging(fn) {
  return function(...args) {
    console.log(`함수 ${fn.name}이(가) 호출되었습니다.`);
    return fn(...args);
  }
}

function sum(a, b) {
  return a + b;
}

const loggedSum = withLogging(sum);
console.log(loggedSum(3, 4)); // 함수 sum이(가) 호출되었습니다. 7
```

```javascript
class Car {

myCar.startEngine(); // Tesla Model 3의 엔진이 시작됩니다.
```

## 비트 NOT 연산자

프로그래머스 기초문제를 풀다가 신기한 답변을 봤어요! 재밌어서 한번 가져와봤습니다.


# 문제


정수 `num1`과 `num2`가 매개변수로 주어질 때, `num1`을 `num2`로 나눈 값에 1,000을 곱한 후 정수 부분을 return 하도록 soltuion 함수를 완성해주세요.


# 재밌는 정답


```javascript
function solution(num1, num2) {
    return ~~(num1/num2*1000);
}
```


혹시 `~~연산자` 에 대해서 들어보셨나요?
`Math.trunc()` 과 비슷한 동작으로 소수점 뒷부분을 제외하고 정수만 반환하지만 `~~연산자` 는 큰 숫자에 대한 제한이 있고, `NaN`과 `null`에 대해선 `0`을 출력한다고 합니다.


`~~연산자` 가 **비트 NOT 연산자** 라고도 하는데, 우선 이 **비트 연산** 자체가 이진수 비트 단위로 직접 연산을 수행하는 방식이다보니, 일반 숫자 연산보다 빠르다고 하네요!
보통 **비트 연산**은 _암호화 및 해시 함수 구현_ 시 사용된다고도 합니다.


**비트 연산**에 대해서 추가적으로 궁굼하신 분들은 제가 노션에 따로 정리 해두었으니, 궁굼하신 분들은 보셔용!

## 원본 레포지토리에 새로운 브랜치를 로컬 레포지토리로 가져오기
1. 먼저, 원본 레포지토리를 `upstream`으로 등록합니다 (이미 등록되어 있다면 생략 가능).

    ```shell
    git remote add upstream <원본 레포지토리 URL>
    ```

2. `upstream`에서 최신 변경 사항을 가져옵니다.

    ```shell
    git fetch upstream
    ```

3. 가져온 변경 사항 중 새로 생긴 브랜치를 확인합니다.

    ```shell
    git branch -r
    ```


    여기서 `upstream/새로운브랜치이름` 형식으로 새로 생성된 브랜치를 확인할 수 있습니다.

4. 이제 새로 생긴 브랜치를 로컬로 가져옵니다.

    ```shell
    git checkout -b 새로운브랜치이름 upstream/새로운브랜치이름
    ```


이렇게 하면 `upstream`에서 새로 생긴 브랜치를 로컬에 복사하여 사용할 수 있습니다.

- 너무 많은 이모지를 모두 사용하려 한다기보다는, 자주 사용되는 이모지를 사용하며, 최대한 커밋의 내용에 크게 벗어나지 않는 선에서 이모지를 선택해 사용하면 좋을 듯 합니다.
- gitmoji를 사용한다 하더라도 커밋 컨벤션의 중요한 핵심은 <u>**“다른사람이 봐도 쉽게 알 수 있도록 간략하게 적기”**</u>
- 자주 사용하는 리스트

```javascript
⚡️ :zap: 성능 개선
💄 :lipstick: UI/스타일 파일 추가/수정
✨ :sparkles: 새 기능
📝 :memo: 문서 추가/수정
🐛 :bug: 버그 수정
🔥 :fire: 코드/파일 삭제
🚚 :truck: 리소스 이동, 이름 변경
📦 :package: 컴파일된 파일 추가/수정
🔨 :haer: 개발 스크립트 추가/수정
♻️ :recycle: 코드 리팩토링
```

- 예를 들어, 스크립트의 수정이 있었는데 💩 이모지를 사용하거나, 👽 이런 이모지를 사용하는 것은 금합니다(커밋의 내용과 너무 상반되는 이모지)

### 채택 이유

- 팀원 내 gitmoji를 사용해본 인원이 많으며, (최소 3명 이상) 이모지로 한눈에 파악이 가능
- 영어에 약한 사람에게는 오히려 영단어를 찾아보면서 커밋을 작성하는것이 생산성을 저하 시킬 수 있음
- [설치 및 사용법](https://velog.io/@heejin-k/%ED%98%91%EC%97%85%EC%9D%84-%EC%9C%84%ED%95%9C-%EC%BB%A8%EB%B2%A4%EC%85%98-Gitmoji-%EC%82%AC%EC%9A%A9%ED%95%98%EA%B8%B0)

# PR Rules


## git flow 전략으로 사용


### 채택 이유

- 코드 안정성 확보(완성도 up)
- 버전과 변경사항을 추적하기 쉽고 핫픽스가 필요할때도 원하는 수정 사항만 반영해서 적용시킬수 있기 때문
- TBD를 사용하지않는 이유
    - 생명 주기가 짧은 브랜치를 그때그때 생성하여, master에 바로 적용 시키는 방식
    - 빌드타임 or 런타임에 적용할 수 있는 플래그를 on/off를 통해 기능 단위로 컨트롤 할 수있게끔 외부 설정이 필요함 → 이 과정이 초기 세팅에 많은 번거로움을 가져오며, 다수가 작업할 경우 컴플릿이 일어날 확률이 높음
    - 변경사항의 추적이 어렵다는 점도 있음

### 사용되는 브랜치


`main(master)` → 최종 배포 서버


`release` → 테스트 배포 서버


`develop` → 개발 브랜치

- 생명주기가 짧은 브랜치
- `feature`
    - 이슈 생성 시 자동으로 `feature/#이슈번호_이슈제목` 형식으로 브랜치가 만들어짐([Github Action 설정](https://velog.io/@jifrozen/I-project-%ED%94%84%EB%A1%9C%EC%A0%9D%ED%8A%B8-%EC%84%B8%ED%8C%851-issue-%EC%83%9D%EC%84%B1%EC%8B%9C-branch-%EC%9E%90%EB%8F%99%EC%83%9D%EC%84%B1-%EB%90%98%EB%8F%84%EB%A1%9D-github-actions-%EC%84%A4%EC%A0%95) 예정)
    - 해당 브랜치에서 작업 완료 후 PR → 머지가 완료되면 연결된 이슈 브랜치는 삭제됨

### 작업 플로우


작업자는 이슈 생성 후 git fetch를 통해, 생성된 원격 브랜치를 로컬 브랜치로 가져온 다음 작업 후 PR을 올리면 된다.

1. 작업 시작 전 선 이슈 생성하여, 본인이 작업할 컴포넌트에 개발 목록을 TODO로 작성하여, 체크리스트를 통해, 작업 진행도 공유. 해당 컴포넌트에 마일스톤 연결
2. 자동 생성된 브랜치(**Github Action 이용하여 생성된 브랜치)**를 fetch를 통해 로컬로 가져와 작업
3. PR 올릴때 양식 제목(#이슈번호_PR 제목)
4. 연결 이슈, 변경 사항(내용), 테스트 결과
    1. 이슈를 연결할때에는 이슈 연결 키워드 본문에 작성

    ```javascript
    - close: 단순 이슈를 닫을때
    - fix / fixes / fixed: 문제나 버그 수정 완료
    - resolve / resolves / resolved: 해결을 완료
    
    // 사용법 : PR 본문에 사용
    close #100(여기서 #을 입력하면 아래 드롭다운 목록에 생성된 이슈 목록이 뜸)
    ```


## 코드 리뷰

- PR은 가장 작은 단위로 작성(이슈도 동일)
    - 예) Button 컴포넌트면 Button만 PR
    - 커밋 내역은 최대한 30개 이하로 → 너무 길어지면 코드 리뷰가 길어짐
    - 작은 단위로 작성하면 코드 리뷰하는데에도 10~15분 내외로 가능함(효율적인 코드 리뷰를 위해)
- 필수 리뷰어 2명 이상
    - 1명은 팀장 리뷰어로 지정
    - 그외에는 해당 컴포넌트를 사용하게 될 작업자로 지정
- 유의사항
    - 해당 코드 리뷰의 방식이 비효율이라고 생각하신다면 코드리뷰에 참여하지않아도됨

        단, 모든 프로젝트의 코드와 흐름에 대해서 파악하고 있어야함(필수조건)

    - 코드 리뷰가 1시간 이상 넘어갈 경우 디스코드 채널에 리마인드를 통해서 언급 필수
- 이전에 말했던 AI 코드 리뷰를 도입하는 방식은 학습자 입장으로는 적합하지 않다고 판단
- 프로젝트 완성도 중요하지만 대체제를 도입한다면 학습에 도움도 안되고, 완성을 시킬 자신이 없다면 개인 역량을 늘릴 것(작업 속도 개선 무엇때문에 속도가 안나는지 고민해보는 것도 좋다고 생각함→개선점 찾기)

## 효율적인 issue/PR 템플릿

## 새로운 기능 추가(feature)


### 이슈 템플릿


```javascript
## **✨ Feature Request: [기능 이름] 추가**

### **📌 개요**
현재 **[기존 상태]** 에서 **[새로운 기능이 필요한 이유]**  
이를 해결하기 위해 **[새로운 기능]** 을 추가하려 합니다.

---

### **🎯 요구사항**
✅ **[기능 A]** → [구체적인 동작 방식]  
✅ **[기능 B]** → [구체적인 동작 방식]  
✅ **[기능 C]** → [구체적인 동작 방식]  

🔹 **예상되는 사용자 인터랙션**:
1. [사용자가 어떤 행동]을 하면,  
2. [어떤 변화]가 UI에서 발생해야 한다.

---

### **💡 기대 효과**
✔ **사용자 경험(UX) 개선**: [기능이 추가되면 UX가 어떻게 개선되는지]  
✔ **퍼포먼스 최적화**: [새로운 기능이 성능 개선에 어떤 영향을 미치는지]  
✔ **비즈니스 가치 향상**: [이 기능이 서비스에 어떤 가치를 추가하는지]  

---

### **🛠 작업 내용**
- [ ] **UI 디자인 적용**
- [ ] **기능 구현**
- [ ] **API 연동**
- [ ] **테스트 및 QA 진행**

---

### **📌 참고 자료**
📌 **디자인 링크**: [Figma, Zeplin 등]  
📌 **API 문서**: [Swagger, Notion 문서 등]  
📌 **관련 기술 스택**: [사용할 라이브러리, 기술 등]  

---

### **📌 Assignee & Labels**
👨‍💻 **Assignee**: @FE 팀  
📅 **Due Date**: YYYY-MM-DD  
🔖 **Label**: `feature`, `UI/UX`, `frontend`
```

## 기존 코드 리팩토링(Refactoring & 개선 작업)


### 이슈 템플릿


```javascript
## **🛠 Refactor: [파일/기능명] 리팩토링 및 개선 작업**

### **📌 개요**
현재 **[기존 코드]** 에서 **[문제가 되는 부분]**  
이를 해결하기 위해 **리팩토링을 진행**합니다.

---

### **🔍 문제점**
- **코드의 응집도가 낮음**
- **중복 코드가 많음**
- **비효율적인 API 호출 발생**
- **UI/UX 개선 필요**

---

### **🎯 개선 방향**
✅ **불필요한 코드 제거 및 최적화**  
✅ **API 호출 최적화 (useQuery 적용 등)**  
✅ **UI 구조 개선 및 유지보수성 향상**  

---

### **🛠 작업 내용**
- [ ] **코드 구조 개선**
- [ ] **불필요한 렌더링 제거**
- [ ] **UI 최적화**

---

### **📌 Assignee & Labels**
👨‍💻 **Assignee**: @FE 팀  
📅 **Due Date**: YYYY-MM-DD  
🔖 **Label**: `refactor`, `performance`, `frontend`
```

## 리액트 활용을 위해 필요한 CS 지식 범위

# 들어가기앞서..


해당 게시글을 작성하는 이유는 **리액트 hooks의 대한 동작원리에 대해 정확히 파악**하고, 빠른 이해를 위해 알고있어야하는 CS 개념에 대해서 정리하기 위한 글이다.


CS 개념에 대해 설명하기전에 많은 기업들이 CS 지식을 기본으로 요구하는 경우가 많다.


처음에 나는 방대한 양의 CS 지식을 <u>**‘어디서부터 어디까지 알고있어야 하는가’**</u>에 대한 고민을 한 적이 있다.


나는 전공자도 아닐뿐더러, 지금부터 CS 공부를 하기에는 너무 많은 시간을 필요로 하다는 것을 알고 있다.


하지만 현실적으로 모두 다 익히기에는 **불가능**에 가깝다.

## **✅ 프론트엔드 개발자가 리액트를 이해하기 위해 필요한 CS 개념**


리액트를 제대로 활용하려면 **자료구조, 메모리 관리, 비동기 처리, 렌더링 최적화** 같은 CS 개념이 중요하다


### **1️⃣ 자료구조 (Data Structures)**


리액트 내부에서는 여러 자료구조가 사용되는데, 특히 **큐(Queue)와 트리(Tree)** 개념이 중요하다.


**🔹 필수 자료구조 개념**


| **자료구조**                 | **리액트에서 활용되는 부분**                                    |
| ------------------------ | ---------------------------------------------------- |
| **Queue (큐, FIFO)**      | `setState`의 상태 업데이트 요청을 큐에 저장 후 한 번에 처리 `(Batching)` |
| **Stack (스택, LIFO)**     | `useEffect` `Cleanup`에서 언마운트 시 정리할 작업을 저장할 때 사용      |
| **Linked List (연결 리스트)** | 리액트의 Fiber는 **연결 리스트 형태**로 구현됨                       |
| **Tree (트리 구조)**         | 리액트의 컴포넌트 구조, 가상 DOM(Virtual DOM), *Fiber 구조         |
| **Hash Table (해시 테이블)**  | 리액트 내부에서 빠른 데이터 조회를 위해 해시 테이블을 활용                    |


*Fiber는 **리액트의 내부 데이터 구조이자 렌더링 엔진이다.**


### **2️⃣ 메모리 관리 & 렌더링 최적화**


리액트는 성능 최적화를 위해 메모리를 효율적으로 관리해야한다.


**🔹 필수 메모리 관련 개념**


| **개념**                           | **리액트에서 활용되는 부분**                                        |
| -------------------------------- | -------------------------------------------------------- |
| **Heap & Stack 메모리**             | 리액트의 상태(`State`)는 Heap에 저장됨. `Stack`은 함수 실행 컨텍스트 관리      |
| **Garbage Collection (가비지 컬렉션)** | `useEffect`에서 정리(`cleanup`) 함수(`return () => {}`)를 쓰는 이유 |
| **Memoization (메모이제이션)**         | `useMemo`, `useCallback`이 불필요한 연산을 줄이는 원리                |
| **Reference & Value Type**       | `useState`에서 객체 상태 관리할 때 얕은 비교(===), 불변성 유지 필요           |
| **Memory Leak (메모리 누수)**         | `useEffect`에서 이벤트 리스너 해제 (`cleanup`)를 하지 않으면 메모리 누수 발생   |


📌 **이 개념을 모르면** **`useEffect`****를 제대로 활용하기 어렵고,** <u>**메모리 누수 버그**</u>**를 만들기 쉽다.**


### **3️⃣ 비동기 처리 (Asynchronous Programming)**


리액트에서 비동기 처리는 매우 중요함.


<u>**API 호출, 이벤트 핸들링, 상태 업데이트**</u>는 대부분 비동기로 동작한다.


**🔹 필수 비동기 개념**


| **개념**                      | **리액트에서 활용되는 부분**                                     |
| --------------------------- | ----------------------------------------------------- |
| **Event Loop & Call Stack** | 리액트의 상태 업데이트가 비동기로 처리되는 이유 (setState → 비동기 업데이트)      |
| **Microtask & Macrotask**   | `Promise`와 `setTimeout`이 실행되는 순서 차이 (`useEffect`와 관련) |
| **Promise & async/await**   | fetch API로 데이터 가져오기 (`useEffect` 안에서 `async` 함수 사용)   |
| **Debounce & Throttle**     | 검색창 입력 처리, 스크롤 이벤트 최적화 (`onChange` 이벤트 핸들링)           |


📌 **비동기 개념을 모르면** **`useEffect`****에서 API 요청을 잘못 처리해서 무한 호출 문제가 생길 수 있다.**


### **4️⃣ 리액트의 렌더링 원리 & 최적화**


리액트의 핵심은 **렌더링 최적화이다.**


 렌더링 과정과 성능 최적화를 알면 불필요한 렌더링을 막을 수 있다.


**🔹 필수 개념**


| **개념**                                    | **리액트에서 활용되는 부분**                     |
| ----------------------------------------- | ------------------------------------- |
| **Virtual DOM (가상 DOM)**                  | 리액트의 성능 최적화 방식, 실제 DOM과의 차이점 이해       |
| **Reconciliation (재조정 알고리즘)**             | `React.memo`가 컴포넌트 리렌더링을 막는 원리        |
| **Diffing Algorithm (비교 알고리즘)**           | 리액트가 변경된 부분만 효율적으로 업데이트하는 방법          |
| **Component Re-rendering (컴포넌트 리렌더링 원리)** | `useMemo`, `useCallback`을 왜 써야 하는지 이해 |
| **Batching (배치 처리)**                      | 여러 `setState`가 하나의 렌더링 사이클에서 처리되는 원리  |


📌 **이 개념을 이해하면 불필요한 리렌더링을 줄이고 성능 최적화를 쉽게 할 수 있다.**


### **5️⃣ 네트워크 & 브라우저 동작 원리**


리액트는 클라이언트(브라우저)에서 동작하므로, 네트워크와 브라우저 렌더링 원리를 알아야한다.


**🔹 필수 개념**


| **개념**                            | **리액트에서 활용되는 부분**                         |
| --------------------------------- | ----------------------------------------- |
| **HTTP & REST API**               | fetch 또는 axios를 사용해서 API 요청을 보낼 때         |
| **CORS (교차 출처 리소스 공유)**           | API 요청 시 CORS 에러 해결 방법                    |
| **DNS & 캐싱 전략**                   | 이미지 최적화, `useMemo` & `useCallback` 최적화 원리 |
| **CSR vs SSR vs SSG**             | 리액트, Next.js 같은 프레임워크가 페이지를 렌더링하는 방식      |
| **LocalStorage & SessionStorage** | JWT 토큰 저장, 사용자 데이터 유지 (`useEffect`에서 활용)  |


📌 **CSR(클라이언트 사이드 렌더링)과 SSR(서버 사이드 렌더링)의 차이를 이해하는 것이 좋다.(NextJs)**

### 알고리즘


## 코딩테스트에 많이 나오는 알고리즘

1. 정렬 (Sorting)

- 대표 알고리즘: 버블 정렬, 선택 정렬, 삽입 정렬, 퀵 정렬, 병합 정렬, 힙 정렬
- 예제 문제: 배열을 오름차순/내림차순으로 정렬하기, k번째 작은 수 찾기

2. 탐색 (Search)

- 선형 탐색 (Linear Search): 하나씩 확인하는 방법
- 이진 탐색 (Binary Search): 정렬된 배열에서 절반씩 나눠가며 찾는 방법
- DFS (깊이 우선 탐색), BFS (너비 우선 탐색): 그래프 탐색 알고리즘

3. 자료구조 (Data Structures)

- 스택(Stack), 큐(Queue), 덱(Deque)
- 해시 테이블 (Hash Table)
- 힙 (Heap, 우선순위 큐)
- 트리 (Tree) 및 그래프 (Graph)

4. 동적 계획법 (Dynamic Programming, DP)

- 부분 문제를 해결하며 결과를 저장해 중복 계산을 줄이는 기법
- 예제 문제: 피보나치 수열, 배낭 문제, 최장 공통 부분 수열 (LCS)

5. 그리디 알고리즘 (Greedy Algorithm)

- 매 순간 최적의 선택을 해서 전체적으로 최선의 해를 찾는 방식
- 예제 문제: 거스름돈 문제, 최소 신장 트리 (Kruskal, Prim)

6. 분할 정복 (Divide and Conquer)

- 문제를 작은 문제로 나누고 개별적으로 해결한 후 합치는 방식
- 예제 문제: 병합 정렬, 퀵 정렬, 이진 탐색

7. 그래프 알고리즘 (Graph Algorithms)

- DFS, BFS (그래프 탐색)
- 최단 경로 알고리즘: 다익스트라, 플로이드-워셜, 벨만-포드
- 최소 신장 트리 (MST): 크루스칼, 프림 알고리즘

8. 비트 마스킹 (Bit Manipulation)

- 비트 연산을 활용한 문제 해결 방식
- 예제 문제: 부분 집합 구하기, XOR 활용 문제

9. 백트래킹 (Backtracking)

- 가능한 모든 경우를 탐색하되, 불가능한 경로는 조기에 중단하는 방식
- 예제 문제: N-Queen 문제, 순열 및 조합 생성

10. 문자열 알고리즘 (String Algorithms)

- KMP, 라빈-카프 (Rabin-Karp) (문자열 검색)
- 트라이 (Trie, 접두사 트리)
- Z-알고리즘

## 프론트엔드에서 활용할 수 있는 알고리즘 사례

프론트는 보통 UI/UX 중심이지만, 성능 최적화나 데이터 처리에서 알고리즘이 중요한 역할을 함


1. 정렬 (Sorting)


사례: 상품 리스트 정렬

- 쇼핑몰에서 가격, 리뷰 수, 평점 순으로 상품을 정렬할 때 정렬 알고리즘이 필요해.
- Array.prototype.sort()를 활용하지만, 대량의 데이터를 다룰 때 퀵 정렬(Quick Sort)이나 병합 정렬(Merge Sort) 같은 고성능 정렬 알고리즘을 고려할 수도 있어.

```javascript
const products = [
{ name: "상품 A", price: 20000 },
{ name: "상품 B", price: 15000 },
{ name: "상품 C", price: 30000 },
];

products.sort((a, b) => a.price - b.price); // 가격 오름차순 정렬

console.log(products);
```


2. 탐색 (Search)


사례: 자동완성 기능 (이진 탐색, Trie)

- 검색창에서 사용자가 입력하면 실시간으로 추천 검색어를 제공하는 기능.
- 정렬된 데이터라면 이진 탐색 (Binary Search) 을 활용해서 빠르게 찾을 수 있어.
- 많은 데이터를 다룰 경우, Trie (트라이, 접두사 트리) 를 사용하면 검색 속도를 최적화할 수 있어.

```javascript
function binarySearch(arr, target) {
let left = 0, right = arr.length - 1;
while (left <= right) {
let mid = Math.floor((left + right) / 2);
if (arr[mid] === target) return mid;
else if (arr[mid] < target) left = mid + 1;
else right = mid - 1;
}
return -1;
}

const words = ["apple", "banana", "grape", "melon", "orange"];

console.log(binarySearch(words, "grape")); // 2
```


3. 동적 계획법 (DP)


사례: 이미지 로딩 최적화 (Lazy Loading)

- 브라우저에서 한 번 로드한 이미지를 캐시하여 재사용하는 방식.
- 메모이제이션(Memoization) 을 이용하면 불필요한 중복 로딩을 방지할 수 있어.

```javascript
const imageCache = new Map();

function loadImage(url) {

if (imageCache.has(url)) {

return Promise.resolve(imageCache.get(url));

}

return fetch(url)

.then(response => response.blob())

.then(blob => {

imageCache.set(url, blob);

return blob;

});

}
```


4. 그리디 알고리즘 (Greedy Algorithm)


사례: 반응형 웹의 최적 레이아웃 배치

- CSS Grid/Flexbox가 그리디 방식으로 최적의 배치를 계산해줌.
- 예를 들어, 화면 너비에 맞춰 카드형 UI에서 최대한 많은 요소를 한 줄에 배치하는 방식.

```javascript
.container {

display: flex;

flex-wrap: wrap;

gap: 10px;

}

.item {

flex: 1 1 auto;

min-width: 200px;

}
```


5. 그래프 알고리즘


사례: 소셜 네트워크 추천 시스템 (DFS, BFS)

- 친구 추천, 유사 상품 추천 등을 구현할 때 그래프 탐색 (DFS, BFS) 알고리즘을 활용할 수 있어.

```javascript
function bfs(graph, start) {

let queue = [start];

let visited = new Set();

while (queue.length) {

let node = queue.shift();

if (!visited.has(node)) {

console.log(node);

visited.add(node);

queue.push(...graph[node]);

}

}

}

const socialGraph = {

A: ["B", "C"],

B: ["A", "D", "E"],

C: ["A", "F"],

D: ["B"],

E: ["B", "F"],

F: ["C", "E"]

};

bfs(socialGraph, "A"); // A → B → C → D → E → F
```


6. 문자열 알고리즘


사례: 실시간 입력 검증 (KMP 알고리즘)

- 입력값이 특정 패턴을 포함하는지 빠르게 확인하는 검색 기능.

```javascript
function kmpSearch(text, pattern) {

const n = text.length, m = pattern.length;

let lps = Array(m).fill(0), j = 0;

for (let i = 1; i < m; i++) {

while (j > 0 && pattern[i] !== pattern[j]) j = lps[j - 1];

if (pattern[i] === pattern[j]) lps[i] = ++j;

}

j = 0;

for (let i = 0; i < n; i++) {

while (j > 0 && text[i] !== pattern[j]) j = lps[j - 1];

if (text[i] === pattern[j]) j++;

if (j === m) return i - m + 1;

}

return -1;

}

console.log(kmpSearch("hello world", "world")); // 6
```


7. 백트래킹 (Backtracking)


사례: 캘린더 일정 추천 (N-Queen 방식)

- 사용자의 일정과 충돌하지 않는 최적의 회의 시간 추천.

8. 비트마스킹 (Bit Manipulation)


사례: 사용자 권한 관리

- 사용자 권한을 비트 연산으로 관리하면 성능 최적화 가능.

```javascript
const READ = 1;  // 001

const WRITE = 2; // 010

const EXECUTE = 4; // 100

let userPermission = READ | WRITE; // 011

console.log((userPermission & READ) > 0);  // true

console.log((userPermission & EXECUTE) > 0); // false
```


결론


프론트엔드 개발에서도 검색 최적화, UI 배치, 데이터 처리, 성능 개선 등 다양한 곳에서 알고리즘이 쓰임


## 코딩 테스트 문제 유형 & 최적 알고리즘

코딩 테스트 문제 유형 & 최적 알고리즘 정리


1. 그래프 탐색 & 최단 경로


| **문제 유형**        | **최적 알고리즘**                                    |
| ---------------- | ---------------------------------------------- |
| 최단 경로 찾기         | BFS (가중치 없음) / 다익스트라 (가중치 있음) / 플로이드-워셜 (모든 쌍) |
| 미로 찾기 (최단거리 이동)  | BFS (너비 우선 탐색)                                 |
| 네트워크 연결 (최소 비용)  | 크루스칼 / 프림 (최소 신장 트리)                           |
| 지도에서 섬 개수 찾기     | DFS / BFS (그래프 탐색)                             |
| A에서 B까지 모든 경로 찾기 | DFS / 백트래킹                                     |


2. 정렬 & 탐색


| **문제 유형**        | **최적 알고리즘**              |
| ---------------- | ------------------------ |
| 정렬된 배열에서 특정 값 찾기 | 이진 탐색 (Binary Search)    |
| k번째 작은/큰 수 찾기    | 힙 정렬 (Heap Sort) / 퀵 셀렉트 |
| 중복 제거 및 정렬       | 해시셋(Set) + 정렬            |
| 가장 많이 등장한 요소 찾기  | 해시맵(HashMap)             |


3. 동적 계획법 (DP)


| **문제 유형**                | **최적 알고리즘**   |
| ------------------------ | ------------- |
| 피보나치 수열                  | DP (메모이제이션)   |
| 배낭 문제 (Knapsack Problem) | DP (O(NW))    |
| 최장 증가 부분 수열 (LIS)        | DP + 이진 탐색    |
| 최대 부분 합 구하기              | DP (카데인 알고리즘) |
| 문자열 유사도 비교 (LCS)         | DP            |


4. 그리디 알고리즘 (Greedy)


| **문제 유형**         | **최적 알고리즘**    |
| ----------------- | -------------- |
| 거스름돈 문제           | 그리디 알고리즘       |
| 회의실 배정 (최대 활동 선택) | 그리디 (시작 시간 정렬) |
| 최소한의 동전으로 금액 맞추기  | 그리디            |
| 최소 신장 트리 (MST)    | 크루스칼 / 프림      |


5. 문자열 처리


| **문제 유형**            | **최적 알고리즘** |
| -------------------- | ----------- |
| 문자열 검색 (부분 문자열 찾기)   | KMP / 라빈-카프 |
| 가장 긴 공통 부분 문자열 (LCS) | DP          |
| 애너그램 판별              | 정렬 / 해시맵    |
| 회문(팰린드롬) 여부 확인       | 투 포인터       |


6. 조합 & 순열 & 백트래킹


| **문제 유형**       | **최적 알고리즘** |
| --------------- | ----------- |
| 모든 순열 구하기       | 백트래킹        |
| N-Queen 문제      | 백트래킹        |
| 비밀번호 생성 (조합)    | DFS / 백트래킹  |
| 최적의 경로 찾기 (TSP) | 백트래킹 / DP   |


7. 비트마스킹 & 수학


| **문제 유형**               | **최적 알고리즘** |
| ----------------------- | ----------- |
| 부분 집합 구하기               | 비트마스킹       |
| 소수 판별                   | 에라토스테네스의 체  |
| 최대공약수(GCD) & 최소공배수(LCM) | 유클리드 호제법    |


8. 트리 & 이진 트리


| **문제 유형**          | **최적 알고리즘** |
| ------------------ | ----------- |
| 이진 트리 순회 (DFS/BFS) | DFS / BFS   |
| 트라이(Trie) 활용       | 문자열 탐색      |
| 이진 검색 트리 (BST) 탐색  | 이진 탐색       |


✅ 결론


1. 모든 알고리즘을 외울 필요는 없지만, 문제 유형별로 어떤 알고리즘이 유리한지는 알아야 함.


2. 비슷한 문제를 반복해서 풀어보면, 패턴이 보이면서 더 빠르게 적용 가능.


3. 코딩 테스트를 준비할 때, 이 정리표를 보고 어떤 알고리즘을 연습할지 선택하면 효율적!
