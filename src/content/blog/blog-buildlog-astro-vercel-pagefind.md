## 왜 또 블로그를 옮겼나

글이 세 군데에 나뉘어 있었다. velog에 15편, tistory에 9편, Notion에 53편. 각 플랫폼을 쓴 이유는 그 당시에는 다 있었는데, 시간이 지나고 보니 문제가 하나 생겼다. 내 글인데 내 것이 아닌 느낌.

velog는 개발자 커뮤니티가 좋았고, tistory는 SEO가 잘 됐고, Notion은 쓰기 편했다. 그런데 URL은 남의 도메인이고, 디자인은 내 손에 없고, 검색 유입도 플랫폼 알고리즘에 기댔다. "내 글이 내 도메인에 있어야 한다"는 생각이 결국 세 번째 이전을 결정하게 했다.

프레임워크는 Astro 6를 골랐다.

## Astro를 고른 이유

정적 사이트 생성기는 선택지가 꽤 많다. Next.js도 있고 Hugo도 있고 Eleventy도 있다. 그중 Astro를 고른 이유는 단순했다. 빌드 결과가 `.html` 파일로 깔끔하게 떨어진다는 것.

Next.js도 정적 내보내기를 지원하지만 React 런타임과 청크 파일이 같이 붙는다. Astro는 기본적으로 JavaScript를 거의 내보내지 않는다. 블로그 독자 입장에서 페이지를 열면 그냥 HTML이 온다는 게 체감상 다르다. 처음 로딩에서 빈 화면이 보이지 않는다.

기능 면에서는 Content Collections가 결정적이었다. `src/content/blog/` 폴더에 마크다운 파일을 넣고 Zod 스키마로 frontmatter를 정의하면, 빌드 타임에 타입 검증이 돌아간다. title이 120자를 넘거나 slug에 대문자가 들어가면 빌드가 실패한다. 이게 귀찮은 것 같지만 실제로는 편하다. 실수를 배포 전에 잡아준다.

View Transition과 Image Optimization은 기본 탑재라 별도 설정이 거의 필요 없었다. `<ViewTransitions />` 컴포넌트 하나 추가하면 페이지 전환이 부드러워진다.

## Vercel을 고른 이유

배포 플랫폼 선택은 5분도 안 걸렸다. GitHub에 push하면 자동으로 배포된다. 실제로 측정해보니 빌드에서 CDN 반영까지 13초였다.

무료 티어에서 preview deployment가 된다는 것도 마음에 들었다. PR을 열면 자동으로 preview URL이 붙는다. 혼자 운영하는 블로그라 PR 리뷰 같은 건 없지만, 글을 올리기 전에 모바일에서 preview URL로 미리 보는 용도로 쓴다.

도메인 연결도 Vercel 대시보드에서 몇 줄이면 된다. `vercel.json`에 보안 헤더와 redirect 규칙을 넣으면 그대로 반영된다.

## 마이그레이션 — 가장 길었던 구간

77편을 옮기는 데 자동화를 최대한 활용했는데도 생각보다 시간이 걸렸다.

velog는 RSS 피드를 제공해서 15편은 비교적 수월하게 가져왔다. tistory는 RSS에 본문 전체가 담겨서 9편을 가져왔는데, 카테고리 정보가 RSS에 제대로 안 나왔다. 카테고리별 아카이브 페이지를 스크래핑하려고 시도했다가 구조가 달라서 미완으로 두었다. 아직 발굴 안 된 글이 10~30편 정도 더 있을 것으로 추정한다.

Notion은 53편이었다. 공식 API로 페이지 트리를 걸어 다니며 마크다운으로 변환했다. Notion의 블록 구조가 마크다운으로 1:1 변환이 안 되는 경우가 있어서 callout, toggle, 컬럼 레이아웃은 수동 정리가 필요했다.

가장 번거로웠던 건 description이었다. 이미지 마크다운(`![alt](url)`)이나 HTML 태그가 섞인 글이 21편이었다. Astro 스키마가 description을 순수 텍스트로 요구해서 스크립트로 일괄 정제했다.

기존 URL 보존을 위해 redirect를 38개 추가했다. velog 슬러그와 Notion 페이지 ID 기반 URL 패턴이 달랐기 때문이다.

## Pagefind 검색 — 추가 호스팅 없이 검색 달기

정적 사이트에서 검색은 항상 선택지가 불편하다. Algolia는 외부 의존성이 생기고 API 호출 제한이 있다. FlexSearch는 브라우저에서 인덱스를 로드해야 해서 초기 로딩이 무거워진다.

Pagefind를 선택한 이유는 빌드 시점에 인덱스를 생성해서 정적 파일로 배포하기 때문이다. 외부 서버가 없고 추가 비용이 없다. `npm run build` 후 `npx pagefind --site dist`를 실행하면 `dist/pagefind/` 폴더에 인덱스가 생성된다.

단점도 있다. 한글 검색이 약하다. Pagefind는 기본적으로 공백 기반 토크나이징이라 "리액트"와 "React"를 연결하지 못한다. 이걸 보완하기 위해 한글-영어 태그 매핑을 만들었다. `tagMapping.cjs`에 "리액트" → ["react", "리액트"] 같은 매핑을 정의하고, `augment-tags.mjs`로 76편의 frontmatter에 양방향 태그를 추가했다. 적용 전 "리액트" 검색 매칭이 12편이었는데 적용 후 26편으로 늘었다.

## OG 이미지 자동 생성 — 82장을 손으로 그릴 수 없다

소셜 공유 시 썸네일이 없으면 밋밋하다. 82편 각각에 OG 이미지를 만들어야 했다.

Satori와 Resvg를 사용해 빌드 타임에 생성했다. `src/pages/og/[...slug].png.ts` 라우트를 만들고, 글 제목과 카테고리를 받아서 JSX 레이아웃을 PNG로 렌더링한다. 한글 폰트를 바이너리로 임베드하는 부분이 가장 까다로웠다. Pretendard 폰트 파일을 `ArrayBuffer`로 읽어서 Satori에 넘겼다. 카테고리별로 배경 색상이 다르게 나오도록 컬러 맵도 정의했다.

빌드할 때마다 82장이 자동으로 생성된다. 나중에 글이 늘어도 신경 쓸 필요가 없다.

## 그리고 가장 미세했지만 가장 짜증 났던 버그

컨테이너 너비를 720px에서 1200px로 넓혔다. 그런데 페이지 전환할 때마다 컨테이너 너비가 순간적으로 튀는 현상이 생겼다. 1200px로 보이다가 전환 직후 잠깐 좁아졌다가 다시 1200px이 되는 식이었다.

원인은 View Transition 중 스크롤바였다. 페이지가 전환되는 순간 `overflow: hidden`이 걸리면서 스크롤바가 사라지고, 그 너비만큼 레이아웃이 밀렸다. CSS 한 줄로 해결되긴 했다.

```css
scrollbar-gutter: stable;
```

다만 이것만으로는 부족했고, Astro의 `after-swap` 이벤트에서 클래스 클린업도 같이 걸어야 완전히 잡혔다. 결국 3중 방어 코드가 됐다. 육안으로 잘 안 보이는 버그가 원인 파악에 가장 오래 걸렸다.

## 접근성 — Pa11y를 CI에 꽂다

GitHub Actions에 빌드 + Pa11y 접근성 검사 워크플로우를 추가했다. WCAG 2.1 AA 기준으로 홈, 블로그 목록, 상세 페이지 등 5개 URL을 검사한다.

처음 실행하자마자 전부 실패했다. footer, pagination, TOC의 색상 대비가 2.54:1로 기준인 4.5:1에 한참 못 미쳤다. CSS 변수 한 줄을 바꿨다.

```css
--fg-subtle: #9ca3af; /* 이전 */
--fg-subtle: #5a6573; /* 변경 후, 4.5:1 이상 */
```

5개 URL 전부 통과했다. CI 빨간불이 결국 디자인 토큰 한 줄짜리 문제인 경우가 꽤 많다. 자동화 검사가 없었다면 이 상태로 계속 운영했을 것이다.

## 다음 단계

현재 slug가 hangul-romanize 라이브러리로 생성된 음역 형태라 `deiteobeiseusqllite` 같은 식이다. `database-sqlite`처럼 의미 단위로 다듬을 필요가 있다. redirect를 유지하면서 슬러그를 교체해야 해서 일이 조금 크다.

tistory 잔여 글 발굴도 남아 있다. 카테고리 아카이브 스크래핑을 보강하면 10~30편 정도 더 가져올 수 있을 것으로 본다.

JSON-LD 구조화 데이터도 Person/Organization 타입으로 확장하면 구글 리치 결과에 나올 가능성이 높아진다. 글이 100편을 넘으면 검색 엔진으로 Algolia나 FlexSearch 도입도 재평가할 생각이다.

## 글을 쓰는 도구가 글을 결정한다

마이그레이션을 마치고 느낀 건, 도구가 단순할수록 글에 집중하기 쉽다는 것이다.

Notion 시절엔 블록을 어떻게 배치할지 생각하느라 시간을 썼다. 지금은 에디터에서 마크다운을 쓰고 `git commit`을 누른다. 그게 전부다. 글이 배포되기까지 13초 걸린다.

정적 사이트 + 마크다운 + GitHub 커밋이라는 흐름이 군더더기를 제거했다. 글을 더 많이 쓰게 될지는 모르겠지만, 적어도 쓰기로 결심했을 때 앞을 막는 게 없다. 그걸로 충분하다.
